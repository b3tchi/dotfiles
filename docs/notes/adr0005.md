# ADR [[cat002]] [[product]]

## title
No d3d12 GPU acceleration outside the WSLg-blessed path on WSL2

## context
Mesa's d3d12 driver renders via /dev/dxg without DRM nodes (drisw
path), so glxinfo reports a GPU renderer under any X server — but
presentation is broken outside WSLg's blessed path. Evidence from
[[poc001]] and [[poc002]]: GLX swapbuffers presents black windows
(renders ~100 FPS unseen, reproduced after clean VM reboot); EGL on
X11 segfaults on the missing DRI3 device; weston's GL renderer lost
the D3D12 device under WebGL load and crashed the compositor.
Worse, these failures poison the /dev/dxg paravirt channel VM-wide
(dxgkio ioctl errors -22/-75) — GPU apps then crash in both the WSLg
sway session and the xrdp session, with stale artifacts left on the
Windows host, until wsl --shutdown. Measured llvmpipe performance
roughly matches the WSLg GPU path for tested real workloads anyway.

## decision
GALLIUM_DRIVER=d3d12 is forbidden in any non-WSLg display stack on
WSL2 (xrdp sessions, custom weston instances, any nested X server).
GPU-accelerated GL is used only inside WSLg RAIL windows; everything
else renders on llvmpipe.

## consequences
Positive: no VM-wide GPU poisoning, stable custom display stacks,
one simple rule enforced in xrdp/xinitrc and session overlays.
Negative: no GPU acceleration over RDP — heavy 3D stays CPU-bound.
Revisit if mesa's d3d12 DRI3/present support matures upstream, or
via a VirtualGL-style offscreen-render + readback approach.
