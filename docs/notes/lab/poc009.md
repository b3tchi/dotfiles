# PoC [[cat002]] [[board]]

## hypothesis
Running the region overlay inside an ALREADY-WARM quickshell process (the
mod+d launcher pattern: window pre-created but hidden, an IPC call just flips
visible=true) collapses the 67-150ms open black gap to <=1 frame, because the
gap is QML-engine / software-renderer warmup.

## method
Scratch dir /tmp/poc-warm-qml (throwaway). Minimal quickshell config: a
PanelWindow { visible: false } holding a fullscreen Image, plus an IpcHandler
whose call sets the source and flips visible=true — the same shape as the real
overlay and the same pattern as quickshell/overlay/shell.qml (mod+d). Process
started and left to settle 3s BEFORE recording, so QML engine, scene graph and
software renderer are all warm. Then :10 recorded at 60fps (ffmpeg x11grab,
lossless ffv1) while the open was driven purely by IPC — no process spawn.
Per-frame luma via signalstats YAVG. QT_QUICK_BACKEND=software, as
qs-screenshot.sh sets over xrdp.

POSITIVE CONTROL (essential — the first two runs were false passes without it):
the displayed grab was tinted 45% red via magick. An UNtinted grab is pixel-
identical to the live screen, so a flat luma trace cannot distinguish "mapped
perfectly, no blink" from "never mapped at all". With the tint, a successful
map is unmistakable (luma jumps to ~59-77).

## result
INVALIDATED. The warm instance blinks just as badly as the cold one.

  t=1.517  luma 41.31   baseline desktop
  t=1.533  luma 16.78   <- BLACK (limited-range Y=16), holds 5 frames ~100ms
  t=1.633  luma 59.40   tinted image finally paints

black_frames=5 (~100ms) from a process warmed for 3 seconds. The shipped
cold-start path measures 3-7 black frames / 67-150ms. Same range: warming buys
nothing.

CONCLUSION: cold start was never the cause. The gap is the fullscreen opaque
PanelWindow MAP (X clears the window to its black background pixel) plus Qt
first-paint of a 2560x1440 image on the software backend. Both costs are paid
per-open regardless of process age, so no amount of pre-warming or in-process
hosting removes them.

Gotcha worth keeping: an IpcHandler function named `show` is unreachable from
the CLI — `quickshell ipc call <target> show` collides with the `ipc show`
subcommand and silently prints the target listing with exit 0 instead of
calling. Renaming to `go` fixed it. A silent exit-0 no-op is a nasty trap.

## recommendation
Do NOT pursue "move the region overlay into the running quickshell / mod+d
style" as a blink fix. It was the cheapest-looking option and it does not work.
It remains valid only as a way to keep the dim while ACCEPTING the ~100ms
blink.

This also retires the pre-warm direction recorded on dotfiles-02h — that issue
suggested a persistent warm instance as the likely fix; measured, it is not.

The only measured path to zero blink stays [[poc008]] (GDK + X Shape live
overlay: black_frames=0, cold start included), which costs the dim. So the
fork is now evidence-backed on both sides: dim + ~100ms blink (any QML design,
warm or cold), or no dim + no blink (GDK shape). Nothing in between exists
without a compositor.
