# PoC [[cat002]] [[board]]

## hypothesis
Servo (servoshell nightly — same engine Verso wraps) launches on this Manjaro/AMD Vega X11 box and renders the akm-graph force-graph viewer (localhost:4810, HTML + Canvas2D + d3-force) correctly, as a candidate lightweight native webview host to replace running the viewer in a full browser.

## method
Scratch dir /tmp/poc-webview-host (throwaway). Verso has NO prebuilt binary (gitlab.com/verso-browser/verso + GitHub mirror releases both empty, no tags, not in repos) — so building Verso = full Servo-from-source (mozjs/SpiderMonkey, GBs, hours). Substituted servoshell nightly (servo/servo-nightly-builds tag 2026-07-12, servo-x86_64-linux-gnu.tar.gz, 48MB -> 134MB bin, all libs resolved) since Verso only wraps the same Servo engine — the render question is engine-level. Ran ./servoshell http://localhost:4810/ against the live akm-graph-d daemon (43 nodes/155 links). Screenshotted display :0 (physical, AMD radeonsi hardware) and :10 (xrdp, software EGL).

## result
VALIDATED render. Force-graph draws correctly on BOTH displays: colored nodes, links, HTML label pills, legend chips (archived/stories/specs/impl/features/decisions/categories/pocs/personal/hubs), status line. No crash. RSS ~255MB at start -> ~430MB rendered, ~4 procs. xrdp :10 fell to software EGL (libEGL DRI3 error: no accelerated rendering — an xrdp limitation, not Servo); :0 used hardware radeonsi. Landscape :0 window spread the graph legibly; tall xrdp window bunched it (zoomToFit weak in that aspect).

## recommendation
Servo CAN render the viewer — Canvas2D coverage is NOT a blocker (the high-risk unknown is killed). BUT do not adopt Servo/Verso as the host: (1) ~430MB is a full browser engine, NOT lighter than the browser it would replace — a webview saves browser chrome/tabs/extensions, not the engine cost, which is the hundreds-of-MB floor for ANY webview (WebKitGTK/Servo/Blink alike). (2) No Verso prebuilt exists, so adopting a Servo webview means owning a Servo source build in bootstrap, or shipping servoshell (a browser, not an embeddable webview lib). Next spike if a native host is still wanted: Wry/WebKitGTK (lighter ~150-300MB, cargo-buildable here, webkit2gtk-4.1 present). If the real goal is minimal RAM, drop web engines entirely -> native Rust GUI (egui/gtk) rendering the graph, at the cost of rewriting the viewer and losing force-graph/HTML. Note: the original GPU-heat complaint was already solved by the force-graph 2D backend (dotfiles-olm) in the browser — so webview work is optional resource/UX polish, not required.
