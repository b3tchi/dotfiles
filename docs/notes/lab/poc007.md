# PoC [[cat002]] [[board]]

## informs
[[sp008|cursor-driven-preview-bridge]]

## hypothesis
Which webview host best renders the akm-graph viewer / sp008 file previews —
Servo (servoshell), Wry (WebKitGTK) and QtWebEngine — compared across
render-correctness, RAM (RSS + PSS at 1/2/3 concurrent), load speed (cold-start
+ whole-page), content (JS graph vs static image), and display (AMD hardware :0
vs xrdp software :10). Single combined record for the whole webview-host
investigation, backed by a reusable benchmark.

## method
Reusable harness committed at `bench/webview/` (bench.sh + timesrv.py). A python
beacon server serves the real viewer assets with a `/__done` beacon injected
(fires when the app sets title "OK nodes=N") plus a fullscreen image page.
Per engine it measures: cold-start (launch->first GET), whole-page
(launch->render beacon), and total PSS (/proc/PID/smaps_rollup) at 1/2/3
concurrent instances. Run: `bench/webview/bench.sh all --display :0 --content graph`
(also `--content image`, `--display :10 --xauth ~/.Xauthority` for xrdp).
Harnesses: servoshell nightly (prebuilt 134MB), a ~30-line wry/tao app (1.4MB,
system webkit2gtk-4.1), a generated qml WebEngineView (qt6-webengine 282MB dep).
Verso has NO prebuilt (gitlab/github releases empty) — servoshell stands in
since Verso only wraps the same Servo engine. Box: AMD Vega 8, 18GB, Manjaro;
xrdp = software EGL (llvmpipe, DRI3 unavailable).

## result
RENDER: all three render the force-graph correctly on BOTH :0 (hardware) and :10
(xrdp software) — screenshots confirm nodes/links/labels/legend. QtWebEngine +
Wry are chrome-less; servoshell shows a url bar.

RAM — PSS MiB, akm-graph, settled, display :0 (first window | +each added):
| engine        | 1x  | 2x  | 3x  | +window | fully-settled RSS |
|---------------|-----|-----|-----|---------|-------------------|
| wry/webkitgtk | 196 | 314 | 419 | ~110    | ~400              |
| servo         | 199 | 336 | 475 | ~138    | ~439              |
| qtwebengine   | 363 | 579 | 790 | ~213    | ~625              |

RAM — PSS MiB, IMAGE via web engine, display :10 (xrdp):
| engine        | 1x  | 2x  | 3x  | +window |
|---------------|-----|-----|-----|---------|
| wry/webkitgtk | 178 | 280 | 379 | ~100    |
| servo         | 250 | 446 | 635 | ~192    |
| qtwebengine   | 315 | 501 | 683 | ~184    |

NATIVE image (Qt Quick Image, no web engine): ~155 MiB single proc — the tiered
path, no browser engine for image/video/code/pdf/3D.

LOAD (median of 3, seconds):
| engine        | cold-start :0 | whole-page :0 | whole-page :10 | img-web :10 |
|---------------|---------------|---------------|----------------|-------------|
| servo         | 0.27          | 0.36          | 0.49           | -           |
| wry/webkitgtk | 0.53          | 0.69          | 0.67           | -           |
| qtwebengine   | 0.69          | 0.90          | 0.85           | 0.71        |

CROSS-DISPLAY: Wry on :0 (hardware) + Servo on :10 (xrdp) ran simultaneously,
both rendering, fed by one daemon — combined ~820 MiB (394 + 429). Per-window
display targeting via DISPLAY/XAUTHORITY; RAM additive per engine (minus shared
.so). Matches the quickshell per-display session model.

Key facts: (1) PSS scales sub-linearly — first window pays the whole engine, each
added window only its private JS-heap/DOM/canvas; RSS double-counts shared .so.
(2) Load is DISPLAY-INDEPENDENT (~0.4-0.9s app-ready on both hardware and xrdp) —
app-ready is CPU-bound; software rendering only costs interactive FPS, not
startup. (3) Same content per window in the RAM sweep; different content gives
similar marginals (private anon heap not shared) scaled by page weight.

## decision
**Host chosen: Wry (WebKitGTK).** Lightest RAM (196 MiB first window, ~110/added
— the lowest), tiny 1.4MB binary, engine already installed (webkit2gtk-4.1), a
clean chrome-less window, and standard web support so the viewer/previews run
as-is. Traded QtWebEngine's quickshell-stack integration (heaviest, +282MB
Chromium) and Servo's marginally faster boot (no prebuilt, source build) for the
lightest, already-present, standalone host. Feeds sp008 ([[sp008]]) preview host
+ any native akm-graph app-window.

## recommendation
Ranking (lighter + faster = better): Wry/WebKitGTK < Servo < QtWebEngine on both
RAM and per-window cost; Servo fastest cold-start; all render + all sub-1s.
- Lightest standalone host -> Wry (1.4MB binary, engine preinstalled, chrome-less,
  ~110 MiB/extra window).
- quickshell/Qt6-integrated host -> QtWebEngine (heaviest ~625MB + 282MB Chromium
  dep, but embeds as a native qs surface).
- Servo: renders + fastest boot, but no prebuilt Verso -> Servo source build to
  adopt as a webview; servoshell is a browser not an embeddable lib.
NO webview cuts RAM vs a browser (engine is the cost, ~400-790MB at 3x). For
sp008: one REUSED webview (cold start paid once, warm nav after) + Qt Quick NATIVE
renderers (~155MB shared runtime) for image/video/code/pdf/3D; reserve the web
engine only for real HTML. Multi-window scales cheap (+100-210 MiB each), and
mixes across local + xrdp displays. The akm-graph GPU-heat that started this was
already fixed by the force-graph 2D backend ([[dotfiles-olm]]) — webview host work
is optional resource/UX polish. Reproduce any figure with `bench/webview/bench.sh`.
