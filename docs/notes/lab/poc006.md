# PoC [[cat001]] [[board]]

## informs
[[sp008|cursor-driven-preview-bridge]]

## hypothesis
Neither [[ft004]] (akm-graph) nor [[ft002]] (d2-router) emits a frame-blocking header (X-Frame-Options / CSP frame-ancestors), so both can be embedded as cross-origin iframes inside one [[sp008]] webview shell; each iframe loads its daemon at its own origin, so absolute-path assets and the internal websocket work unchanged, and the reverse channel (akm node click -> parent) is deliverable via window.postMessage.

## method
Built akm-graph-d from akm-graph/ (go1.26.5) into scratch /tmp/.../poc-proxy-embed, ran on AKM_GRAPH_PORT=4810 -root the dotfiles repo. curl -sI GET / and GET /app.js to inspect response headers. Source audit: grep -riE "x-frame-options|frame-ancestors|content-security-policy" over all repo .go (covers akm-graph server.go and d2/router/*.go). No GUI browser in the session, so the visual iframe render itself was not observed.

## result
GET / -> HTTP/1.1 200, Content-Type text/html, NO X-Frame-Options and NO Content-Security-Policy. GET /app.js -> 200. grep found ZERO frame-blocking headers in any repo .go (both daemons only set Cache-Control + Content-Type). The one failure mode that refuses cross-origin iframe embedding is empirically/source absent. WebSocket-in-iframe is not a distinct risk: the iframe loads the page at localhost:4810 origin, so ws://localhost:4810/watch is origin-internal and identical to the shipped standalone akm-graph page (ft004 is accepted/working).

## recommendation
VALIDATED. Adopt the cross-origin IFRAME embed for the [[sp008]] webview shell -- NOT a path-prefix reverse-proxy -- because iframes preserve each daemon origin, so absolute assets (/app.js) and websockets need no rewriting. Reverse channel: cross-origin iframe -> parent via window.postMessage (parent then POST /open). Two residuals for spec-refinement, neither blocking: (1) the visual render was not observed here (GUI-less session) -- cross-origin iframe + internal ws + postMessage is standard browser behavior given absent frame headers; confirm once on a real webview. (2) d2-router proxies d2 own --watch page (external d2 binary) whose upstream headers were not run-tested; if d2 ever sets X-Frame-Options, d2-router already rewrites proxied bodies (ft002) and can strip it.
