# PoC [[cat002]] [[board]]

## informs
[[sp016|clipcat-backend-swap]]

## hypothesis
A clipnotify+xclip loop on X11 captures every CLIPBOARD change — rapid
successive copies included (acceptance: at worst coalesce to newest) —
storing each entry byte-exact as a file under $XDG_RUNTIME_DIR, with no
daemon. Basis for a bespoke file-store clipboard backend replacing clipcat
(dotfiles-8il, dotfiles-i9i, dotfiles-apl all unresolvable at the
clipcatctl surface).

## method
Scratch /tmp/poc-clipnotify (discarded). clipnotify 1.0.2 built from source
(gcc, -lX11 -lXfixes; packaged in Arch extra as clipnotify 1.0.2-3, so the
real install is a pacman dep). Store loop: while clipnotify -s clipboard;
do xclip -o > $STORE/$(seq).clip; done on throwaway Xvfb :98. Tests: 5
copies at 50ms intervals; 3 at 300ms; multiline+literal-backslash-n
byte-exact via od.

## result
VALIDATED, exceeding acceptance. T1: all 5 rapid 50ms copies captured as
ordered distinct files — no coalescing, no drops (XFixes events queue in
the X connection buffer during the xclip read). T2: paced copies all
captured. T3: byte-exact including real \\n, real \\t, and a literal
backslash-n as two bytes — the exact case irreversible through clipcatctl
(dotfiles-i9i). File store gives: deterministic newest-first ordering
(seq/mtime, beats 8il), id-keyed get (filename, cat = raw bytes, beats
i9i), memory-only at filesystem level (tmpfs dir), no daemon to go deaf
(beats apl 25% BadAtom born-deaf).

## recommendation
Adopt the bespoke clipnotify+file-store as the sp016 backend. sp016
## solution to be revised citing this poc + [[poc011]] (clipse rejected).
Design notes for the real build (not covered by this poc): the store loop
needs the same TARGETS secret-gate pre-check as clip-feed.sh before
reading the payload; empty-selection and owner-quit need graceful skip;
clipcat stays in-tree one cycle for rollback like copyq did. Build fresh
via the normal lifecycle; this loop script is throwaway.
