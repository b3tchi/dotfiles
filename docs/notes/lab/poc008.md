# PoC [[cat002]] [[board]]

## hypothesis
A minimal Wry (Rust) webview over the system WebKitGTK engine loads the akm-graph viewer (localhost:4810) and renders the force-graph Canvas2D correctly at RSS meaningfully below Servo (~430MB, poc007) and a full browser — the lightest host that still runs the viewer as-is.

## method
Scratch /tmp/poc-wry (throwaway). cargo project: wry 0.45 + tao 0.30, ~30-line main.rs (WebViewBuilder::new_gtk on tao default_vbox, --url). System deps present: webkit2gtk-4.1, libsoup-3.0. cargo build --release = 2m21s, binary 1.4MB. Ran on display :0 (AMD radeonsi hardware X11) against live akm-graph-d on :4810 (44 nodes/156 links). Measured RSS via ps across poc-wry + WebKit web/network processes; screenshotted :0.

## result
VALIDATED render — cleaner than Servo. Full force-graph renders correctly: nodes, links, HTML label pills, legend chips, status; window has NO browser chrome (no url bar/tabs). Layout well-spread on landscape :0. RSS ~400MB total across 3 procs (poc-wry main 131MB + WebKitWebProcess + WebKitNetworkProcess). Binary 1.4MB. NO crash.

## recommendation
RENDER validated, but the minimal-RAM hypothesis is INVALIDATED: Wry ~400MB is on par with Servo ~430MB (poc007). WebKitGTK is a full browser engine — the ~400MB floor holds across BOTH engines for this page; the 1.4MB binary is misleading (engine is pre-installed but not free). No webview meaningfully cuts RAM vs a browser, because the engine IS the cost. What Wry DOES buy over a browser/Servo: a dedicated chrome-less app window (cleanest UX of the three tested), tiny binary, engine already on the box. So a webview host here is UX polish (native window, no tab clutter), NOT a resource win. Since force-graph already solved the original GPU heat (dotfiles-olm), adopting Wry is optional. IF adopted: Wry is the pick over Servo (lighter binary, prebuilt engine, cleaner window, cargo-buildable). For an actual RAM cut, only a non-web-engine native GUI (egui/gtk, ~tens of MB) qualifies — at the cost of rewriting the viewer and losing force-graph/HTML.
