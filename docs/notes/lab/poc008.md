# PoC [[cat002]] [[board]]

## hypothesis
On :10 (xorgxrdp, no compositor) a GTK/GDK window whose X Shape BOUNDING
region is cut down to just a rubber-band outline can track a live drag over
the real desktop, capture mouse + Esc, and never present a full-screen black
frame — removing the need for the frozen-PNG design and the blink it causes
(dotfiles-02h).

## method
Scratch dir /tmp/poc-live-region (throwaway, no worktree — this proves X
server + GDK behaviour, not repo code). ~130-line python3 + gi (Gtk 3.0) +
pycairo script, same stack as quickshell/qs-focus-border.py. Driven by a
synthetic xdotool drag 600,400 -> 1800,1000 while recording :10 at 60fps
(ffmpeg x11grab, lossless ffv1) and reading per-frame luma via signalstats
YAVG. Outline rendering verified separately with a full-res mid-drag scrot,
since a 2px line is sub-pixel in the 480x270 trace. Versions: GTK 3, xorgxrdp
virtual Xorg on :10, no compositor running.

## result
VALIDATED on all three questions.
1. Live outline: renders crisply over the LIVE desktop — full-res mid-drag
   crop shows the #16a085 outline composited over readable, moving terminal
   text. accent_pixel_fraction=0.0098 of the crop ~= 11k px, matching a
   3600px perimeter at 2px + antialiasing. Desktop is NOT frozen, NOT dimmed.
2. Input: GRAB success -> SELECTED 1200x600+600+400, exactly the synthetic
   drag. Esc handled on the same grab.
3. Blink: black_frames=0, luma flat 42.12..42.42 across the whole 6s run,
   INCLUDING a cold python/GTK start. Compare the shipped frozen-PNG overlay:
   3-7 full-black frames (RGB mean 1.37/255) per open, 67-150ms.

Three findings that cost most of the time and must not be re-learned:
- WSLg leaves a wayland-0 socket in XDG_RUNTIME_DIR and GTK auto-connects to
  it even with WAYLAND_DISPLAY UNSET. The window silently lands on the idle
  WSLg compositor: invisible on :10, zero events, and a FAKE "no blink" pass.
  Any GDK tool here must force GDK_BACKEND=x11.
- X intersects the INPUT region with the BOUNDING region. A window shaped down
  to an outline canNOT receive events in its hollow middle, no matter what
  input_shape_combine_region says. Setting input to full-screen is not enough
  (measured: zero events). A seat/pointer GRAB bypasses shape and is required
  — this is what slop does.
- An EMPTY bounding region makes the window not viewable, and X refuses to
  grab to a non-viewable window (GRAB not-viewable). Keep >=1px shaped. Also
  the grab must run AFTER the map, not from realize (same error otherwise).

## recommendation
Adopt the shaped live-overlay approach IF the dim is expendable — that is the
real trade, and it is a UX decision, not a technical one. Shape is binary: a
pixel is window or not. Dimming the desktop outside the selection requires
blending with what is behind, which genuinely requires a compositor. So:
live+no-dim, or frozen+dim. Cannot have both without picom, and picom on :10
forces --no-use-damage (full-frame RDP re-encode glitches) — see the
compositor-free design.

Implementation must be GDK/python, NOT quickshell/QML: quickshell PanelWindow
exposes no X Shape access, which is precisely why the QML overlay had to
freeze the screen and fake translucency, and why qs-focus-border.py is already
GDK. Precedent is in-repo and running.

Note for the "just run it inside the existing quickshell, like mod+d" idea:
that solves cold start but NOT this — a warm quickshell still maps a
fullscreen opaque window and still needs the scrot freeze (~200ms) + decode.
The shape approach needs no warm process at all: cold GDK start measured zero
blink, because no opaque window ever maps. Warming is a workaround for a cost
the shape design does not have.

Does NOT address dotfiles-ed3 (exit expose storm) — but largely dissolves it:
with no fullscreen window mapped, there is no unmap to trigger the storm.
Worth re-measuring if this is adopted.
