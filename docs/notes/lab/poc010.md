# PoC [[cat002]] [[cat001]] [[board]]

## informs

[[sp014]] (archived, `status: done`) — this PoC re-opens the backend choice
behind the shipped shared-clipboard-history system.

## hypothesis
clipcat 0.25.0 can replace copyq 16.0.0 as the [[sp014]] shared-clipboard-
history backend: it drops password-manager copies from history, and its
history can be kept entirely off persistent disk — the two properties the
shipped copyq backend guarantees via its `x-kde-passwordManagerHint` automatic
command and per-tab `store_items=false`.

## method
Throwaway scratch dir (discarded), no repo file touched. `clipcat 0.25.0-1`
and `xorg-server-xvfb 21.1.24-1` extracted from their pacman packages into the
scratch tree and run from there — neither is installed on this host and no
sudo was used. Every daemon ran on its own Xvfb (`:96`–`:98`) with isolated
`XDG_CONFIG_HOME` / `XDG_CACHE_HOME` / `XDG_RUNTIME_DIR`, never on the live
i3 session, the live clipboard, or the real `~/.config/clipcat`.

Selection owner reused from `copyq/test-copyq.sh` (the python-xlib
multi-target KeePassXC simulation), extended to log every `SelectionRequest`
target — which turns "was the item dropped" into the sharper "was the payload
ever even requested". `clipcatctl insert` was deliberately NOT used to seed
capture tests: the epic already paid for that false-pass trap three times.

Three harness bugs were found and fixed before any result was trusted, each of
which had produced a plausible-looking but wrong run:

1. backgrounding a shell *function* makes `$!` the subshell, so the killed pid
   was not `clipcatd`; an orphaned daemon then silently served every later
   phase. Fixed by backgrounding `env` directly and also killing the pid file.
2. `run_cell` invoked inside `$(...)` lost its owner-kill to the subshell.
3. clipcat takes clipboard ownership when it drops a sensitive item, so cells
   contaminated each other until each got a fresh Xvfb and a unique payload.

Every assertion below is from the run after all three were fixed, and each
question carries a negative control that was observed to behave oppositely.

## result
VALIDATED on both blocking questions, with one caveat and one residual risk.

Q1 — password-manager copies ARE dropped, and the payload is never even read.
`sensitive_mime_types` matches the advertised X11 target by EXACT atom name;
there is no prefix normalisation. Cross-matrix, unique payload + fresh X per
cell:

| sensitive_mime_types | owner advertises | result |
|---|---|---|
| `["x-kde-passwordManagerHint"]` | `x-kde-passwordManagerHint` | dropped |
| `["application/x-kde-passwordManagerHint"]` | `x-kde-passwordManagerHint` | CAPTURED |
| `["x-kde-passwordManagerHint"]` | `application/x-kde-…` | CAPTURED |
| `["application/x-kde-passwordManagerHint"]` | `application/x-kde-…` | dropped |
| `[]` (CONTROL) | `x-kde-passwordManagerHint` | CAPTURED |
| `["UTF8_STRING"]` (CONTROL) | (plain copy) | dropped |

The `[]` control captures, so the fixture demonstrably delivers the item; the
`UTF8_STRING` control drops, so the filter mechanism is live. On a matching
cell clipcat requested ONLY `TARGETS` from the owner and never `UTF8_STRING`
— the secret never left the owning process. The plain-copy control on the same
config shows `REQ UTF8_STRING`, proving reads do otherwise happen. This is
STRONGER than the shipped `i3/scripts/clip-feed.sh` standard (which lets a
secret reach a 0600 tmpfile before dropping it), and stronger than the desk
research assumed. Daemon logs `Sensitive content detected, ignore it`.

Q2 — history can be kept off persistent disk, but NOT in memory only.
There is no `store_items=false` equivalent. `history_file_path` is mandatory
and is a DIRECTORY (`clips` + `header.json`), created eagerly:

- `history_file_path = "/dev/null"` → daemon REFUSES to start:
  `Could not create HistoryManager … Failed to create directory /dev/null`
- `max_history = 0` → daemon starts but captures nothing (`clips` 0 bytes);
  a working "no history" mode, useless for a picker
- clips are written CONTINUOUSLY while the session runs, not only at exit —
  the plaintext is on the filesystem the whole time, and survives SIGKILL
- pointing it at tmpfs works: capture normal, payload in the tmpfs file, and
  `clear_history_on_start = true` yields an empty history on restart

So the copyq guarantee is reproducible via a DIFFERENT mechanism —
`history_file_path` under `$XDG_RUNTIME_DIR` (tmpfs, 0700, destroyed by
systemd-logind at last logout) gives "never on persistent disk" plus "cleared
at logout" — but it depends on config discipline and logind cleanup rather
than on the daemon never writing at all. Both designs remain swap-exposed.

Secondary questions:

- Q3 no self-capture loop: `clipcatctl insert` yields exactly 1 entry, not 2.
- Q4 parseable but thin: `length` gives the count; `list` is `<hex-id>: <text>`
  truncated to a 100-char preview (`preview_length`); full text needs
  `get <id>` per entry. No JSON output. Ids are content hashes, not row
  indices, so the picker changes shape (N+1 calls, id-keyed not row-keyed).
- Q5 multi-display works: two daemons on `:97`/`:98` with separate
  `local_socket`s each captured only their own display, zero cross-talk.
- Q6 much lighter: RSS 12.3 MiB, Private 9.9 MiB, 4 linked libs (libc, libm,
  libgcc_s, ld-linux) vs copyq 63 MiB RSS / 16 MiB private / 96 libs. No Qt6,
  no KDE frameworks — which is what makes Termux/proot viable.

RESIDUAL RISK (not settled here, and the reason this is not a clean green):
the exact atom real KeePassXC puts on the wire was NOT verified — neither
copyq nor keepassxc is installed on this host. clipcat matches EXACTLY and its
default is the bare `x-kde-passwordManagerHint`, while the shipped copyq rule
matches the prefixed `application/…`. Both can be right (copyq is believed to
prefix bare atoms when mapping X11 targets to MIME names), but if that belief
is wrong the clipcat default silently leaks every password copy — a failure
that is invisible until someone greps their history.

## recommendation
Adopt clipcat as a viable replacement — no hard blocker was found — but do NOT
treat the swap as a drop-in. Three things must land with it:

1. Pin `history_file_path` under `$XDG_RUNTIME_DIR` and set
   `clear_history_on_start = true`. This replaces `store_items=false`, which
   has no equivalent. Document it exactly as `copyq/dot.yaml` documents the
   per-tab flag, including that a plaintext file exists for the whole session.
2. FIRST, settle the wire atom against a real KeePassXC copy before anything
   ships — run the cross-matrix shape above against a live password-manager
   copy and confirm which spelling appears. Exact matching plus a silent
   failure mode makes this the highest-risk item in the migration, and it is
   cheap to close.
3. Re-shape the picker for id-keyed, preview-truncated `list` + per-entry
   `get`. `quickshell/qs-clip.sh set <row>` is row-indexed today; clipcat is
   hash-keyed, so the wrapper contract changes even though it stays thin.

The motivating wins are real and measured: the Qt6/KDE chain disappears
entirely (96 linked libs → 4), private RSS drops 16 → 9.9 MiB, and the
Termux/proot blocker is removed. The security posture on secrets is better,
not worse — clipcat never reads the payload at all. The trade is that
memory-only becomes a configuration property rather than a daemon property.
