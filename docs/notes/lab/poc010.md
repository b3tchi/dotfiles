# PoC [[cat002]] [[board]]

## informs
[[sp008|cursor-driven-preview-bridge]]

## hypothesis
An image preview rendered with Qt Quick native (QML Image) costs far less RAM than rendering it through a web engine — proving the sp008 tiered-renderer design where only HTML needs QtWebEngine and image/video/code/3D use native Qt.

## method
Scratch /tmp/poc-qtimg (throwaway). Generated a 1920x1080 test PNG (ImageMagick). 12-line QML (Window + Image, PreserveAspectFit, async). Ran via qml6 on display :0 (AMD X11) — no compile. Measured RSS; screenshotted. Compared against poc009 QtWebEngine akm-graph host (~625MB) on the same box.

## result
VALIDATED. Native QML Image renders the test image correctly at RSS ~155MB, SINGLE process, NO QtWebEngineProcess / no Chromium. vs poc009 QtWebEngine ~625MB across 4 procs. ~4x lighter, one process. The 155MB is the qml6 QML-runtime + Qt Quick scenegraph + GPU-context floor paid ONCE per app — a compiled QQuickView app trims it, and additional native renderers (video/text/pdf/3D) in the SAME qml runtime add little on top.

## recommendation
Tiered-renderer design for sp008 CONFIRMED with numbers. Build the preview shell as ONE Qt Quick app (aligns with quickshell qt6): the ~155MB Qt runtime is paid once, and native renderers cover the common cases cheaply — QML Image (image), Qt Multimedia/mpv (video), Text+highlighter (code), QtPdf (pdf), Qt Quick 3D (STL/CAD). QtWebEngine is added behind a lazy QML Loader ONLY for actual HTML/web content, spiking +~450MB (Chromium) and destroyable on leave to reclaim. So a cursor sweep over images/code/3D stays ~150-250MB; only landing on an HTML file pays the Chromium tax. This kills the earlier worry that every preview costs ~400-625MB — only HTML does. Combined with poc007/008/009 (webview hosts for the graph): the akm-graph viewer itself is JS so it needs a web engine (QtWebEngine if quickshell-integrated, Wry if standalone-lean), but the sp008 file previews mostly do NOT.
