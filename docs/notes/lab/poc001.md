# PoC [[cat002]] [[board]]

## hypothesis
A user-launched `weston --backend=rdp` inside WSL2 Arch can host a rootful
Xwayland + i3 session reachable from Windows mstsc, with usable
GPU-accelerated OpenGL inside (D3D12 renderer, not llvmpipe).

## method
Scratch dir `/tmp/poc-wslg-xwayland-rdp` (external tooling, no worktree).
Stack: weston 15.0.1 (FreeRDP 3.26.0), xorg-xwayland 24.1.11, mesa 26.1.1,
i3 4.25.1, WSL2 kernel 6.6.114.1-microsoft, Intel iGPU via /dev/dxg
(no /dev/dri present). Steps: self-signed TLS cert (openssl); weston RDP on
:3390 — tried both pixman and --renderer=gl with GALLIUM_DRIVER=d3d12;
rootful `Xwayland :2 -geometry 1600x900` as client of weston socket;
i3 with default config (must unset inherited SWAYSOCK/I3SOCK or i3 grabs
sway's IPC socket); connect `mstsc.exe /v:localhost:3390`; verdict via
`GALLIUM_DRIVER=d3d12 glxinfo/glxgears` on :2; firefox WebGL aquarium as
GPU load test.

## result
Invalidated for daily use (narrow GL claim validated, approach unusable).
Works: RDP connect, weston desktop, rootful Xwayland + i3 inside; glxinfo
in session reports `D3D12 (Intel(R) Graphics)`, direct rendering yes,
GL 4.6; glxgears renders 113 FPS (GL weston) / 35 FPS (pixman weston).
Mesa 26.1 defaults to llvmpipe in WSL — GALLIUM_DRIVER=d3d12 required.
Key finding: D3D12 works with NO /dev/dri => mesa uses drisw (GPU render +
shm copy), DRI3/DRM not needed.
Fails: cursor composited in-frame (rdp backend has no cursor plane) and
encode pipeline fell 3.5 s behind ("time until next presentation ...
-3590 msec") => severe cursor lag; glxgears window black (present path
broken under GL renderer); firefox WebGL => "D3D12: Removing Device." =>
weston SIGSEGV, session lost, stale GPU artifact left on Windows host
(cleared with Ctrl+Shift+Win+B).

## recommendation
Reject upstream weston RDP backend as accelerated daily desktop on WSL2:
in-frame cursor lag is inherent, GL-renderer-on-d3d12 crashes under WebGL
load, app GL presents black windows. WSLg-fork performance (RAIL/VAIL
shared-memory) is not reproducible with upstream weston over loopback RDP.
Transferable finding: d3d12 via drisw needs no DRM node => should work
under any X server including xorgxrdp. Next approach: xrdp + xorgxrdp
(client-side cursor, native i3 session, desktop-tuned encode) with
GALLIUM_DRIVER=d3d12 — hypothesis for a follow-up PoC.
