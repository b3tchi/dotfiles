# PoC [[cat002]] [[board]]

## hypothesis
Under an xrdp + xorgxrdp session in WSL2 Arch, GALLIUM_DRIVER=d3d12 gives
GPU-accelerated GL to apps (drisw presentation, no DRM node needed — the
transferable finding from [[poc001]]), and xrdp's client-side pointer
eliminates the cursor lag that invalidated the weston RDP approach.

## method
Scratch dir /tmp/poc-xrdp-d3d12. Stack: xrdp 0.10.6-1 + xorgxrdp 0.10.5-1
(AUR), xorg-server 21.1.22, mesa 26.1.2, i3 4.25.1, WSL2 kernel
6.6.114.1-microsoft, Intel iGPU via /dev/dxg (no /dev/dri). Setup: port
3391 in /etc/xrdp/xrdp.ini (3389 risks Windows host RDP collision);
~/.xinitrc exporting GALLIUM_DRIVER=d3d12 + unset SWAYSOCK/I3SOCK + exec
i3 (xrdp startwm.sh sources ~/.xinitrc first on Arch); systemctl start
xrdp xrdp-sesman; mstsc.exe /v:localhost:3391, Xorg session type, PAM
login. Session lands on DISPLAY=:10. Verdicts via glxinfo/glxgears and
firefox (WebGL aquarium, about:support). Gotcha: terminals auto-attach
tmux whose server env lacks GALLIUM_DRIVER — apps launched there silently
get llvmpipe.

## result
Acceleration INVALIDATED; un-accelerated desktop validated.
- glxinfo reports D3D12 (Intel) with direct rendering — but misleading:
  GLX swapbuffers presents BLACK windows (renders ~97 FPS unseen);
  llvmpipe gears present fine (~300 FPS visible via MIT-SHM PutImage).
  Reproduced after full VM reboot with healthy GPU channel (82 FPS black
  vs 261 FPS visible) — inherent, not poisoned-state artifact.
- firefox + MOZ_X11_EGL=1 + d3d12: segfault — 'libEGL warning: DRI3
  error: Could not get DRI3 device' then GPU-process crash.
- Repeated d3d12 attempts poisoned the WSL2 GPU paravirt channel
  globally: dmesg 'dxg: dxgvmb_send_create_sync_object: failed -75',
  'dxgkio_query_adapter_info: Ioctl failed: -22'; firefox then crashed in
  BOTH the WSLg sway session and the xrdp session. Recovery requires wsl
  --shutdown.
- Plain llvmpipe desktop: snappy, client-side cursor (no weston-style
  lag), WebGL aquarium ~17 FPS @ 1000 fish (light use only). Overall feel
  'pretty snappier' than weston RDP.
- Real-world 3D over xrdp/llvmpipe: PrusaSlicer (GDK_BACKEND=x11 needed,
  else GTK probes the WSLg wayland-0 socket and escapes to a WSLg
  window) — viewport rotation smooth on simple models, heavy-poly slow
  but still usable.
- Baseline comparison: the same heavy model in the regular WSLg sway
  session performs similar to xrdp/llvmpipe — WSLg's GPU path adds
  little for this workload (its libEGL probe also fails on missing
  /dev/dri, so it likely falls back to software GL too).

## recommendation
Reject GPU acceleration for any custom display stack on WSL2: mesa d3d12
presentation only works inside WSLg's blessed Xwayland path — elsewhere
GLX presents black, EGL crashes on missing DRI3, and failures poison
/dev/dxg for the whole VM (combined with [[poc001]], two independent
stacks confirm). Accept xrdp + xorgxrdp + i3 + llvmpipe as the workable
full-desktop-over-RDP option for WSL2: good interactive feel, stable,
no GPU. Do not set GALLIUM_DRIVER=d3d12 in xrdp sessions. Future angles
if acceleration ever matters: VirtualGL-style offscreen render + readback,
or wait for mesa d3d12 DRI3/present fixes upstream. Kept running: xrdp
services on :3391 + ~/.xinitrc (promoted from PoC residue to setup by
user choice).
