# Story [[cat002]] [[product]]

## role
[[pn001|i3 desktop user]] switching between the native :0 session and an
xrdp :10 session (Android RDP client)

## want
the clipboard-history picker to show ONE merged history on both displays —
a copy made on native :0 appears in the :10 picker and vice-versa

## because
the sp016 file store bridges one-way only (clip-feed.sh :10 -> :0, so :0 is
the merged history). Verified live: an Android RDP copy shows in the :10
picker, but native :0 copies never reach the :10 picker. A user hopping
between the phone RDP session and the desk expects the same clipboard on
both, not a half-shared one.

## acceptance_criteria
- a copy captured on :0 is retrievable from the :10 picker, byte-exact, and
  vice-versa; each entry appears exactly once per session (no duplicate from
  the bridge round-trip)
- BOTH secret gates still fire on every path — a hint-atom copy is dropped
  before it can cross either direction (the sp016/wtr gates, symmetric)
- NO feedback loop: a file-write bridge must not re-trigger the destination
  clip-store loop into re-capturing and re-bridging (clip-feed already
  writes a store file, not an X selection — preserve that property both ways)
- a defined merge-order rule when :0 and :10 both have entries (two
  independent seq counters -> the newest-first list needs a cross-source
  ordering key; must not reintroduce the non-determinism dotfiles-8il
  motivated the whole pivot to avoid)
- memory-only preserved: nothing crosses to persistent disk; a session
  teardown (logind) still clears its own store
- design must weigh two shapes and pick one in spec: (a) a second feeder
  :0 -> :10 mirroring clip-feed.sh, two stores kept in sync; (b) a single
  shared store both sessions read/write, dropping per-display keying
  (revisits adr0004 two-display model) — record the choice as an ADR
- works when only one session is live (no :10 present -> :0 behaves exactly
  as today, no errors, no orphaned bridge)
