# ADR [[cat001]] [[product]]

## title
Nushell as primary language for scripts and data manipulation

## context
This dotfiles repo runs across native Linux, WSL, proot Arch, and Termux, with a workflow-tooling layer (project registry, WM workspace queries, tmux session management, shell startup) that constantly touches structured data: YAML registries (`projects.yaml`), JSON from WM IPC (`i3-msg -t get_workspaces`), TOML configs, and tables piped between commands.

Options surveyed:
- **Bash** ŌĆö universal, system default, POSIX. Weak at structured data; everything is string-parsing with `awk`/`sed`/`jq` glue. Acceptable for trivial wrappers; fragile beyond.
- **Python** ŌĆö strong stdlib, but heavy startup cost, virtualenv friction, and an awkward fit for short interactive shell-style pipelines.
- **Nushell** ŌĆö structured pipelines as first-class concept, native JSON/YAML/TOML readers, table type, modules (`use`, `export def`, `export-env`), explicit typing. Not POSIX, smaller community, breaking changes between versions.

Bash is already locked in as the system shell (tmux lifecycle in `distro/bashrc`, profile loading). The question is what owns the *workflow-tooling* layer above that.

## decision
**Default: nushell. Bash only when spawning nushell is not worth it.**

Nushell owns any script or module that handles structured data or composes the workflow-tooling layer ŌĆö project registry ([[ft001]]), WM IPC (`wm-ipc.nu`), workspace utilities (`ws-list.nu`, `ws-switch.nu`), and anything else under `nushell/scripts/` or `nushell/actions/`.

Bash is reserved for the cases where invoking nushell costs more than the task itself:
- system-level shell startup (`profile`, `distro/bashrc`) ŌĆö runs before nushell exists in the session
- tmux session lifecycle ŌĆö bash is the outer shell that spawns tmux
- POSIX-required entry points (e.g. `install-rotz.sh`, scripts under `scripts/ubuntu/`) ŌĆö must run before nushell is installed, or in environments where it isn't
- one-liners and throwaway glue ŌĆö where the nushell startup cost dwarfs the work

Everything else is nushell.

Nushell runs inside tmux as the default interactive shell; bash is the outer layer.

## consequences

Positive:
- Structured data is first-class ŌĆö no `awk`/`jq`/`yq` chaining for YAML, JSON, tables.
- Modules (`mod.nu`, `export def`) give a real namespace boundary and discoverable command surface.
- `export-env` enables clean shell-state composition (auto-cd on project workspace detection).
- Consistent ergonomics across all platforms where nushell is installable (Arch, Manjaro, Termux, WSL).

Negative:
- Nushell is pre-1.0 ŌĆö breaking syntax changes between minor versions force periodic script audits. The repo must pin or test against a known nu version on each platform.
- Cannot be the system default shell (not POSIX) ŌĆö bash stays in the boot path, so every interactive session pays a `bash ŌåÆ tmux ŌåÆ nu` startup cost.
- Smaller community than bash/python ŌĆö fewer drop-in answers for niche problems; some workflows require writing primitives that exist for free elsewhere.
- Second runtime to install on every platform (rotz handles this, but it's an additional install step on bootstrap).
- Lock-in: most of `nushell/scripts/` and `nushell/actions/` couple to nushell module conventions; migration away would mean rewriting the workflow-tooling layer.
