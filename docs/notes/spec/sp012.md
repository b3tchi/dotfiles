# Spec [[cat002]] [[board]]

## extends [[ft006]]

## problem
Region screenshot ships today with NO AKM record — no `us###`, no `im###`,
no `ft###`. It arrived straight from commits (8b89148 "region screenshot
selector, replaces ksnip bind") and never entered the lifecycle. That gap is
the root of the drift found while debugging it: the tool ignores conventions
the rest of quickshell already established (per-DISPLAY scoping, session-scoped
kills), carries a `for_window [title="qs-region"]` rule that never matches
(PanelWindow sets no title), and until now had its keybind stranded in
`i3/config.d/native.conf` so WSL/xrdp could not reach it at all
(dotfiles-13q, fixed 08cd2cf).

This spec registers the capability as a new `[[ft###]]` (minted at
spec-writing) and replaces its mechanism.

**Capability boundary.** Region capture -> PNG in `~/Pictures/screenshots` +
the file PATH on the clipboard. One `providing`, one `api_surface`. The
`i3-scrot` Print binds (`Print`, `$mod+Print`, `$mod+Shift+Print` in
`i3/config`) are a SECOND, older capture path and are deliberately OUT of
scope — the duplication is noted, not solved here (YAGNI). `ksnip` is not a
capture path: it is annotation-only (`ksnip -e <file>`), a consumer of the
output.

**Consumers.** The i3 keybind layer across every X11 platform that runs the
shared base config — native :0, xrdp :10 (WSL + desktop), proot/Termux:X11.
Honest caveat: that is arguably ONE consumer class, which the idea-feature
rule ("a feature with one consumer is not a feature") warns against. It is
registered as a Feature on the [[ft003]] precedent — a platform-spanning
environment capability consumed by every platform running i3 — and because no
`us###` exists to hang an `im###` from. Flagged deliberately rather than
silently.

**Problem with the current mechanism.** `quickshell/region/shell.qml` is a
fullscreen opaque PanelWindow showing a frozen `scrot` PNG dimmed 0.35:

1. OPEN BLINK (dotfiles-02h) — the whole screen goes black 67-150ms
   (RGB mean 1.37/255) before the frozen image paints. Measured, 60fps luma
   trace; harness committed at `quickshell/test-blink-measure.sh`.
2. EXIT FLASH (dotfiles-ed3) — unmapping the fullscreen window triggers an
   expose storm; underlying apps repaint (Firefox clears white -> luma 42 to
   76-80). Content-dependent, so it is intermittent.
3. CROSS-SESSION COLLISIONS (dotfiles-8xt) — `${XDG_RUNTIME_DIR}/qs-shot-src`
   is per-USER not per-display, so :0 and :10 clobber each other; the fallback
   `pkill -f "quickshell -p .*region"` is global and kills the other display;
   the IPC selects the instance by config path only.
4. HINT STRIP IN THE GRAB (dotfiles-d7v) — the bind enters the i3 mode BEFORE
   `scrot` runs, so the bar paints its hint strip into the frozen grab.

**Why the obvious fixes are dead.** Both were measured, not argued:
- Pre-warming / hosting inside the running quickshell (the `mod+d` launcher
  pattern) does NOT help: [[poc009]] — a process warmed 3s still blinks
  ~100ms, inside the cold-start range. Cold start was never the cause. The
  cost is the fullscreen opaque MAP (X clears to the black background pixel)
  plus Qt first-paint of a 2560x1440 image on the software backend, both paid
  per-open regardless of process age.
- Staying in QML at all is dead: quickshell PanelWindow exposes no X Shape.
  That is precisely WHY the overlay freezes the screen and fakes translucency.

**Proposed mechanism.** GDK + X Shape live overlay — [[poc008]], validated:
black_frames=0 including a cold start, drag tracked live over the real
desktop, seat grab for input, no `scrot` freeze (the grab happens AFTER
release). Precedent in-repo: `quickshell/qs-focus-border.py` already draws a
shaped, compositor-free ring on both :0 and :10.

**Contract preserved.** `$mod+Shift+s`, the i3 `screenshot` mode, the
`Bar.qml` `modeHints` strip, drag + 2-tap select, `w` whole / `Esc` cancel,
output to file + clipboard path, `ksnip -e` annotation.

**What is dropped, deliberately.** The 0.35 dim + bright hole. Shape is binary
per pixel; blending with what is behind genuinely requires a compositor, and
picom on :10 forces `--no-use-damage` (full-frame RDP re-encode glitches).
User decision: the dim is expendable because the i3 mode + bar hint already
carry the mode signal. Also dropped: `quickshell/region/` entirely, and the
`swaymsg` branches in `qs-shot-action.sh` / `shell.qml` — dead code for a path
nothing binds.

**Constraints the PoCs proved (each cost real debugging time):**
- `GDK_BACKEND=x11` is MANDATORY. WSLg leaves a `wayland-0` socket in
  `XDG_RUNTIME_DIR` and GTK auto-connects even with `WAYLAND_DISPLAY` unset —
  the window lands on an invisible compositor and fakes a clean pass.
- X intersects the INPUT region with the BOUNDING region, so a shaped window
  receives no events in its hollow middle. A seat GRAB is required (as slop
  does), it must run AFTER the map (not from `realize`), and >=1px must stay
  shaped or the grab fails `not-viewable`.
- Outline colour must be the MODE colour `#cb4b16` (`Bar.qml:576/589`), NOT
  the `#16a085` workspace-focused green the PoC borrowed.
- Per-DISPLAY scoping must follow `quickshell/qs-session.sh` (`QS_SID`,
  `qs_same_session()`, `qs_kill_session()`).
- Native :0 needs `qs-live-region` added to `i3/picom.conf` `focus-exclude`
  AND `unredir-if-possible-exclude` BEFORE it is tried — picom matches by
  name, the overlay never takes focus by design (the exact trap 935e568 hit),
  and unredir on a full-screen-geometry window is a hard whole-screen flash.

**Categories + binding decisions.** [[cat002]] (display-platform), following
[[ft003]]. No ADR is overturned. [[adr0005]] (no GPU outside WSLg; everything
renders on llvmpipe) actually FAVOURS this design — shaping a thin outline is
far cheaper than software-rasterising a fullscreen image, which is the
measured ~100ms. [[adr0004]] (xrdp + xorgxrdp + llvmpipe) is the environment
it runs in. [[adr0006]] is unrelated. The compositor-free decision ("no picom
on xrdp; overlays use X Shape, not alpha") has no `adr####` despite governing
this whole design — it lives only in code comments and commits 935e568 /
c984834. DECIDED at spec-writing: not minting one; the in-repo comments are
considered sufficient. Recorded so this is not re-raised as an oversight.

**Dedup.** [[ft001]] project registry, [[ft002]] d2-preview-router, [[ft003]]
i3-config-layering, [[ft004]] akm-graph, [[ft005]] file-preview-daemon — none
cover screen capture. No existing `im###` ([[im001]]..[[im005]]) implements or
consumes capture; there is nothing to migrate.

**Platform scope.** i3/X11 only. xrdp :10 is PROVEN (all PoC measurement ran
there). Native :0 is EXPECTED to work (X Shape is compositor-independent;
`qs-focus-border.py` is precedent) but is UNVERIFIED and unverifiable from
WSL, which has no :0 — it needs an explicit task on the desktop machine, not
an assumption. proot/Termux:X11 untested. Sway is NOT APPLICABLE: nothing
binds `qs-screenshot.sh` there. Note for any future sway work — Wayland has no
X Shape so this design cannot port, but Wayland HAS a compositor, so real
alpha works and it wants the opposite approach.

**Absorbs:** dotfiles-02h (dissolves — no opaque fullscreen map), dotfiles-d7v
(dissolves — nothing captured before selection), dotfiles-8xt (designed out —
spawn-per-invocation, $DISPLAY inherited, no shared state, no daemon).
Relates: dotfiles-ed3 (largely dissolves — no fullscreen unmap to trigger the
storm; re-measure rather than assume).

## solution
Replace the frozen-PNG QML overlay with a **GDK + X Shape live overlay**,
minting [[ft006]] (region-capture) as the registered capability this spec
extends. Approach validated end-to-end by [[poc008]]; the obvious cheaper
alternative was measured and killed by [[poc009]].

**Mechanism.** A GDK window spans the screen but its X Shape BOUNDING region is
cut down to just the rubber-band outline, so every unselected pixel is simply
not part of the window and the real desktop shows through — live, undimmed,
unfrozen, with no compositor involved. Input cannot come from the shape (X
intersects the INPUT region with the BOUNDING region, so a hollow window
receives nothing in its middle); a seat GRAB supplies it instead, as `slop`
does. The capture happens AFTER release, so there is no `scrot` freeze at open
and nothing is grabbed if the user cancels. Same stack and technique as
`quickshell/qs-focus-border.py`, which already proves shaped compositor-free
drawing on both :0 and :10.

**Why not the alternatives.** Staying in QML is impossible, not merely slower:
quickshell `PanelWindow` exposes no X Shape, which is exactly why the current
overlay freezes the screen and fakes translucency. Hosting it in the running
quickshell (the `mod+d` launcher pattern) was the cheapest-looking option and
[[poc009]] invalidated it — a process warmed 3s still blinks ~100ms, because
the cost is the opaque fullscreen MAP plus Qt first-paint, both paid per-open
regardless of process age. Real translucency would need picom, which on :10
forces `--no-use-damage` and the full-frame RDP re-encode glitches that got it
removed.

**Consumed capabilities + binding decisions.** [[cat002]] display-platform.
[[adr0005]] (no GPU outside WSLg; everything renders on llvmpipe) FAVOURS this
shape — outlining a rectangle is far cheaper than software-rasterising a
2560x1440 image, which is the measured ~100ms. [[adr0004]] (xrdp + xorgxrdp +
llvmpipe) is the environment it runs in. [[adr0006]] is unrelated. No ADR is
overturned, so no supersession is required. No existing `im###`
([[im001]]..[[im005]]) covers capture — nothing to dedup against or migrate.
The compositor-free commitment stays comment-documented by decision (see
`## problem`).

**Trade taken, explicitly.** The 0.35 dim + bright hole is dropped. Shape is
binary per pixel; dimming the desktop outside the selection means blending with
what is behind it, and blending is the one thing that genuinely requires a
compositor. The mode signal moves entirely to the i3 `screenshot` mode + the
`Bar.qml` `modeHints` strip, which already render it today. The selection
outline uses the MODE colour `#cb4b16` (`Bar.qml:576/589`) rather than the
`#16a085` workspace-focused green, so the overlay and the bar agree.

**Preserved contract.** `$mod+Shift+s`, the i3 `screenshot` mode, the bar hint
strip, drag + 2-tap select, `w` whole / `Esc` cancel, PNG in
`~/Pictures/screenshots`, the file PATH on the clipboard, `ksnip -e`
annotation. Users should notice only that the blink and the dim are gone.

**Shape of the work** (task breakdown belongs to spec-refinement): a new GDK
overlay replacing `quickshell/region/`; per-DISPLAY scoping adopted from
`quickshell/qs-session.sh` (`QS_SID`, `qs_same_session`, `qs_kill_session`)
which designs out dotfiles-8xt; `i3/picom.conf` exclude entries landing BEFORE
native :0 is attempted; dead `swaymsg` branches removed; the stale
`for_window [title="qs-region"]` rule (which never matched — `PanelWindow` sets
no title) corrected or dropped. dotfiles-02h and dotfiles-d7v dissolve rather
than get fixed; dotfiles-ed3 largely dissolves (no fullscreen unmap to trigger
the expose storm) but must be RE-MEASURED with
`quickshell/test-blink-measure.sh`, not assumed.

**Known open risk.** Native :0 is unverified and unverifiable from WSL (no :0
exists there). Every measurement in [[poc008]] / [[poc009]] ran on xrdp :10.
spec-refinement must carry this as an explicit task on the desktop machine —
not an assumption — and the picom excludes must land first, or the overlay gets
`inactive-dim`ed (the trap commit 935e568 already hit once) and unredir may
fire a whole-screen flash.
