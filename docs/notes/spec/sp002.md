# Spec [[cat001]] [[board]]

## solves
[[us001|mark project as remote so terminals open against the remote host]]

## implements
[[im002|remote project dispatch via tmux-to-workstation]]

## problem

[[us001]] shipped via [[sp001]] / [[im002]]: `project go <name>` dispatches
to [[ft001]] which branches on the optional `ssh` field — local entries
follow today's `cd $path + tmux-start` flow, remote entries delegate to
`tmux-to-workstation connect --cwd <path> -- <ssh>`.

The us001 acceptance criterion *"opening a remote-flagged project lands
automatically in that working directory on the remote host, no manual
connection step"* says *"opening"*, not *"running `project go`"*. The
WM keybinding **mod+Return** is the other primary way [[pn001]] opens a
project terminal:

```text
# sway/config.d/default:119  + i3/config:38
bindsym $mod+Return exec \
  TMUX_TARGET=$(~/.local/bin/tmux-start start 0 \
    $(~/.local/bin/wm-current-workspace)) $term
```

The binding pipes the focused-workspace name into `tmux-start`,
unconditionally local. When the focused workspace's name matches a
remote-flagged project entry in `projects.yaml`, mod+Return still spawns
a **local** terminal at the **local** workspace, not the remote one.
The us001 promise breaks for the keybinding path; [[pn001]] has to drop
into a shell and run `project go <name>` to honor the remote flag.

This spec extends the [[us001]] guarantee to also cover the WM-keybinding
trigger: mod+Return on a remote-flagged workspace must land in the
remote project dir over ssh, no manual step.

### scope of the delta

- Affected `us###`: [[us001]] — AC stays semantically identical
  (*"opening"* already covers WM keybinds), but the AC test surface
  widens from the `project go` flow to also include mod+Return. No new
  AC bullet needed; spec-refinement covers the keybind path in the
  task test_plan.
- Affected `im###`: [[im002]] — body refresh, not supersession. The
  dispatch primitive (`tmux-to-workstation connect`) is unchanged; the
  new trigger composes it from a different entry point (WM bindsym
  instead of `project go`). `## components` and `## approach` gain a
  WM-keybinding paragraph; `## api_surface` names the new dispatcher
  script. Body refresh lands at spec-retro.
- Affected `ft###`: [[ft001]] — additive widening. The project module
  exposes a `lookup <name>` helper (or makes the existing `data`
  public) so the WM-side dispatcher reads the registry through ft001
  rather than re-parsing yaml. Status stays `stable`.
- Affected `adr####`: [[adr0001]] (nushell-primary) honored — the new
  dispatcher lands in `.nu`. [[adr0002]] (bash carve-outs) — sway/i3
  config files are not bash, the bindsym `exec` line stays a small
  one-liner that calls into the nu dispatcher; this spec does not add
  bash. No ADR overturn.

### supersession-or-refresh decision

**Body refresh on `im002`, not supersession.** The accepted shape stays
intact: branch on `ssh` field, delegate remote to `tmux-to-workstation
connect --cwd`. The change is an additive widening: a second trigger
(WM bindsym) consumes the same dispatch primitive. Superseding `im002`
would imply the original shape was wrong, which it isn't — it just
didn't cover this opening mode.

`ft001` widens additively. Adding a `lookup` helper does not change any
existing public surface; consumers of `data` (currently private) /
`list` / `add` / `go` / `export-env` keep working unchanged.

### already-shipped data

None. The existing `projects.yaml` schema is unchanged. Local entries
keep working with today's bindsym. Remote entries gain the new
behavior. No migration.

### constraints

- [[adr0001]] nushell-primary — dispatcher in `.nu`.
- [[adr0002]] bash carve-outs — sway/i3 config line stays a one-liner;
  any non-trivial branching moves into a nu helper invoked from the
  bindsym.
- mod+Return today returns a `TMUX_TARGET` env that `$term`'s bashrc
  uses to attach. The remote flow (via `tmux-to-workstation`) creates a
  local thin tmux session on `tmux -L thin` whose name (e.g. `thin_5`)
  does not exist on the default tmux server. The solution extends
  `tmux-to-workstation` with a `--no-attach` mode that prints the thin
  session id and exits without attaching, then patches `distro/bashrc`
  to pattern-match `TMUX_TARGET` values matching `^thin_` so the attach
  uses `-L thin`. Both branches share the same `TMUX_TARGET` env-protocol.

## solution

Branch the `$mod+Return` bindsym on the focused workspace's project
registry entry — local entries stay on today's `tmux-start` path;
remote entries dispatch via `tmux-to-workstation connect`. The bindsym
remains a single `exec` line (per [[adr0002]]); all branching logic
lives in a new nushell action.

**Composes existing primitives** (no new dispatch surface):

- [[ft001]] (project registry and navigation module) — widens
  additively with a `lookup <name>` helper (or makes the existing
  private `data` callable from outside the module) so the WM-side
  dispatcher reads the registry through ft001. Status stays `stable`;
  existing public surface (`list`, `add`, `remove`, `go`, `export-env`)
  unchanged.
- `tmux-to-workstation connect -g <ws> --cwd <path> -- <ssh>` — the
  already-shipped remote dispatch primitive from [[sp001]]. Same
  invocation shape; new caller.
- `tmux-start start 0 <ws>` — today's local trigger. Untouched on the
  local branch.
- `wm-current-workspace` — already-shipped focused-workspace resolver.
  Continues to drive the dispatch.

**Trigger surface widens, dispatch primitives don't.** The new
action `nushell/actions/wm-launch-terminal` (nu, per [[adr0001]])
takes the terminal command as an argument, resolves the workspace, looks
up the entry via [[ft001]], and branches:

- **No `ssh` field** (or workspace not registered) → today's behavior:
  `TMUX_TARGET=$(tmux-start start 0 <ws>) <term-cmd>`. The terminal's
  bash login shell attaches via existing `bashrc` TMUX_TARGET protocol.
  No regression on local workspaces.
- **`ssh` field set** → `TMUX_TARGET=$(tmux-to-workstation connect
  --no-attach -g <ws> --cwd <path> -- <ssh>) <term-cmd>`. The new
  `--no-attach` flag prints the local thin session id (e.g. `thin_5`)
  and exits without attaching. `$term`'s bash login shell reads
  `TMUX_TARGET`; the patched `distro/bashrc` pattern-matches `^thin_`
  and runs `tmux -L thin attach -t $TMUX_TARGET`, landing in the thin
  pane whose ssh has already opened the remote tmux session at `--cwd`.
  Same env-protocol shape as the local branch — both set
  `TMUX_TARGET`, both attach via bashrc; the only delta is which tmux
  socket bashrc selects, decided by the `^thin_` prefix.

The sway/i3 configs change from

```text
bindsym $mod+Return exec TMUX_TARGET=$(~/.local/bin/tmux-start start 0 \
  $(~/.local/bin/wm-current-workspace)) $term
```

to

```text
bindsym $mod+Return exec ~/.local/bin/wm-launch-terminal $term
```

Both `sway/config.d/default` and `i3/config` change in the same
refinement (single trigger surface across WMs).

**Trade-offs:**

- *Dispatcher as standalone action vs `project` subcommand.* The
  dispatcher is a *trigger-side* concern (WM keybind), not a
  *registry-side* concern (project add/list/go). Keeping it outside the
  `project` module preserves the module's focus on registry CRUD +
  go-dispatch; the WM-side trigger is just another consumer of the
  registry. Cost: one extra file in `nushell/actions/`.
- *`$term` passed as argument, not inferred.* WM owns the terminal
  choice (sway uses one, i3 may use another). Dispatcher stays
  terminal-agnostic. Cost: each WM config must thread the `$term`
  variable into the call.
- *Single `TMUX_TARGET` env-protocol for both branches.* Local sets it
  to the default-server session id (e.g. `myws_0:0`); remote sets it
  to the thin-server session id (e.g. `thin_5`). `distro/bashrc`
  pattern-matches `^thin_` to dispatch the attach via `-L thin`;
  everything else hits the default server. Cost: `tmux-to-workstation`
  gains a `--no-attach` flag, and `distro/bashrc` gains a one-line
  pattern guard. Benefit: one mental model for the trigger hand-off
  regardless of where the project lives.
- *Reuse existing primitives.* No new dispatch primitive minted.
  `tmux-to-workstation connect` and `tmux-start` shipped in [[sp001]];
  this spec only adds a non-attaching exit path to the former and a
  one-line pattern guard to bashrc. Bug-fixes to the remote dispatch
  path (e.g. the `revive`-without-`--cwd` latent noted post-sp001)
  benefit both triggers automatically.
- *bashrc pattern `^thin_` is the only magic prefix.* Couples the
  `tmux -L thin` socket to the `thin_N` session-name convention
  produced by `create-thin-session`. Today's only non-default socket
  is `thin`, so the coupling is safe; if another socket is introduced,
  bashrc gains another branch. Acceptable today; tracked as a known
  rigidity.

**Dedup check.** [[im002]] already solves [[us001]] via the `project
go` trigger. This spec extends the same accepted shape (ssh-branch +
delegate to `tmux-to-workstation`) to a second trigger (WM keybind).
No new `im###` minted — body refresh on `im002` at spec-retro covers
the new trigger entry point.

**Categories.** Stays in [[cat001|workflow-tooling]] — same bucket as
[[sp001]] / [[im002]] / [[ft001]] / [[adr0001]] / [[adr0002]]. No new
ADRs (no decision shifts; the spec reuses existing constraints).

**Inherited constraints:**

- [[adr0001]] (nushell-primary) — dispatcher in `.nu`; ft001 widening
  in `.nu`; tmux-to-workstation widening in `.nu`.
- [[adr0002]] (bash carve-outs) — `distro/bashrc` patch falls under
  carve-out #2 ("system-shell startup chain"). The bindsym lines in
  sway/i3 configs are markup, not bash. No new bash scripts added.
- [[ft001]] widens additively (new `lookup` export). Status stays
  `stable`. Existing public surface (`list`, `add`, `remove`, `go`,
  `export-env`) untouched.
- Backwards-compat: local entries and unregistered workspaces keep
  today's behavior. The `^thin_` bashrc guard is a no-op when
  TMUX_TARGET does not start with that prefix.

**Anti-patterns to reject:**

- *Embedding yaml-parse logic in `wm-launch-terminal`.* Use ft001's
  `lookup` instead. Duplicating yaml-schema knowledge across modules
  is a bug surface — the [[sp001]] retro flagged exactly that risk for
  `parse_project_entry`.
- *Hardcoding the thin-socket prefix anywhere except `distro/bashrc`.*
  `wm-launch-terminal` calls `tmux-to-workstation connect --no-attach`
  and takes whatever string comes back. If `create-thin-session` later
  changes the prefix, only bashrc needs an update. Don't grep
  for `thin_` in nu code.
- *Silent fallback to local on bogus ssh profile or unreachable host.*
  `tmux-to-workstation connect --no-attach` must surface ssh errors
  the same way `project go` does today. The dispatcher must not catch
  and degrade to local cd — the workspace name asked for remote.
- *Bash wrapper around dispatcher logic.* Violates [[adr0001]] /
  [[adr0002]]. Only `distro/bashrc` may contain new bash — and only
  the one-line pattern guard.
- *Inferring `$term`.* The WM owns the terminal choice. The
  dispatcher takes it as an argument and stays terminal-agnostic.
- *Mixing local and remote tmux servers in the bashrc guard.* The
  guard only adds `-L thin` to the attach command; it does not
  touch `tmux new-session` or any other surface. Anything outside
  attach stays default-server.

**Known limitations (not addressed by this spec):**

- Workspace name must match the project name exactly (today's rule,
  unchanged). No fuzzy matching, no aliases.
- Workspace numeric-only names (e.g. `1`, `2`) fall through to local
  with the workspace name as session group (today's behavior).
- Real-remote verification requires a reachable ssh host
  (`~/.ssh/config.d/<profile>` set up).
- i3 verification requires an i3-running platform; in environments
  without one, the i3 bindsym landing remains code-only until the
  user reaches an i3 session. Task 8 documents the gap if relevant.
- The `^thin_` pattern in bashrc couples bash to the
  `create-thin-session` naming convention. If a future change renames
  thin sessions, both files move together.

- The existing `if [[ -n "$TMUX_TARGET" ]]; then ... tmux attach ... fi`
  block gains a pattern check: when `$TMUX_TARGET` matches `^thin_`,
  the attach uses `tmux -L thin attach-session -t "$TMUX_TARGET"`;

#### edge_cases
- `TMUX_TARGET=thin` (exactly, no underscore) → does NOT match

- `TMUX_TARGET=thin_5:0` (thin session with explicit window) → still

  `attach -t thin_5:0`.
- `TMUX_TARGET=myproject_thin_0:0` (project named with `thin`
  substring but not prefix) → does NOT match `^thin_`, falls through
  to default-server attach. The prefix anchor is load-bearing.
- TMUX env var (the one tmux sets when nested) coexists with our
  TMUX_TARGET. The attach block stays inside the `[[ -z "$TMUX" ]]`
  guard (or equivalent) if today's bashrc has one — no double-nest.

### Task 4: Create `nushell/actions/wm-launch-terminal` dispatcher

#### type
feature

#### effort
2h

#### depends
- Task 1 (project lookup)
- Task 2 (tmux-to-workstation --no-attach)

#### files_touched
- `nushell/actions/wm-launch-terminal`

#### success_criteria
- New nu script. Signature: `wm-launch-terminal <term_cmd: string>`.
  Resolves workspace via `wm-current-workspace`; looks up workspace
  name in registry via `project lookup`.
- **Local branch** (lookup → `null`, or entry without `ssh`):
  `TMUX_TARGET=$(tmux-start start 0 <ws>) <term_cmd>` — identical
  shape to today's bindsym one-liner.
- **Remote branch** (entry with `ssh`):
  `TMUX_TARGET=$(tmux-to-workstation connect --no-attach -g <ws>
  --cwd <path> -- <ssh>) <term_cmd>`. Both branches set
  `TMUX_TARGET` for `<term_cmd>` to consume via patched bashrc.
- The dispatcher exits the same status as the launched `<term_cmd>`.
  ssh / tmux-start errors surface visibly; no silent fallback.

#### edge_cases
- Workspace name not registered (e.g. user renamed workspace to
  something random) → `lookup` returns null → local branch fires
  with workspace name as session group. No regression vs today.
- Workspace name numeric-only (`1`, `2`) → `wm-current-workspace`
  returns it; `lookup` returns null → local branch with numeric name.
  Same as today.
- `term_cmd` contains spaces (e.g. `"wezterm-gui start"`) → must be
  passed unsplit to the inner exec. Use nu's `^$term_cmd` or
  equivalent that respects word boundaries.
- Remote `path` contains spaces (e.g. `"/tmp/with space"`) → quoted
  through to `tmux-to-workstation`; the existing single-quote
  escape there handles it. Verify end-to-end.
- `tmux-to-workstation --no-attach` fails (ssh error) → propagate
  non-zero exit; do NOT spawn `<term_cmd>` (avoid orphaned local
  terminal with no remote session to attach to).
- `wm-current-workspace` returns empty (no WM IPC / no $env.PROJECT)
  → fall back to "local" workspace name (today's behavior).

#### test_plan

  attached to local tmux session `local1_N`. PWD inside pane is

- Manual: register a remote project `ws → /tmp/myapp --ssh ws`;

- Manual: rename workspace to `random-unregistered`; same dispatch

- Manual: pass multi-word term: `wm-launch-terminal "wezterm-gui

- Manual: simulate ssh failure (use bogus ssh profile on a remote

  non-zero exit. Catches: silent fallback to local terminal.
