# Spec [[cat002]] [[cat001]] [[board]]

## problem

No shared clipboard history exists across the graphical i3 sessions. Each X
display ([[cat002]] display-platform) owns an independent CLIPBOARD, so a copy
made on the native `:0` session is invisible to the xrdp `:10` session and vice
versa, and neither keeps history â€” only the single most-recent selection
survives, and it vanishes when its owning app exits. Users want a browsable,
cross-display clipboard history with paste-under-cursor â€” a workflow-tooling
([[cat001]]) productivity capability the system provides once and every
graphical session consumes.

**Capability boundary.** A shared clipboard-history browser: every *text* copy
from any graphical i3 session lands in one in-memory history; a quickshell
window browses it and pastes the picked entry under the cursor. Distinct from
[[ft006]] (region-capture, which only drops a screenshot's file *path* on the
clipboard) â€” no dedup overlap. No existing `ft###` covers this; a new one is
minted at spec-writing.

**Backend (off-the-shelf, secret-safe).** copyq server, configured
memory-only (zero disk writes, history cleared on logout) and set to ignore
any clipboard item carrying the `x-kde-passwordManagerHint=secret` atom
(KeePassXC / Bitwarden), so password-manager copies are never captured.
Optional clear-on-`blurlock` tie-in (xautolock already runs). The copyq server
binds the platform-*primary* display: native `:0`, WSL `:10` (WSL runs only
`:10`, no native display), proot its single Termux:X11 display.

**Cross-display feeder (native-desktop only).** The native desktop is the sole
host that runs two X displays at once â€” native `:0` plus an xrdp `:10` session
serving a phone client ([[adr0004]] xrdp stack). There a bash feeder
([[adr0002]] â€” bash for tight glue, timeout-guarded like the existing
`clip-sync.sh`) watches `:10`'s CLIPBOARD and pushes changes into the copyq
server on `:0` (`:0` captured natively). WSL and proot are single-display â€”
no feeder.

**quickshell picker.** A QML history-browser window (the project's quickshell
stack) lists history via the copyq API; on select it sets CLIPBOARD+PRIMARY,
detects the focused window class via xdotool, and synthesizes `ctrl+shift+v`
for terminals / `ctrl+v` for GUI apps (layout-safe on dvorak; handles large
text; no retype).

**Consumers.** Session-layer infrastructure for every graphical i3
environment, consumed via [[ft003]] (i3-config-layering): the picker keybind
plus the copyq/feeder autostart land in the shared i3 config
(`native.conf` + `wsl.conf`).

**Migration.** No ad-hoc `im###` implements clipboard history today; the
closest prior art is the `i3/scripts/clip-sync.sh` PRIMARYâ†”CLIPBOARD bridge
(dependency-free xclip polling) whose style the feeder follows â€” a pattern
reference, not a migration target.

## solution

Deliver as a new feature `ft007` (`shared-clipboard-history`, minted with this
spec), consumed by every graphical i3 environment via [[ft003]]
(i3-config-layering) â€” the autostart and picker keybind land in the shared i3
config (`native.conf` + `wsl.conf`). Three composed parts:

1. **Backend** â€” a `copyq` server bound to the platform-primary display (native
   `:0`, WSL `:10`, proot its single Termux:X11 display), run memory-only (no
   disk persistence; history cleared on logout) and configured to drop any item
   bearing the `x-kde-passwordManagerHint=secret` atom so password-manager
   copies are never captured. copyq is taken off-the-shelf for its scriptable
   read/query API and native secret + storage controls.
2. **Cross-display feeder** â€” native-desktop only (the sole dual-display host,
   `:0` + xrdp `:10` per [[adr0004]]): a bash daemon ([[adr0002]] â€” tight
   timeout-guarded glue in the `i3/scripts/clip-sync.sh` mould) that watches
   `:10` CLIPBOARD and pushes changes into the `:0` copyq server (`:0` captured
   natively). WSL and proot are single-display â€” no feeder.
3. **quickshell picker** â€” a QML history-browser window (the project quickshell
   stack) that lists history via the copyq API and, on select, sets
   CLIPBOARD+PRIMARY, class-detects the focused window with xdotool, and
   synthesizes `ctrl+shift+v` (terminal) / `ctrl+v` (GUI) â€”
   paste-under-cursor, dvorak-safe, no retype.

The backend and secret-retention choice (copyq Â· memory-only Â· hint-drop Â·
native-desktop feeder â€” over clipcat, encrypted-disk, or continuous
live-mirror) is captured here in `## solution`; by design it is **not** promoted
to a standalone `adr####`. Binding decisions remain [[adr0002]] and [[adr0004]]
under [[cat002]] [[cat001]].

Trade-offs: memory-only history is lost on logout (deliberate â€” secrets never
persist); copyq's bundled GUI is unused; the feeder adds one native-desktop
daemon; paste-under-cursor leaves the picked entry on the clipboard (expected).

## plan

**File tree**

```
copyq/dot.yaml                     NEW  rotz install copyq + link config
copyq/copyq.conf                   NEW  memory-only settings
copyq/commands.ini                 NEW  ignore x-kde-passwordManagerHint items
i3/scripts/clip-feed.sh            NEW  native :10 -> :0 feeder (bash)
i3/scripts/clip-paste.sh           NEW  set clip + class-detect + synth paste
quickshell/config/ClipHistory.qml  NEW  history-browser UI
quickshell/qs-clip.sh              NEW  launcher (qs-*.sh pattern)
i3/config                          EDIT base: picker keybind ([[ft003]])
i3/config.d/native.conf            EDIT copyq server :0 + clip-feed.sh autostart
i3/config.d/wsl.conf               EDIT copyq server :10 autostart (no feeder)
```

**Conventions**

- Bash glue is `#!/bin/sh` + `set -u`, every `xclip` call wrapped in
  `timeout` ([[adr0002]]; the `i3/scripts/clip-sync.sh` mould).
- copyq config is version-controlled and linked via `copyq/dot.yaml` rotz
  links â€” not `dtlf.sh` (new config rule).
- The picker keybind and the quickshell autostart live in the **base**
  `i3/config`; the copyq server + feeder autostarts live in the per-platform
  **overlays** â€” exactly [[ft003]]'s api_surface split (base owns keybinds +
  qs autostart, overlays own platform autostarts).
- QML follows the existing `quickshell/config/` patterns; the launcher script
  matches the `quickshell/qs-*.sh` convention.

**Anti-patterns (hard rejects in review)**

- No bare `xclip -o` without `timeout` â€” a hung selection owner wedges the
  loop (the clip-sync.sh lesson).
- **No persistent-disk history.** History is memory-only (copyq item store on
  tmpfs / item-saving disabled); a password written to disk can leak. Verify
  nothing lands under `~/.config/copyq` / `~/.local/share/copyq` on disk.
- **Never trust inherited `DISPLAY`** in the feeder or paste script â€” derive
  it explicitly per session (the stale-DISPLAY class of bug just fixed in
  tmux). The feeder addresses `:0` and `:10` by explicit `DISPLAY=` prefix.
- No `xdotool type` for paste (dvorak / unicode / large-text hazards) â€” set
  the selection and synthesize the paste key.
- No password-manager-hint items in history â€” dropped at the copyq layer.

**Known limitations**

- History is lost on logout (by design â€” secrets never persist).
- Text only; images / rich clipboard are not captured into history.
- The hint-filter relies on the password manager marking copies with
  `x-kde-passwordManagerHint` (e.g. KeePassXC with the option enabled);
  managers that don't mark copies are not filtered.
- The feeder is native-desktop-only; if WSL ever runs two displays at once,
  revisit.

## tasks

### Task 1: copyq backend â€” install + memory-only + hint-drop

#### type
task

#### effort
6h

#### depends
none

#### files_touched
`copyq/dot.yaml`, `copyq/copyq.conf`, `copyq/commands.ini`

#### success_criteria
- `rotz install`/`link` installs copyq and links the config; a copyq server
  starts bound to the current `DISPLAY`.
- After `printf hello | xclip -selection clipboard`, `copyq read 0` returns
  `hello` (plain text captured).
- A clipboard item carrying `x-kde-passwordManagerHint=secret` is NOT
  captured: `copyq size` is unchanged after such a copy.
- After a copyq server restart (logout analogue), `copyq size` is `0` â€” no
  history persisted to disk; no files appear under the copyq data dir on
  real disk (tmpfs/disabled-save verified).

#### edge_cases
Image / non-text clipboard (skip, text-only); empty clipboard; very large
text (multi-MB) not truncated silently; two copies in rapid succession both
recorded in order.

#### test_plan
`copyq/test-copyq.sh` (bash, headless via `Xvfb :99`): (1) copy plain text â†’
assert `copyq read 0` equals it; (2) copy content with the hint atom set via
`xclip -selection clipboard -t application/x-kde-passwordManagerHint` recipe
â†’ assert `copyq size` did not increment; (3) restart server â†’ assert
`copyq size` == 0 and `find <datadir> -type f` on real disk is empty. Each
assertion names its scenario; not existence-only.
