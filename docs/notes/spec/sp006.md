# Spec [[cat001]] [[board]]

## solves
[[us006|visualize-note-relations]]

## problem
The AKM zettel graph (us / sp / im / ft / adr / cat / pn / poc plus the
product, board, and akm hub notes under docs/notes/) is only readable
file-by-file — there is no way to see the whole knowledge graph, spot
clusters, or catch dangling [[wikilinks]] visually. The `akm` CLI has
list/read/write subcommands but no graph export and no visualization
surface.

Proposed capability: a live graph-visualization daemon rendering the
full zettel graph with the GPU-accelerated cosmos library
(@cosmograph/cosmos). Approved brainstorm direction:

- Go daemon `akm-graph-d` (engine) + mandatory nushell wrapper
  `akm graph start|stop|open|status` per [[adr0003]] — all three
  conditions hold: long-running listener, websocket push on file
  change, not modellable as a nushell pipeline. Zettel parsing
  (frontmatter + wikilink extraction) lives in Go because it sits on
  the fsnotify watch-loop hot path; this is flagged explicitly against
  the [[adr0003]] scope boundary (no rewriting existing nu data work —
  none exists for graph extraction today).
- Nodes: every typed zettel + hub notes; color by type, size by
  degree, hover tooltip with id / alias / status. Labels/tooltips
  only — no preview panel in v1.
- Edges: every [[wikilink]] occurrence, deduped; dangling links render
  as ghost nodes (visual broken-ref diagnostic).
- Page + vendored cosmos.js embedded via go:embed (offline-safe, no
  CDN, no build step on target). `GET /api/graph` JSON, `WS /watch`
  live push, `GET /api/status`, `POST /api/stop`, port
  AKM_GRAPH_PORT (default 4810).

Consumers: (1) the developer browsing/auditing the knowledge graph in
a browser via the nu wrapper; (2) scripts and skills consuming the
machine-readable `GET /api/graph` JSON for link analysis — orphan and
dangling-ref detection, future `akm check` integration. No existing
implementation carries graph-viz logic ad-hoc today, so there are no
migration targets; [[im003]] (neovim-d2-diagramming) is the closest
consumer-pattern precedent (daemon + browser preview).

Dedup check: no existing feature covers this — [[ft001]] (project
registry), [[ft002]] (d2-preview-router), [[ft003]] (i3 config
layering) were surveyed; [[ft002]] is the architectural precedent
(single-port Go daemon, lazy work, nu wrapper, pidfile/log under
~/.cache) and its conventions should be mirrored. Constraints inherit
from [[cat001]] (workflow-tooling): [[adr0001]] nushell-first user
surface, [[adr0002]] bash boundary untouched, [[adr0003]] Go daemon
gate + nu wrapper mandate + rotz install guarding `command -v go`.

A new ft### (akm-graph daemon) is minted at spec-writing time, not
here.

## solution
Solve [[us006|visualize-note-relations]] by minting and building
[[ft004|akm-graph]]: a Go daemon `akm-graph-d` mirroring [[ft002]]
conventions — single port (`AKM_GRAPH_PORT`, default 4810), pidfile +
log under `~/.cache/akm-graph/`, control API, rotz `go build` install
guarded by `command -v go`. The daemon walks `docs/notes/**.md` +
`docs/board.md`, parses frontmatter + `[[wikilinks]]` in Go (the
parse sits on the fsnotify watch-loop hot path; explicitly flagged
against the [[adr0003]] scope boundary — no existing nu graph
extraction is being rewritten), serves a go:embed viewer page with
vendored cosmos.js (`@cosmograph/cosmos`, offline, no CDN), exposes
`GET /api/graph` JSON for scripts/skills, and pushes rebuilt graphs
over `WS /watch` on file change. User surface is the nushell wrapper
`akm graph start|stop|open|status` per [[adr0001]] + [[adr0003]];
the [[adr0002]] bash boundary is untouched. Taxonomy: [[cat001]]
workflow-tooling.

Trade-offs taken: wikilink/frontmatter parsing in Go duplicates what
nushell could do offline (accepted — hot path, no prior nu
implementation); vendoring cosmos.js pins its version (accepted —
offline-safe beats freshness for a local tool).

Confidence gate: the single unproven assumption is that
`@cosmograph/cosmos` ships (or bundles to) a single standalone
browser file suitable for vendoring via go:embed. A PoC validates
this before spec-refinement.
