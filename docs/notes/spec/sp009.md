# Spec [[cat001]] [[board]]

## extends [[ft004]]

## problem

[[sp008]] (the [[ft005]] file-preview-daemon) embeds the [[ft004]]
akm-graph viewer as a cross-origin iframe ([[poc006]] validated the
embed). Its reverse channel â€” selecting an akm node in the preview should
open that zettel's source file in nvim â€” needs the akm-graph viewer page
to **emit the selected node's id/path** to its parent frame via
`window.postMessage`. Today `GET /` node click only isolates the node +
neighbors (client-side `selectNode`); it broadcasts nothing outward.

This is an **additive widening of [[ft004]]'s `## api_surface`**: on node
select, the viewer posts `{type, id}` (and, where resolvable, the source
path) to `window.parent`, guarded by an explicit target-origin. No change
to the existing isolate behavior, `/api/*`, or the websocket â€” the emit is
purely additive, so a standalone akm-graph page (no parent frame) is
unaffected. No `us###` drives this; it is feature-side, driven by
[[sp008]] Task 10. Whether the widening rides as an in-place `ft004`
refresh or a superseding `ft###` is the spec-writing call.

This sits in [[cat001]] (workflow-tooling). No binding [[adr0003]] /
[[adr0001]] conflict â€” the emit is client-side JS in the already-embedded
viewer page.

## solution

Server-side reverse open, routed per-parent â€” **not** the
`window.postMessage` relay this problem statement and the old alias
assumed. That relay (the shipped [[ft005]] shell listener from [[sp008]]
Task 10) is superseded for the akm case: it only works while akm-graph is
embedded in the preview shell, and leaves "which nvim" unsolved once the
daemon runs standalone.

Anchor: each nvim instance owns a preview window (slot `N`), and the
reverse open routes home through that per-slot parent binding. **That
binding is greenfield** â€” today every nvim pushes to a hardcoded
`--window 1` and preview-d holds only a process-global `$NVIM`
(`os.Getenv` in `handleOpen`), with no per-slot nvim address and no
nvimâ†’slot ownership. sp009 builds the binding: nvim registers its
`v:servername` against an allocated slot, preview-d stores it per-slot,
and `/open {slot}` routes to it. The embed side is likewise greenfield â€”
`handleFile`/`renderAkmEmbed` are slot-less today, so the slot must be
threaded from the `/preview<N>` route down to the akm iframe src.

1. **Widen [[ft004]] `## api_surface` (in place):** `/api/graph` nodes
   gain a `path` field â€” the zettel's source file under the notes root,
   `null` for ghost/dangling nodes. Add a same-origin
   `POST /api/open {id, slot}` on the akm-graph daemon.
2. **Slot passed at embed time:** the preview shell embeds akm-graph with
   `?slot=N` on the iframe src â€” the same embed-time channel [[adr0006]]
   established for backend embedding.
3. **Click routes server-to-server:** node click â†’ akm-graph client
   `POST /api/open {id, slot:N}` (same-origin, no CORS) â†’ akm-graph daemon
   resolves `id â†’ path`, then POSTs [[ft005]] preview-d `/open {path,
   slot:N}` daemon-to-daemon (no CORS, no browser relay).
4. **Widen [[ft005]] `## api_surface` (in place):** `/open` accepts an
   optional `slot`, routing to slot `N`'s parent nvim (`nvim --server
   <slot-N-addr> --remote <path>`) instead of the global `$NVIM`. Absent
   `slot` keeps today's `$NVIM` behavior â€” additive, no supersession.
5. **No-op guards:** ghost node / unresolved path â†’ no call (matches
   [[us009]] acceptance criteria); dead parent nvim â†’ the existing `/open`
   `424` surfaces.

Both widenings are additive: a standalone akm-graph page with no `?slot`
never calls `/api/open`, so existing behavior is untouched. Raising the
target nvim's window to the foreground (`i3-msg` / `swaymsg` / none, per
WM â€” the platform-conditional "depends") is deferred as a follow-up,
tracked in bd, not part of this spec's core.

Binds [[cat001]]; widens [[ft004]] + [[ft005]]; reconciles the [[us009]]
reverse-channel criteria; embed-origin mechanics per [[adr0006]]; embed
feasibility evidence [[poc006]].

## plan

Scope B â€” full multi-slot per-parent. The slotâ†’nvim binding and the
embed-side slot threading are both **greenfield** (verified against
current code); this breakdown builds them.

File tree (touched):

```
preview/
  session.go      # slot struct gains nvimAddr; SlotManager alloc + set/get addr
  server.go       # POST /register (alloc slot + bind addr); handleFile reads ?slot
  proxy.go        # handleOpen optional slot -> per-slot nvim; renderAkmEmbed ?slot
  static/app.js   # shell encodes ?slot=<window N> onto /file loads
  *_test.go
nushell/actions/
  preview         # `register` verb; start captures slot; send uses own slot
  preview-test
nvim/plugins/
  preview.lua     # own-slot state: register v:servername, send --window <slot>
  preview-test
akm-graph/
  graph.go        # Node.path from Note.AbsPath (root-relative), omit for ghost
  server.go       # POST /api/open {id,slot}
  open.go (new)   # id->path resolve + preview-d /open client (mirror proxy.go idiom)
  static/app.js   # selectNode -> POST /api/open {id,slot}; read ?slot from URL
  *_test.go
```

Conventions:
- Root-relative paths on the wire (match preview-d `resolveInRoot` +
  `redrawMsg.Path`); akm-graph converts `Note.AbsPath` via `filepath.Rel`,
  forward slashes.
- Cross-daemon HTTP = scoped `http.Client{Timeout}` at `127.0.0.1:<port>`
  (mirror `preview/proxy.go` `daemonHealthy` / `resolveD2URL`). preview-d
  default port 4200 (`$PREVIEW_PORT`).
- Client query read via `URLSearchParams` (akm `app.js` precedent:
  `?fixture`, `?backend`).
- `slot` is an int; absent/omitted = "no slot" â†’ `/open` falls back to the
  global `$NVIM` path (back-compat with the shipped [[ft005]] `/open`).
- Nu wrapper stays the mandatory user entry ([[adr0003]] / [[adr0001]]);
  the new `/register` + `/api/open` are internal daemon-to-daemon calls.
- `?slot` rides the embed-time param channel [[adr0006]] established.

Anti-patterns (binding):
- No raw client input to a subprocess: `slot` must be validated (int,
  known slot) before use; the existing `/open` arg-vector `nvim` exec
  stays (no shell string).
- Every new cross-daemon call carries a timeout (dead peer â†’ error, not
  hang; surface as the existing `424`/`5xx`, never a stalled goroutine).
- Standalone paths must keep working: akm page with no `?slot` must not
  break; `/open` with no `slot` must keep using global `$NVIM`;
  `/file/<path>` with no `?slot` renders as today.
- **Do NOT delete** the shipped `preview/static/app.js` postMessage
  listener (T10) â€” sp009 supersedes it *for akm* only; it still serves the
  d2 path / sp010. Removing it is a regression.
- No `git add -A` in implementation (concurrent sessions); explicit
  pathspec only.

Known limitations:
- Slot allocation is per preview-d process; a daemon restart reallocates,
  nvim re-registers on next `PreviewStart`.
- WM focus (raise the target nvim window) is out of scope â€” deferred to bd
  `dotfiles-7n6` (`i3-msg`/`swaymsg`/none shim).
- d2 reverse channel is [[sp010]]; it may reuse the `/register` +
  `/open {slot}` built here.

#### success_criteria
- The `slot` struct carries an nvim server address; `SlotManager` gains
  `SetNvimAddr(n int, addr string)` and `NvimAddr(n int) string` (`""`
  when unset) â€” the per-slot half of "open in the nvim that owns slot N"
  ([[us009]] AC: correct target nvim).
- `POST /register` body `{nvim, slot?}` allocates a free slot when `slot`
  is omitted (responds `{slot: N}`) and binds the nvim addr to it;
  re-registering an explicit slot updates the addr (last-wins).
- Concurrent registers with no slot receive distinct slot numbers (never
  the same N for two different nvims).

#### edge_cases
Empty `nvim` addr â†’ 400. Re-register existing slot â†’ addr updated, not a
new slot. Concurrent allocation â†’ distinct slots (mutex-guarded).
`NvimAddr` for an unknown slot â†’ `""`.

#### test_plan
Go, against the real `SlotManager` (not a mock): register with no slot â†’
returns `N` and `NvimAddr(N)==addr`; two goroutines register concurrently
â†’ two distinct slots asserted; re-register `N` with `addr2` â†’
`NvimAddr(N)==addr2`; empty addr â†’ 400.
