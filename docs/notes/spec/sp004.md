# Sp [[board]]

## problem

`akm adr read|list|write` shipped as the guinea pig for the per-type CLI
namespace pattern (commits 21cffc2 → 7d1e439, validated end-to-end via
skill-creator subagent eval). The pattern proved out: skill body shrank
from ~225 to ~205 lines, file-touching collapsed into one `printf … |
akm adr write --stdin` invocation, all `verification_checklist` items
passed in a fresh `akm init` sandbox, and id capture works through the
structured `Id: adr####` first-line output.

Six typed namespaces still ride the flat `akm write <type> <name>` form
and their owning skills still compose body markdown via raw
`$AKM_ROOT/docs/notes/...` writes:

- `us` (story-write)
- `pn` (persona-write)
- `ft` (feature-write)
- `im` (implementation-write)
- `cat` (category-write — partially: `akm cat check` exists; `akm cat
  list / read / write` do not yet)
- `sp` (spec-writing — different shape; sp### is created by `akm write
  sp` already, but `## solution`, `## tasks`, status flips, and the
  board.md section moves all happen via raw edits in the lifecycle
  skills)

Each typed skill carries its own duplicated boilerplate: AKM_ROOT
resolution, frontmatter shape, H1 categorization rules (where they
apply), staging policy. That boilerplate diverged subtly enough that
the adr-write skill-creator eval caught a verification-checklist
mismatch in adr-write's H1 (`# ADR` vs the title) that nobody noticed
in normal use — strong evidence that the other five untouched skills
carry their own quiet drift.

The shape to land for each type follows the adr template exactly:

- `akm <type> read <id>` — id validation against the type prefix,
  delegate to the existing `find_zettel` resolver.
- `akm <type> list [--json]` — filtered view, JSON for pipelines.
- `akm <type> write <name>` — stdin-driven body, allocates next id,
  composes frontmatter + H1 (`# <Type> [[cat###]] [[product]]` where
  categorization applies) + footer, stages on the default branch,
  prints `Id: <id>` on stdout for capture.

Per-type quirks to design through, not blockers:

- `us`: H1 carries tag wikilinks (not `[[cat###]]`); `## role` references
  `[[pn###|alias]]`; `## acceptance_criteria` bullets are body, not
  frontmatter — same stdin pattern works.
- `pn`: no category, no H1 wikilinks beyond `[[product]]`; simpler than
  adr.
- `ft`: H1 carries one or more `[[cat###]]`; body has structured
  sections (`## api_surface`, `## data_model`, `## sample`,
  `## components`); supersession via `## superseded_by` (same shape as
  adr).
- `im`: mandatory `solves: [[us###]]` body link (validate against
  existing `us###`); H1 carries `[[cat###]]`; `## components`
  references code paths under user cwd (not AKM root); accepted-vs-
  proposed status.
- `cat`: append-only, `status: stable` only, no supersession
  lifecycle; minimal body (`## name / ## summary / ## scope_notes`).
  Also gains `akm cat list` + `akm cat read` (alongside the existing
  `akm cat check`).
- `sp`: special-case. Already minted by `akm write sp` + board.md `##
  idea` insertion. The unique surfaces are status flips (`idea → spec
  → ready → done`), section moves in `docs/board.md`, and the epic
  finale in `docs/archive.md` — none of which fit a one-shot `write`
  verb. Stage 8 should expose verbs like `akm sp solution <id>
  --stdin`, `akm sp refine <id> --stdin`, `akm sp ready <id>`, `akm sp
  done <id>`. Larger surface than the other types.

Out of scope for this spec (defer):

- Supersession verb on the CLI (`akm <type> supersede <old> --with
  <new-name>`). The adr-write skill still does this raw; once the
  pattern is stable across the typed reads/writes, supersession
  becomes the next CLI layer.
- Retiring the flat `akm write <type> <name>` form. Both forms can
  coexist until every typed skill has migrated.
- Skill-creator evals for every type. Open question: run the same
  PostgreSQL-style eval against `story-write` / `persona-write` /
  `feature-write` / `implementation-write` / `category-write` /
  `spec-writing` after each migration, or only after all six are
  done?

Acceptance gates this spec must satisfy:

- One commit per typed namespace landing (CLI + skill update + eval
  evidence), reviewable in isolation. No multi-type mega-commit.
- Each migration validates against a skill-creator subagent eval in a
  fresh `akm init` sandbox before merging. The adr-write eval is the
  template.
- Documentation in `docs/notes/akm.md` reflects the per-type API
  shape consistently — no half-and-half references.
