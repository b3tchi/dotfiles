# Spec [[cat001]] [[board]]

## problem

The dotfiles AKM workspace was scaffolded before the canonical 8-category
taxonomy existed, so it carries a single home-grown category
`[[cat001|workflow-tooling]]`. Every other project (`samples-demo`,
`stock-design-system`, and any new repo bootstrapped via `akm init`) uses
the canonical set, which is now baked into `akm init`:

- cat001 Technology choices
- cat002 Architecture patterns
- cat003 API design
- cat004 Data modeling
- cat005 Infrastructure
- cat006 Security
- cat007 Testing
- cat008 Process

Eight wikilinks point at the old `cat001` and must keep resolving after the
migration: `adr0001`, `adr0002`, `ft001`, `im001`, `im002`, `sp001` (H1),
`sp002` (H1 + body prose).

The migration has to decide three things:
1. How to retire the existing `cat001=workflow-tooling` entry — rename in
   place (alias swap to a canonical name) or supersede via a new
   `cat`/`adr` entry that records the schema shift.
2. Which canonical category each currently-tagged zettel maps to. The
   shipped ADRs (`adr0001` nushell choice, `adr0002` bash vs nushell) read
   as `cat001 Technology choices`; the implementations and features
   (`im001`, `im002`, `ft001`) and specs (`sp001`, `sp002`) read as a mix
   of `cat005 Infrastructure` (remote project dispatch, terminal handoff)
   and `cat008 Process` (workflow ergonomics) — exact mapping is part of
   the spec stage.
3. Whether the canonical set ever wants a `workflow-tooling` slot back
   (the original intent), in which case the right answer is to *extend*
   the canonical default rather than discard the existing zettel.

Constraints:
- Wikilinks must remain valid post-migration (no dangling references).
- ADRs are append-only — re-categorising an existing ADR is a body edit
  of the H1 tag, not a supersession of the decision itself.
- The 8-category canonical set is fixed today (matches `akm init`); any
  change to it should be an explicit decision, not a side effect of this
  migration.

Out of scope (defer to spec-writing):
- Per-zettel category assignment.
- Decision on whether to rename `cat001` or extend the canonical set with
  a 9th slot.
- Any tooling change to `akm` (the CLI is fine; this is a data migration).
