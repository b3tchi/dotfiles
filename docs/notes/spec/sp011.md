# Spec [[cat001]] [[board]]

## solves
[[us010|akm-graph live highlight without reload]]

## problem

[[ft005]]'s shell (`preview/static/app.js` `showFile`) swaps the content
iframe's `src` on every websocket frame. When both the outgoing and the
incoming path are akm zettels, that swap reloads the `/file/<path>` embed
page and with it the cross-origin [[ft004]] viewer iframe
(`renderAkmEmbed` → `http://127.0.0.1:4810/?slot=N`), so zettel→zettel
cursor sweeps reboot the force-graph on every step: camera position,
layout momentum, legend/isolate state and the poster cache all reset, and
the viewer never indicates WHICH zettel the editor is on.

The delta ([[us010]], decided with the user 2026-07-15):

- **Skip the reload** only for zettel→zettel transitions — the shell must
  detect "current backend is the akm embed AND next path is an akm
  zettel" and, instead of swapping `src`, forward the new current path as
  a message into the running viewer. Everything else (zettel→code,
  code→zettel, first load) keeps today's swap semantics.
- **Visual highlight only** on the [[ft004]] side: the viewer marks the
  current node with a distinct ring/color + label emphasis. Explicitly NOT
  the existing click-isolate behavior and NOT camera pan/zoom — sweeping
  the cursor across files must stay visually calm. Unknown/ghost/absent
  node → clear previous highlight, silent no-op.
- **Contract widening on [[ft004]]** — the viewer gains an inbound
  postMessage surface (`highlight <path-or-id>`), the forward twin of the
  [[us009]]/[[adr0007]] reverse channel (`node select → POST /api/open`).
  Same discipline applies in reverse: origin-guarded listener, id/path
  resolved against the current graph, never trusting raw sender input.
  ft004's `## api_surface` must be updated when this ships.
- **Message routing**: shell (4200) → embed page (4200, same-origin) →
  viewer iframe (4810, cross-origin). The shell can reach the inner
  viewer window and postMessage with an explicit `http://127.0.0.1:4810`
  target origin; the exact relay shape (direct `frames[0]` handle vs a
  small relay script in the embed page) is a spec-writing decision.
- Zettel-ness of a path is already decided server-side by preview-d's
  render routing; the shell currently has no such classifier — either the
  daemon annotates the redraw frame (e.g. `{"path": ..., "backend":
  "akm"}`) or the shell learns the mapping. Daemon-side annotation keeps
  the classifier in one place ([[ft005]] owns render routing).

Not in scope: d2 embeds (user: "specific to akm"), pinning the graph
across non-zettel files (rejected — swap-as-today chosen), any isolate or
camera behavior change in [[ft004]]. Click semantics stay untouched:
bare node click keeps isolate; the [[us009]] open-in-editor action lives
on a dedicated label glyph (user 2026-07-15), so the highlight from this
spec never collides with either.

Binding context: [[adr0001]] (nushell surface unaffected), [[adr0003]]
(both daemons stay compiled Go), [[adr0007]] (embed/reverse-channel
postMessage discipline this mirrors). Consumed: [[ft004]], [[ft005]].
No ADR is overturned; no category change on [[im004]].

Supersession vs refresh: body refresh — [[im004]] (graph viewer) and
sp008's shipped shape stay valid; ft004/ft005 gain surface, nothing is
replaced. `## solution` and task breakdown are spec-writing /
spec-refinement scope.

## solution

Server-side, daemon-to-daemon — the forward twin of the [[adr0007]]
reverse channel, deliberately NOT a browser postMessage relay (same
rationale: works standalone and embedded, daemons hold the trusted
path/id state, no cross-origin matrix in the shell). Approved by the
user 2026-07-15. Both daemons stay compiled Go per [[adr0003]]; the
nushell wrapper surface ([[adr0001]]) is untouched. Category
[[cat001]].

**[[ft005]] preview-d** already owns render routing (it classifies akm
zettels in `handleFile`/`renderAkmEmbed`) and each slot's current path
(`handlePreviewSet`). Two additions:

- On `POST /preview<N>` with an akm-zettel path, forward
  `POST /api/highlight {path, slot}` to akm-graph-d (peer HTTP client
  with scoped timeout — the [[sp009]] `/open` client pattern in
  reverse), then update the stored slot path.
- Broadcast the shell redraw frame ONLY when the slot's previous path
  was not an akm zettel (backend transition → iframe swap as today).
  akm→akm transitions broadcast nothing: the shell iframe stays put,
  satisfying the no-reload AC of [[us010]]. Failure of the highlight
  POST degrades to the old broadcast (reload) rather than a dead
  preview.

**[[ft004]] akm-graph-d** gains the inbound surface (contract widening
flagged in the problem; `ft004.api_surface` updates at spec-retro):

- `POST /api/highlight {path, slot?}` — resolve path → node id against
  the in-process graph (trusted map, mirror of `/api/open`'s id → path
  direction). Unknown/ghost path → store "no highlight" for the slot,
  200, no broadcast error. Store per-slot current highlight; push a
  `{type: "highlight", id, slot}` message over the existing `/watch`
  WS hub.
- `/watch` connections carry the embed's `?slot=N` so the viewer
  receives its slot's highlight; on connect the stored highlight is
  primed (same pattern as preview-d's slot priming), which makes the
  first-load-highlight AC fall out for free.
- Viewer (`static/app.js`): apply a visual ring/color + label emphasis
  through the existing node-styling layer (the isolate/legend
  machinery, `selectedKeep` precedent) — explicitly no isolate, no
  camera motion; highlight state cleared on graph rebuild reprime.

Trade-offs taken: preview-d gains a runtime coupling to akm-graph-d on
zettel moves (accepted — localhost POST, already lazy-spawns the daemon
for the embed; failure path falls back to reload). Per-slot highlight
state lives in akm-graph-d (accepted — it already holds per-graph state
and the WS hub; keeping it out of the viewer makes multi-window and
standalone use consistent).

Proof: no PoC needed — every mechanism is shipped. Daemon-to-daemon
client + slot routing: [[sp009]] (`/api/open`); WS hub + rebuild
broadcast + client priming: [[im004]]; node styling hooks: the shipped
isolate/legend features (`selectedKeep`, per-node color gates in
`akm-graph/static/app.js`).
