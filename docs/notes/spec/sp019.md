# Spec [[cat002]] [[cat001]] [[board]]

## problem
Desktop notifications are broken on any session that is not the first to
start. Quickshell hosts a `NotificationServer` inside every per-display bar
instance (`quickshell/config/shell.qml`), but `org.freedesktop.Notifications`
is a **user session-bus singleton** — both live sessions share
`/run/user/1000/bus`. The local `:0` bar and the xrdp `:10` bar
([[adr0004]]) both request the name; the first owner keeps it, so every
notification renders on that one bar and the other bar's server is dead. On
this host both sessions are permanently live, so notifications silently land
on whichever session started first.

The fix the user endorsed: the bar should *listen to* a notification server,
not *be* one. Decouple the server from the bar — exactly the pattern the
shared clipboard history [[ft007]] and the `qs-stats-daemon` singleton
already use: one owner writes a store + a live-state file, every consumer
reads it.

Second, add a **notification history browser** — a `mod+Shift+n` picker that
recalls past notifications, mirroring the `mod+Shift+v` clipboard picker.
Built on the shared combobox control [[ft008]] (the same `Combo` the four
i3 dialogs consume, [[im006]]), hosted in the main quickshell instance like
`ClipHistory`, delivered through the [[ft003]] keybind + host layer.

Intent: mint one new `ft###` at spec-writing — a **notification service**
(daemon + store + broadcast + browser), analogous to [[ft007]]. Boundary:
a single user-scoped `quickshell -p notif` daemon owns
`org.freedesktop.Notifications` behind a user-level flock (one process for
all sessions), appends each notification to a **persistent** store under
`~/.local/state/qs-notif/` (file-per-entry, seq+timestamp), writes an
atomic live-state file under `$XDG_RUNTIME_DIR` that every bar follows with
`tail -F`, and accepts dismiss commands over a FIFO. Bars become read-only
consumers — the `NotificationServer` and notif-state properties are ripped
out of `shell.qml`/`Bar.qml` and replaced by a state-file source. Retention
is age-capped (~2 days), pruned on write. The browser is view + dismiss
only (Enter/Del dismisses; no clipboard copy, no action-invoke —
`actionsSupported` stays false).

Consumers: every bar host (live ticker + bell count, all sessions) and the
new browser dialog. Migration target: `quickshell/config/shell.qml` +
`Bar.qml` notif wiring (the current per-bar `NotificationServer`).

Categories: [[cat002]] (graphical session stack) primary, [[cat001]]
(workflow-tooling glue, picker family). Keybind: `$mod+Shift+n` toggles the
browser; the current name-workspace bind relocates to `$mod+Shift+o`.

Two decisions to record as ADRs at spec time: (1) the notif server is a
**user-bus singleton daemon and bars are read-only consumers** — the
analogue of the bespoke-store decision [[adr0011]]; (2) the **persistent
notif store deviates** from [[ft007]]/[[adr0011]] memory-only and the
copy-never-paste stance [[adr0010]] — justified by the short 2-day age cap
plus recall value, accepting that notif text (codes, private messages)
lands on persistent disk.

## solution
Split the notification stack into **one owner + many read-only consumers**,
the shape [[ft007]] shipping-clipboard-history and the `qs-stats-daemon`
singleton already prove on this repo.

**Owner — a user-scoped `quickshell -p notif` daemon.** A dedicated
quickshell profile hosts the single `NotificationServer` and owns
`org.freedesktop.Notifications`. It is launched from `qs-start.sh` but
guarded by a **user-level flock** (not the per-display `qs_kill_session`
scoping in `qs-session.sh`, which correctly keeps two *bars* apart but is
the wrong axis for a bus-singleton): the first session to come up starts
the daemon, every later session's launch attempt no-ops on the lock — the
exact singleton guard `qs-start.sh` already wraps around `qs-stats-daemon`.
This is the fix: no bar ever requests the bus name again, so the
first-owner-wins collision between the `:0` and `:10` bars ([[adr0004]])
cannot happen. On each notification the daemon does two writes — append to
a persistent file-per-entry store (seq + timestamp, the `clip-store` shape
from [[ft007]]) and rewrite an atomic live-state file under
`$XDG_RUNTIME_DIR` (count / lastText / seq / hasCritical). Dismiss + prune
arrive over a FIFO command channel; retention is age-capped (~2 days),
pruned on write. Store scripts follow the `#!/bin/sh` house rule
([[adr0002]]).

**Consumers — the bars, read-only.** `NotificationServer` and the
notif-state properties are ripped out of `quickshell/config/shell.qml` and
the ticker/bell wiring in `Bar.qml`; the ticker, bell icon and count are
driven by `tail -F` on the daemon's live-state file — the same source model
`Bar.qml` already uses for `qs-stats`. Every session renders identical
notifications because they read one file.

**Browser — a picker on the shared combobox [[ft008]].** A new
`quickshell/config/NotifHistory.qml` instantiates `Combo`
(`filterMode: external`, newest-first, like the clip picker [[im006]]),
hosted in the MAIN instance alongside `ClipHistory`, reached over a
two-verb `qs-notif.sh toggle/list/dismiss` contract and the [[ft003]]
keybind + host layer. It is **view + dismiss only** — Enter/Del dismisses,
no clipboard copy and no action-invoke (`actionsSupported` stays false),
sidestepping the input-path failure class [[adr0010]] closed for the clip
picker. Keybind `$mod+Shift+n`; the current name-workspace bind relocates
to the free `$mod+Shift+o`.

The work delivers **one new `ft###`** (a *notification-service* capability
— daemon + persistent store + broadcast + browser, the [[ft007]] analogue
for notifications), minted at spec-refinement once its `api_surface` is
concrete ([[ft009]] precedent: minted at refinement, `proposed` until
ship). Two decisions get recorded as ADRs at refinement: (1) the notif
server is a **user-bus singleton daemon; bars are read-only consumers** —
the analogue of the bespoke-store decision [[adr0011]]; and (2) the
**persistent notif store deliberately departs** from the memory-only stance
[[ft007]]/[[adr0011]] hold for clipboard, justified by the short 2-day age
cap plus recall value, accepting that notification text lands on persistent
disk. Categories [[cat002]] (graphical session stack) primary, [[cat001]]
(workflow-tooling, picker family).

Refinement deltas (from the `## solution` sketch):
1. **Kill-sweep exclusion.** `qs-start.sh` opens with
   `qs_kill_session -x quickshell`, which would kill the notif daemon
   on every restart of whichever session spawned it (the daemon
   inherits that session's `DISPLAY`, so the environ match hits). The
   sweep must spare any pid whose `/proc/<pid>/cmdline` carries the
   notif profile path — the daemon is user-level, not session-level,
   and only the flock section may restart it.
2. **Del-key rule.** Combo's filter input is a focused `TextField`;
   a bare Del edits the filter text. Del-dismiss therefore fires only
   while the filter is EMPTY (intercepted caller-side in
   `NotifHistory.qml`, never by widening [[ft008]]); with text in the
   filter, Del keeps its editing meaning and Enter is the dismiss key.
3. **Count semantics.** The state-file `count` is the daemon's live
   TRACKED count (today's bell semantics), not the store size — the
   store is history and outlives dismissal.
4. **Single writer.** Only the daemon process mutates the store and
   the state file. Browser/bell dismissals travel over the FIFO to the
   daemon; `qs-notif.sh list` reads the store directly (read-only).
   A dismiss with no daemon alive fails loudly (no FIFO reader — exit
   nonzero within a bounded timeout), never blocks, never writes.
5. **Hash-check restart.** Mirroring the `qs-stats-daemon` guard: a
   content hash over `quickshell/notif/` + `qs-notif-store.sh` decides
   restart-on-change under the same flock, so QML/script edits land on
   the next `qs-start.sh` without manual pkill.

Conventions:
- Store contract lives in `qs-notif-store.sh`'s header (the
  `clip-store.sh` mould); every consumer is written against it.
- Entry writes are atomic: payload to a `.tmp` in the store dir,
  `ln`(2)ed into the final seq name; consumers glob `??????.notif`
  only, so work files are never visible entries.
- Ids are opaque strings end to end — equality and existence only,
  never parsed or `parseInt`ed (the dotfiles-g5b bug class).
- State-file updates are atomic tmp+rename; `tail -n +1 -F` re-emits
  the whole file per swap and every consumer set is idempotent (the
  exact `feedProc` shape in `Bar.qml`).
- Test discipline copied from the clip/combo suites: isolated
  `XDG_RUNTIME_DIR` + `XDG_STATE_HOME`, isolated session bus
  (`dbus-run-session`) for daemon scenarios, Xvfb for UI scenarios,
  named scenarios, `SELFTEST=1` wires one wrong expectation that MUST
  fail.
- Scripts are `#!/bin/sh`, `set -u`, `umask 077` on store writers
  ([[adr0002]] + family convention).

Anti-patterns (auto-reject in review):
- Any bar/overlay instance hosting a `NotificationServer` or
  requesting the bus name again — the entire point ([[adr0012]]).
- Scoping the daemon per display / per session (`qs_kill_session`
  axis) instead of the user-level flock — wrong axis for a user-bus
  singleton.
- Placing the persistent store under `$XDG_RUNTIME_DIR` (defeats
  [[adr0013]]) or hardcoding `~/.local/state` without honouring
  `XDG_STATE_HOME` (breaks test isolation).
- Prune by count only, or unbounded retention — the age cap is the
  [[adr0013]] contract.
- Clipboard copy, action-invoke, or `actionsSupported: true` in the
  browser ([[adr0010]] posture; view + dismiss only).
- Widening [[ft008]] `Combo` for Del handling or notif-specific
  chrome — caller-side only, like the clip picker's status bar.
- `parseInt`/arithmetic on entry ids; positional-index dismissal
  (dismiss carries the row's OWN id, [[ft008]] row-object contract).
- bash-isms in the sh scripts; mktemp//tmp work files outside the
  store dir.

Known limitations:
- The daemon inherits the spawning session's X connection; if that
  session dies, the daemon dies with it and notifications are unowned
  until any session's next `qs-start.sh` run (i3 reload/restart or
  login). No watchdog is built here.
- Notifications sent while no daemon runs are lost (the user bus does
  not queue for absent names) — same exposure as today's
  first-owner-wins, minus the permanent wrong-owner failure.
- Live tracked state (bell count, hasCritical) resets on daemon
  restart; the history store persists.
- No popup window is added — ticker + bell remain the render surface
  (parity with today).
- [[ft010]]/[[im008]] stay `proposed` through refinement; the
  `→ stable`/`→ accepted` flips land at work-merge/spec-retro.

### Task 1: Store backend — qs-notif-store.sh + persistent contract

#### type
feature

#### effort
6h

#### depends
none

#### files_touched
- quickshell/qs-notif-store.sh (new)
- quickshell/test-notif-store.sh (new)

#### success_criteria
- The script header DEFINES the store contract (the `clip-store.sh`
  mould): store dir `${XDG_STATE_HOME:-$HOME/.local/state}/qs-notif/`
  mode 0700; entry `NNNNNN.notif` (six-digit zero-padded monotonic
  seq); line 1 header `<epoch>\t<urgency>\t<app>`; line 2 summary
  (control whitespace folded to spaces); line 3+ raw body; ids opaque;
  lexicographic order IS arrival order (AC2).
- `append` reads the body from stdin with summary/urgency/app/epoch
  from args, writes atomically (`.tmp` + `ln`, collision = bounded
  retry, never clobber), continues the seq across restarts (leading
  zeros stripped before arithmetic — `$((000008))` is an octal error),
  and prunes entries whose header epoch is older than
  `QS_NOTIF_MAX_AGE` (default 172800 s) on every write (AC2).
- `dismiss <id|latest>` removes exactly the named entry (shape-check
  `^[0-9]{6}\.notif$` before any path use; `latest` resolves to the
  highest seq); missing entry exits 1 having removed nothing.
- `state` rewrites the live-state file
  (`${QS_NOTIF_STATE:-$XDG_RUNTIME_DIR/qs-notif.state}`) atomically
  via tmp+rename with exactly the lines `count N`, `critical 0|1`,
  `seq N`, `last <epoch>\t<text>` — a reader never sees a partial
  file (AC3 source format).
- Unset `XDG_RUNTIME_DIR` fails `state` loudly (exit 78, the family
  convention); unset `HOME`+`XDG_STATE_HOME` fails store verbs the
  same way. `#!/bin/sh`, `set -u`, `umask 077` throughout
  ([[adr0002]]).

#### edge_cases
- Body containing tabs/newlines/unicode/HTML survives byte-exact in
  the entry (only the summary line and state `last` fold whitespace).
- Empty body: entry has header + summary, zero body lines — `append`
  still succeeds; preview later renders the summary.
- Age-cap boundary: an entry exactly at the cap is kept; one second
  past it is pruned; `QS_NOTIF_MAX_AGE=0` prunes everything but the
  entry just written.
- Two concurrent `append`s (simulated) resolve the `ln` collision as
  retry — both entries exist, no payload lost.
- Store dir deleted mid-session is recreated on next write.

#### test_plan
`test-notif-store.sh`, headless, no X — pure sh against isolated
`XDG_STATE_HOME`/`XDG_RUNTIME_DIR` in `$TMP`. Named scenarios:
`append-atomic-visible`, `seq-continues-after-restart`,
`octal-seq-008`, `age-prune-on-write`, `age-boundary-kept`,
`dismiss-by-id`, `dismiss-latest`, `dismiss-missing-exit1`,
`state-atomic-rewrite` (reader loop during 100 rewrites never sees a
partial line set), `unicode-body-byte-exact`, `xdg-unset-exit78`,
`collision-retry`. Mutation checks: swapping `ln` for `mv` fails
`collision-retry`; removing the prune fails `age-prune-on-write`.
`SELFTEST=1` flips one expectation and must FAIL.

### Task 2: Daemon profile — quickshell/notif hosts the single server

#### type
feature

#### effort
6h

#### depends
Task 1

#### files_touched
- quickshell/notif/shell.qml (new)
- quickshell/test-notif-daemon.sh (new)

#### success_criteria
- `quickshell -p quickshell/notif` hosts the ONLY
  `NotificationServer`; flags replicate today's server byte-for-byte:
  `bodyMarkupSupported: true`, `imageSupported: false`,
  `actionsSupported: false`, `persistenceSupported: false`,
  `keepOnReload: true` (AC1, AC4 posture).
- On each notification the daemon (a) pipes summary/body/urgency/app
  to `qs-notif-store.sh append` (body on stdin — never argv, so
  arbitrary bytes survive) and (b) rewrites the state file via the
  `state` verb with tracked count / hasCritical / bumped seq / folded
  last-text — two writes per event, both through the store script
  (single-writer, refinement delta 4) (AC2, AC3).
- The daemon owns the FIFO
  (`${QS_NOTIF_FIFO:-$XDG_RUNTIME_DIR/qs-notif.cmd}`): created 0600
  at startup, read line-wise; `dismiss <id>` removes the store entry
  AND `.dismiss()`es the matching tracked notification if live, then
  rewrites state; `dismiss latest` targets the newest. Malformed FIFO
  lines are ignored without crashing the reader.
- `notify-send` from two different environments against one daemon
  yields entries in one store and a state file whose `count`
  increments both times — proving session-independence of the owner
  (AC1).
- The QML contains no rendering `Window`/`PanelWindow` — the profile
  is headless service only; presentation stays in the bars.

#### edge_cases
- Critical-urgency notification flips `critical 1`; dismissing it
  (and no other critical tracked) flips it back to `0`.
- Notification with markup body: state `last` and the entry summary
  fold/strip like today's ticker text (`<[^>]*>` stripped, newlines
  to spaces); the stored body keeps the raw markup.
- FIFO deleted mid-run: daemon recreates it and keeps reading
  (reader loop restarts like `Bar.qml`'s `onExited: running = true`
  subscriptions).
- Burst of notifications (10 in <1 s): all 10 entries land, seq
  strictly monotonic, final state `count` = 10.
- Replacing-id notifications (same `notify-send -r` handle): tracked
  count does not double-increment.

#### test_plan
`test-notif-daemon.sh`: `dbus-run-session` wraps each scenario so the
daemon owns `org.freedesktop.Notifications` on an ISOLATED bus (never
the live user bus); isolated `XDG_STATE_HOME`/`XDG_RUNTIME_DIR`;
`QT_QPA_PLATFORM=offscreen` first, Xvfb fallback if quickshell
refuses offscreen. Scenarios: `owns-bus-name` (a second
`quickshell -p notif` start on the same bus does NOT become owner),
`notify-lands-in-store`, `state-tracks-count`, `critical-flag-cycle`,
`fifo-dismiss-id`, `fifo-dismiss-latest`, `fifo-garbage-ignored`,
`burst-10-monotonic`, `markup-folded-state-raw-store`. Assertions run
against the store/state files (sh-visible effects, [[adr0002]] test
stance). `SELFTEST=1` negative control.

### Task 3: Launch — user-level flock singleton in qs-start.sh + rotz link

#### type
feature

#### effort
4h

#### depends
Task 2

#### files_touched
- quickshell/qs-start.sh
- quickshell/dot.yaml

#### success_criteria
- `qs-start.sh` launches the daemon behind a USER-LEVEL flock on
  `$XDG_RUNTIME_DIR/qs-notif-daemon.lock`, mirroring the
  `qs-stats-daemon` guard exactly: `flock -w 5 9`, `pgrep -f` the
  profile path, `setsid ... 9>&-` so the daemon never inherits the
  lock fd; a second session's start no-ops on the running daemon
  (AC1).
- Hash-check restart (refinement delta 5): a sha1 over
  `quickshell/notif/*.qml` + `qs-notif-store.sh` cached in
  `~/.cache/`; changed hash ⇒ pkill + restart under the lock,
  unchanged ⇒ the running daemon is left alone (another session's
  bar may be reading it).
- The opening `qs_kill_session -x quickshell` sweep SPARES the
  daemon: pids whose `/proc/<pid>/cmdline` contains the notif
  profile path are skipped (refinement delta 1) — asserted by a
  restart cycle on the spawning session leaving the daemon pid
  unchanged when the hash is unchanged.
- `qs-start.sh` exports `QS_NOTIF_FILE` (the state-file path) for
  `Bar.qml`, alongside the existing `QS_STATS_FILE` export.
- `quickshell/dot.yaml` links `notif: ~/.config/quickshell-notif`
  (the `overlay` link precedent); `rotz link quickshell --force`
  passes.

#### edge_cases
- Both sessions start simultaneously: flock serialises; exactly one
  daemon after both complete.
- Stale lock file from a crashed shell does not block (flock is
  advisory on the fd, not the file's existence).
- Daemon dead but lock file present: pgrep miss ⇒ relaunch succeeds.
- `QS_RDP=1` path: daemon launch runs there too (the xrdp session
  may be first up) — guard is unconditional, unlike the overlay skip.

Extend `test-notif-daemon.sh` with a `launch` phase running the real

`second-start-noop` (two runs, one spawn), `parallel-start-single`
(two backgrounded runs under the real flock, exactly one spawn),
`hash-change-restarts`, `hash-same-leaves-pid`,
