# Spec [[cat001]] [[board]]

## extends [[ft002]]

## problem

[[sp008]] (the [[ft005]] file-preview-daemon) embeds the [[ft002]]
d2-preview-router diagram page as a cross-origin iframe ([[poc006]]
validated the embed). Its reverse channel — selecting a diagram (or a
linked element within it) in the preview should open the backing `.d2`
source file in nvim — needs the d2 diagram page to **emit the source
`.d2` path** to its parent frame via `window.postMessage`. Today the
`GET /{project}/{filename}` page renders the live diagram but broadcasts
nothing outward on click.

This is an **additive widening of [[ft002]]'s `## api_surface`**: the
served diagram page posts `{path}` (the `{project}/{filename}` it renders,
resolvable to the `.d2` source) to `window.parent` on click, guarded by an
explicit target-origin. Because [[ft002]] reverse-proxies the upstream
`d2 --watch` page, the emit is injected in the proxy's body-rewrite layer
(which already rewrites streamed bodies) rather than in upstream d2. No
change to existing proxy/websocket behavior; a standalone d2 page (no
parent frame) is unaffected. No `us###` drives this; it is feature-side,
driven by [[sp008]] Task 10. In-place `ft002` refresh vs superseding
`ft###` is the spec-writing call.

This sits in [[cat001]] (workflow-tooling). Ties to the [[poc006]]
residual: if upstream d2 sets a frame-blocking header, the same
body-rewrite layer strips it. No [[adr0003]] / [[adr0001]] conflict.
