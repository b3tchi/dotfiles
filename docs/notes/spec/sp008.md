# Spec [[cat001]] [[board]]

## problem

The developer ([[pn001|developer]]) works primarily in a TUI (nvim, nnn)
and today cannot see rich file content without leaving the editor —
images, video, 3D models (STL/CAD), and syntax-highlighted code all
force a context switch. The ask is a **cursor-driven preview bridge**: as
the cursor moves over a file reference in nvim or over an entry in nnn, a
floating webview window shows a preview of that file; moving to another
file redraws it. This is a new horizontal capability with no existing
`ft###` — a dedup survey of [[ft001]] (project registry), [[ft002]]
(d2-preview-router), [[ft003]] (i3-config-layering), and [[ft004]]
(akm-graph) found no content-preview feature. A new `ft###` is minted at
spec-writing time.

**Boundary.** One localhost Go daemon (the [[adr0003]] compiled-helper
precedent — websocket fan-out + concurrent renderers exceed the nushell
sweet spot) that: (1) accepts a "cursor now on file X" signal from the
editor, (2) renders X and pushes to a single persistent floating webview
over websocket, (3) accepts a reverse "open file Y" signal from the
webview and drives the editor to open it. One webview instance is the
initial target, but the daemon's client/session model should not preclude
**multiple concurrent preview windows** later (e.g. one per editor split
or per device) — a future extension, not in the first slice. Consumers
are **nvim and nnn**
— genuinely reusable, not one-consumer glue. The thin editor-side client
(POST the path; nvim `CursorMoved` autocmd + `$NVIM` `--remote` open
socket; nnn preview + open hooks) is nushell per [[adr0001]].

**Progressive rendering.** Cursor sweeping many files must stay cheap, so
each renderer has two tiers: a fast static **thumbnail** (default, shown
on hover) and a **live** view promoted on explicit activate. Renderers:
code → syntax HTML (chroma); markdown → HTML (goldmark); image →
downscaled then full-res; video → poster frame (ffmpeg) then `<video>`
playback; STL → orbit directly (skip headless-GL thumbnail).

**Special interactive backends.** Two existing daemons are proxied as
preview types rather than reimplemented: `.d2` files render live via
[[ft002]] (d2-preview-router); akm zettels/graph render via [[ft004]]
(akm-graph). These carry the **reverse channel** — selecting an akm node
or d2 element in the webview opens that item's source file back in nvim.

**Constraints.** Offline-safe: all viewer libs (three.js/model-viewer,
highlight assets) vendored via `go:embed`, no CDN, no target-side build —
the precedent [[ft004]] already sets. Localhost-only, same no-auth stance
as [[ft002]]/[[ft004]]. The floating webview window must render across
native i3 (X11), WSLg (Sway), and Termux:X11 — a browser `--app` window
pointed at `localhost:$PORT` is the portable surface, which pulls in
[[cat002]] (display-platform) as a secondary category alongside the
primary [[cat001]] (workflow-tooling).

**Granularity.** This stays one atomic `ft###` — the renderers and the
reverse channel are internal components of a single bidirectional bridge,
mirroring how [[ft002]] serves many diagram types under one feature. The
heavy/independent parts (video tier, STL orbit, d2-proxy, akm-proxy +
reverse-open, editor-signal client) split at task level during
spec-refinement, not into separate features.

## solution

A Go daemon `preview-d` — [[adr0003]] binds it: long-running, holds a
websocket listener, proxies streaming/websocket bodies, no nushell
structural fit → Go (stdlib + minimal libs) is required over
nushell/bash. Default `localhost:$PREVIEW_PORT` (4820, after
[[ft002]] 4800 and [[ft004]] 4810), no-auth localhost-only per the
[[ft002]]/[[ft004]] stance. All viewer libs (three.js / `<model-viewer>`,
highlight assets) vendored via `go:embed` — offline, no CDN, no
target-side build, the [[ft004]] precedent.

Per [[adr0003]] the binary is an engine, not an interface: a mandatory
nushell wrapper `nushell/actions/preview` ([[adr0001]]) owns the
user-facing verbs (`start` / `stop` / `status` / `open` / `send`). No
consumer calls the binary directly.

**Editor → daemon.** Debounced HTTP `POST /preview {path}` through the
wrapper. nvim drives it from a `CursorHold` / `CursorMoved` autocmd; nnn
from its preview hook. The daemon renders the path and pushes to the open
webview over websocket, reusing [[ft004]]'s live-push loop. Two render
tiers keep a cursor sweep cheap: a fast static **thumbnail** by default,
a **live** view promoted only on explicit activate. Renderers: chroma
(code), goldmark (markdown), downscale (image → full-res on activate),
ffmpeg poster frame (video → `<video>` on activate), three.js /
`<model-viewer>` (STL → orbit; skips the headless-GL thumbnail and loads
live directly).

**Special interactive backends.** `.d2` is proxied to [[ft002]]
(d2-preview-router) and akm zettels/graph to [[ft004]] (akm-graph) rather
than reimplemented — the daemon embeds their pages as preview types.

**Webview → editor (reverse channel).** The webview issues
`POST /open {path}`; the daemon opens it in the running editor via
`nvim --server $NVIM --remote`. This powers selecting an akm node or d2
element in the preview → its source file opens in nvim.

**Webview surface.** A browser `--app` window pointed at the port,
floating under i3 (X11), Sway (WSLg), and Termux:X11 — which pulls in
[[cat002]] (display-platform) alongside the primary [[cat001]]
(workflow-tooling). Chosen over a bespoke webkit binary for portability
across those three surfaces at the cost of footprint. One webview
instance is the first target; the daemon's session model must not
preclude multiple concurrent windows later.

**Consumes** [[ft002]], [[ft004]]. **Binds** [[adr0003]], [[adr0001]].

The proxy-embed of two other daemons' pages inside one webview shell
(cross-port, iframe or equivalent) is the least-proven part of this
shape and is being de-risked by a PoC before spec-refinement — see the
`[[poc###]]` back-link appended on validation.
