# Spec [[cat001]] [[board]]

## problem

The developer ([[pn001|developer]]) works primarily in a TUI (nvim, nnn)
and today cannot see rich file content without leaving the editor â€”
images, video, 3D models (STL/CAD), and syntax-highlighted code all
force a context switch. The ask is a **cursor-driven preview bridge**: as
the cursor moves over a file reference in nvim or over an entry in nnn, a
floating webview window shows a preview of that file; moving to another
file redraws it. This is a new horizontal capability with no existing
`ft###` â€” a dedup survey of [[ft001]] (project registry), [[ft002]]
(d2-preview-router), [[ft003]] (i3-config-layering), and [[ft004]]
(akm-graph) found no content-preview feature. A new `ft###` is minted at
spec-writing time.

**Boundary.** One localhost Go daemon (the [[adr0003]] compiled-helper
precedent â€” websocket fan-out + concurrent renderers exceed the nushell
sweet spot) that: (1) accepts a "cursor now on file X" signal from the
editor, (2) renders X and pushes to a single persistent floating webview
over websocket, (3) accepts a reverse "open file Y" signal from the
webview and drives the editor to open it. One webview instance is the
initial target, but the daemon's client/session model should not preclude
**multiple concurrent preview windows** later (e.g. one per editor split
or per device) â€” a future extension, not in the first slice. Consumers
are **nvim and nnn**
â€” genuinely reusable, not one-consumer glue. The thin editor-side client
(POST the path; nvim `CursorMoved` autocmd + `$NVIM` `--remote` open
socket; nnn preview + open hooks) is nushell per [[adr0001]].

**Progressive rendering.** Cursor sweeping many files must stay cheap, so
each renderer has two tiers: a fast static **thumbnail** (default, shown
on hover) and a **live** view promoted on explicit activate. Renderers:
code â†’ syntax HTML (chroma); markdown â†’ HTML (goldmark); image â†’
downscaled then full-res; video â†’ poster frame (ffmpeg) then `<video>`
playback; STL â†’ orbit directly (skip headless-GL thumbnail).

**Special interactive backends.** Two existing daemons are proxied as
preview types rather than reimplemented: `.d2` files render live via
[[ft002]] (d2-preview-router); akm zettels/graph render via [[ft004]]
(akm-graph). These carry the **reverse channel** â€” selecting an akm node
or d2 element in the webview opens that item's source file back in nvim.

**Constraints.** Offline-safe: all viewer libs (three.js/model-viewer,
highlight assets) vendored via `go:embed`, no CDN, no target-side build â€”
the precedent [[ft004]] already sets. Localhost-only, same no-auth stance
as [[ft002]]/[[ft004]]. The floating webview window must render across
native i3 (X11), WSLg (Sway), and Termux:X11 â€” a browser `--app` window
pointed at `localhost:$PORT` is the portable surface, which pulls in
[[cat002]] (display-platform) as a secondary category alongside the
primary [[cat001]] (workflow-tooling).

**Granularity.** This stays one atomic `ft###` â€” the renderers and the
reverse channel are internal components of a single bidirectional bridge,
mirroring how [[ft002]] serves many diagram types under one feature. The
heavy/independent parts (video tier, STL orbit, d2-proxy, akm-proxy +
reverse-open, editor-signal client) split at task level during
spec-refinement, not into separate features.

## solution

A Go daemon `preview-d` â€” [[adr0003]] binds it: long-running, holds a
websocket listener, proxies streaming/websocket bodies, no nushell
structural fit â†’ Go (stdlib + minimal libs) is required over
nushell/bash. Default `localhost:$PREVIEW_PORT` (4200), no-auth
localhost-only per the [[ft002]]/[[ft004]] stance. All viewer libs
(three.js / `<model-viewer>`, highlight assets) vendored via `go:embed` â€”
offline, no CDN, no target-side build, the [[ft004]] precedent.

Per [[adr0003]] the binary is an engine, not an interface: a mandatory
nushell wrapper `nushell/actions/preview` ([[adr0001]]) owns the
user-facing verbs (`start` / `stop` / `status` / `open` / `send`). No
consumer calls the binary directly.

**Route model â€” two modes on the one port.** The daemon exposes two
route families, which compose:

- `GET /file/<path>` â€” the **stateless primitive**: renders that one file
  *as-is*, the renderer selected *by the path* (extension â†’ code / image /
  video / STL / akm / d2). Addressable, cacheable, cursor-independent â€” a
  permalink to a single file's render. Flat types render inline; akm/d2
  types resolve to an embedded iframe of [[ft004]]/[[ft002]] ([[poc006]]).
- `GET /preview<N>` â€” the **stateful driver**: live webview window `N`. It
  is a thin shell that loads the current `/file/<path>` and **hot-swaps
  the path** as the editor cursor moves, redrawing over the websocket. The
  `<N>` slot is what makes multiple concurrent preview windows possible
  (one per editor split or device) without the daemon holding a single
  global "current file".

So `/file` owns *how* a path renders; `/preview<N>` owns *which* path a
given window is showing right now. Every renderer is written once for
`/file` and reused live by `/preview<N>`.

**Editor â†’ daemon.** Debounced HTTP `POST /preview<N> {path}` through the
wrapper sets slot `N`'s current path; the daemon pushes it to that
window's websocket, which reloads `/file/<path>`, reusing [[ft004]]'s
live-push loop. nvim drives the POST from a `CursorHold` / `CursorMoved`
autocmd; nnn from its preview hook. Two render tiers keep a cursor sweep
cheap: a fast static **thumbnail** by default, a **live** view promoted
only on explicit activate. Renderers: chroma (code), goldmark (markdown),
downscale (image â†’ full-res on activate), ffmpeg poster frame (video â†’
`<video>` on activate), three.js / `<model-viewer>` (STL â†’ orbit; skips
the headless-GL thumbnail and loads live directly).

**Special interactive backends.** `.d2` is proxied to [[ft002]]
(d2-preview-router) and akm zettels/graph to [[ft004]] (akm-graph) rather
than reimplemented â€” the daemon embeds their pages as preview types.

**Webview â†’ editor (reverse channel).** The webview issues
`POST /open {path}`; the daemon opens it in the running editor via
`nvim --server $NVIM --remote`. This powers selecting an akm node or d2
element in the preview â†’ its source file opens in nvim.

**Webview surface.** A browser `--app` window pointed at
`localhost:4200/preview<N>`, floating under i3 (X11), Sway (WSLg), and
Termux:X11 â€” which pulls in [[cat002]] (display-platform) alongside the
primary [[cat001]] (workflow-tooling). Chosen over a bespoke webkit binary
for portability across those three surfaces at the cost of footprint. The
first slice ships a single window (`/preview1`); the `/preview<N>` slot
model already carries the multi-window future (a second `--app` window on
`/preview2`, one per editor split or device) with no daemon rearchitecture.

The special backends are embedded as **cross-origin iframes**, not a
path-prefix reverse-proxy â€” [[poc006]] validated this: neither [[ft002]]
nor [[ft004]] emits a frame-blocking header, so each daemon loads at its
own origin inside its iframe and its absolute-path assets + internal
websocket work unchanged (no URL rewriting). The reverse channel
(akm-node / d2-element click â†’ open in nvim) rides `window.postMessage`
from the iframe to the shell, which then issues `POST /open`.

**Builds** [[ft005]] (file-preview-daemon â€” the capability this spec
delivers). **Consumes** [[ft002]], [[ft004]]. **Binds** [[adr0003]],
[[adr0001]]. **De-risked by** [[poc006]] (proxy-embed â†’ iframe, validated).

## plan

**File tree** (new module `preview/` mirrors `d2/router/` + `akm-graph/`;
touched integration files under `nushell/` + `nvim/`):

```
preview/
  go.mod
  main.go            # flags (-root, $PREVIEW_PORT=4200), start/stop wiring
  server.go          # mux: /file/, /preview<N>, /open, /watch, /status, /stop
  render.go          # dispatch by extension -> renderer
  render_code.go     # chroma (syntax HTML)
  render_md.go       # goldmark (markdown HTML)
  render_image.go    # downscale thumbnail + full-res
  render_video.go    # ffmpeg poster frame + <video>
  render_stl.go      # stl bytes + embedded three.js/model-viewer viewer
  proxy.go           # akm/d2 iframe target resolution + lazy-spawn
  session.go         # preview<N> slot map + ws hub (ported akm-graph Hub/wsClient)
  path.go            # /file/<path> root-jail sanitization
  embed.go           # go:embed static/
  static/            # shell.html, app.js, style.css, stl-viewer bundle (vendored)
  *_test.go
  dot.yaml           # rotz links + build (mirror akm-graph/dot.yaml)
nushell/actions/preview     # mandatory wrapper: start/stop/status/open/send
nvim/plugins/preview.lua    # CursorHold autocmd + reverse-open
```

**Conventions.** Port akm-graph's proven surface verbatim where it fits:
`Hub` / `wsClient` (session.go), `noCache`, `requireMethod`, `writeJSON`,
`go:embed` + `fs.Sub`, context-cancel `/stop`. Deps already in the
akm-graph `go.sum` lineage: `gorilla/websocket`; add
`alecthomas/chroma/v2` (code) + `yuin/goldmark` (markdown). Wrapper is
nushell per [[adr0001]]; the binary is an engine per [[adr0003]]. rotz
`dot.yaml` mirrors `akm-graph/dot.yaml` for links + build.

**Anti-patterns (ship-blockers).**

- `/file/<path>` is the primary attack surface â€” it serves local disk to
  a webview. It MUST be root-jailed: reject `..`, absolute-path escapes,
  and symlink escapes (resolve + verify under an allowed root set).
- `POST /open` feeds a path to a subprocess (`nvim --server`). Validate
  the path exists and is inside the root before exec â€” never pass raw
  webview input to a shell.
- External renderers (ffmpeg) MUST carry a timeout + output-size cap;
  never block a request goroutine unbounded.
- ws hub uses a bounded send buffer and drops slow clients (the
  akm-graph `wsSendBuffer` pattern) â€” one stalled window must not stall
  the hub.
- No heavy work per hover â€” cursor moves render the thumbnail tier only;
  the live tier (video playback, STL orbit, full-res) is promoted only on
  explicit activate.
- No panics in handlers â€” marshal failure â†’ 500 (the akm-graph
  `writeJSON` pattern).

**Known limitations.** STL has no thumbnail (loads to orbit directly);
video poster needs `ffmpeg` present (degrade to a type icon if absent);
the webview needs a browser `--app` runtime (chromium/equivalent);
[[ft002]]'s proxied upstream d2 `--watch` page headers are unverified
([[poc006]] residual) â€” strip frame headers in `proxy.go` if present; the
akm/d2 reverse channel needs [[ft002]]/[[ft004]] pages to emit
`postMessage`, which is a Feature extension outside this spec (see Task
10).
