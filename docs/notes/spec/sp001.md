# Spec [[cat001]] [[board]]

## solves
[[us001|mark project as remote so terminals open against the remote host]]

## implements
[[im002|remote project dispatch via tmux-to-workstation]]

## problem
[[pn001|developer]] registers projects via [[ft001]] (`project add`,
`project go`, `project list`, persisted in
`~/.config/project/projects.yaml`). Today every entry is local: `project
go <name>` does `cd <path>` plus a local tmux session via `tmux-start`.

For a project that lives on a remote workstation, the developer must
remember the host, port, and remote working directory and manually run
`tmux-to-workstation connect -g <name> -- <ssh-args>` every time. The
project registry already knows the project — it should know the remote
binding too, so `project go` dispatches to the right primitive without
the developer carrying that state in their head.

Constraint: backwards-compat with existing local entries
([[us002|register a location as a project and navigate to it easily]]
is already shipped via [[ft001]] + [[im001]]; remote support must not
change the local path). The solution must respect [[adr0001]]
(nushell-primary for workflow-tooling) and [[adr0002]] (bash-vs-nushell
carve-outs) — extension stays inside the existing nushell module.

## solution
Widen [[ft001]] additively: each `projects.yaml` entry gains an optional
`ssh` field naming a `~/.ssh/config.d/<profile>` Host alias. When `ssh`
is set the project is remote and `path` is interpreted on the remote
side; when absent the entry stays local as today.

```yaml
projects:
  projectx:
    path: /home/user/repos/myproj
    ssh: workstation
```

`project go` branches on `ssh`:

- absent → today's behavior unchanged (local `cd $path` + `tmux-start`).
- present → invoke `tmux-to-workstation connect -g <name> --cwd <path>
  -- <ssh-alias>` (new `--cwd` flag propagates the remote working
  directory to the remote tmux session default-directory via the
  existing `tmux-start` plumbing on the remote side).

ssh alias resolves user/host/port via `~/.ssh/config.d/<profile>` —
project registry stores only the alias name, not the connection
parameters. Single source of truth for ssh config stays under
`~/.ssh/config.d/`.

Registration surface widens on existing `project add`:

- `project add <name> <path> --ssh <profile>` → remote entry.
- `project add <name> [path]` → local entry (unchanged).

`project list` adds a column annotating remote entries with their ssh
profile. Auto-cd in [[ft001]]'s `export-env` block skips entries with
`ssh` set — auto-ssh on every shell startup would fire repeatedly and
break offline; remote is opt-in via explicit `project go`.

Solution composes existing [[ft001]] (widened) + the shipped
`tmux-to-workstation connect` primitive. No new ADR — [[adr0001]] /
[[adr0002]] already cover nushell module extension; the change is
additive within the workflow-tooling layer ([[cat001]]).

Feature lifecycle: [[ft001]] stays `stable` and is body-refreshed
(schema + api_surface widened, no consumer breakage). New
implementation card [[im002]] minted at spec-refinement to record the
composed solution shape.

## roundtrip

End-to-end flow for `project go projectx` where projectx is registered
as `{ path: /home/user/repos/myproj, ssh: workstation }`.

```text
┌──────────┐  ┌────────────┐  ┌──────────────────────┐  ┌─────┐  ┌────────────────┐  ┌─────────────┐
│ dev      │  │ project.go │  │ tmux-to-workstation  │  │ ssh │  │ remote         │  │ remote tmux │
│ (shell)  │  │ (ft001)    │  │   connect            │  │     │  │ tmux-start     │  │ (server)    │
└────┬─────┘  └─────┬──────┘  └──────────┬───────────┘  └──┬──┘  └───────┬────────┘  └──────┬──────┘
     │ project go   │                    │                  │             │                  │
     │ projectx     │                    │                  │             │                  │
     ├─────────────►│                    │                  │             │                  │
     │              │ read yaml          │                  │             │                  │
     │              │ → ssh field set    │                  │             │                  │
     │              │ → branch=REMOTE    │                  │             │                  │
     │              │ tmux-to-workstation connect           │             │                  │
     │              │   -g projectx                         │             │                  │
     │              │   --cwd /home/user/repos/myproj       │             │                  │
     │              │   -- workstation                      │             │                  │
     │              ├───────────────────►│                  │             │                  │
     │              │                    │ ssh workstation  │             │                  │
     │              │                    │ "tmux-start start 0 projectx '' /home/user/repos/myproj"
     │              │                    ├─────────────────►├────────────►│                  │
     │              │                    │                  │             │ new-session -d   │
     │              │                    │                  │             │   -t projectx    │
     │              │                    │                  │             │   -s projectx_0  │
     │              │                    │                  │             │   -c /home/user/repos/myproj
     │              │                    │                  │             ├─────────────────►│
     │              │                    │                  │             │◄─target──────────┤
     │              │                    │◄─"projectx_0:0"──┤◄────────────┤                  │
     │              │                    │ create LOCAL thin session:                        │
     │              │                    │   tmux -L thin new-session -d -s thin_N           │
     │              │                    │   cmd = "ssh -t workstation TMUX_TARGET=projectx_0 bash -li"
     │              │                    │   set-option @remote_target=projectx_0:@<wid>     │
     │              │                    │ tmux -L thin attach -t thin_N                     │
     │              │◄───detach + reattach via thin socket──┤             │                  │
     │              │                    │ (thin pane runs the ssh command)                  │
     │              │                    │                  │ ssh -t      │                  │
     │              │                    │                  │ TMUX_TARGET=projectx_0 bash -li│
     │              │                    │                  ├────────────►│                  │
     │              │                    │                  │             │ bash -l reads    │
     │              │                    │                  │             │ TMUX_TARGET,     │
     │              │                    │                  │             │ attach -t        │
     │              │                    │                  │             │ projectx_0       │
     │              │                    │                  │             ├─attach──────────►│
     │ ◄────────────dev sees remote tmux pane, cwd = /home/user/repos/myproj─────────────────┤
```

Layer roles:

- `project go` — yaml branch; remote → dispatch to tmux-to-workstation.
- `tmux-to-workstation connect` — bootstraps the remote tmux session via
  ssh, then wraps the ssh pane in a **local thin tmux window** (via
  `tmux -L thin`) carrying connection metadata.
- Remote `tmux-start` — creates the remote session at `--cwd` path
  using tmux `new-session -c <dir>`.
- Remote tmux server — owns the actual session state; the local thin
  pane is just a viewport.

Detach / reconnect path (already shipped in tmux-to-workstation):

- ssh drops → thin pane goes `dead` or `broken` → `tmux-to-workstation
  revive` respawns the ssh command in-place and reattaches to the same
  `projectx_0` session on the remote tmux server. Remote state survives
  the disconnect.

  `tmux new-session -c <dir>`

  to remote `tmux-start`
- `docs/notes/ft001.md` — body refresh widening (additive, status stays
  `stable`)
- `docs/notes/im002.md` — new implementation card (minted in this
  refinement transition; spec-writing didn't anchor an `im###`)
- `docs/notes/spec/sp001.md` — this file
- `docs/product.md` — `>> [[im002]]` annotation on the us001 line

**Inherited constraints:**

- [[adr0001]] nushell-primary for workflow-tooling — every change lands
  in `.nu` files. Honored.
- [[adr0002]] bash-vs-nushell carve-outs — none apply (no bash added,
  no system-shell-startup touch). Honored.
- [[ft001]] `## api_surface` stays backwards-compatible — only
  additions (`--ssh` flag, new `ssh` column, dispatch in `go`).

**Anti-patterns to reject during implementation:**

- Auto-ssh on shell startup. `export-env` MUST skip remote entries —
  firing ssh on every shell open is surprising, breaks offline, and
  could spam the remote host.
- Silent fallback to local when ssh profile missing. Let ssh resolution
  errors bubble; never paper over with `cd $path` (the path is the
  remote path, doesn't exist locally).
- `--ssh` accepted without `path`. PWD-default makes no sense for
  remote (PWD is local). Validate at `project add` time.
- Repurposing the `ssh` field as a record/object in v2. Lock it as a
  string (profile name only) now and stick.
- Bypassing `tmux-to-workstation` as the dispatch primitive — keep
  exactly one remote-attach entry point.

**Known limitations (not addressed by this spec):**

- ssh profile resolution depends on the user maintaining
  `~/.ssh/config.d/<profile>`. `project add` does not auto-create or
  validate the Host entry.
- Remote `--cwd` requires the path to exist on the remote at connect
  time; `tmux-start` on remote fails loud if not. No pre-flight check
  from local side.
- Project registry is local-only. No mechanism for syncing
  `projects.yaml` to a remote machine — out of scope.

## tasks

### Task 1: Add optional `cwd` positional to `tmux-start`

#### type
task

#### effort
1h

#### depends
- — none

#### files_touched
- `nushell/actions/tmux-start`

#### success_criteria
- New 5th positional arg `cwd?: string` after the existing `uid?`.
- When `cwd` non-empty, every `tmux new-session` call inside the script
  receives `-c <cwd>`.
- When `cwd` omitted or empty string, no `-c` flag added — today's
  behavior preserved.
- Verify with `tmux display-message -t <session>:0 -p '#{session_path}'`
  matches the cwd value.

#### edge_cases
- `cwd` path does not exist locally → `tmux` itself errors; propagate,
  do not swallow.
- Empty-string cwd → treat as omitted; no `-c` emitted.
- Path with spaces → use nushell string interpolation, never raw
  concatenation.

#### test_plan
- Manual: `tmux-start start 0 testsess "" /tmp; tmux display-message -t
  testsess_0:0 -p '#{session_path}'` → expect `/tmp`. Catches: cwd not
  threaded into `new-session`.
- Manual: `tmux-start start 0 testsess2` → expect today's session_path
  (developer's PWD). Catches: regression from the additive change.

### Task 2: Add `--cwd` flag to `tmux-to-workstation connect`

#### type
task

#### effort
2h

#### depends
- Task 1

#### files_touched
- `nushell/actions/tmux-to-workstation`

#### success_criteria
- `connect` accepts `--cwd <path>` named flag.
- When set, the remote `tmux-start` invocation becomes
  `~/.local/bin/tmux-start start 0 <group> '' <cwd>`.
- When omitted, today's invocation unchanged.
- Remote tmux session's default-directory equals the `--cwd` value
  (verifiable on remote via `tmux display-message -p '#{session_path}'`).

#### edge_cases
- cwd containing spaces → quoted correctly in the ssh command (nushell
  interpolation, not string concat).
- Remote `tmux-start` is an older version that ignores the 5th
  positional → remote session still created; default-directory falls
  back to `$HOME`; no crash. Document as known limitation if observed.
- `--cwd` and `--group` both set → both honored; group still drives the
  session name.

#### test_plan
- Manual on a real remote: `tmux-to-workstation connect -g test --cwd
  /tmp -- <test-host>` → ssh into the thin pane shows remote shell with
  PWD `/tmp` after attaching. Catches: cwd ignored or dropped.
- Manual with a path containing a space: `--cwd "/tmp/with space"` →
  remote session lands in the spaced path, no parse error.

  entry: <name>".
