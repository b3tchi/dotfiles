# Spec [[cat001]] [[cat002]] [[board]]

## problem

**Hotfix ‚Äî the [[sp014]] secret filter is inert on this host.** The shipped
shared-clipboard-history system drops password-manager copies by matching the
X selection target atom `application/x-kde-passwordManagerHint`, enforced at
two independent gates: the copyq automatic command in
[copyq/commands.ini](../../../copyq/commands.ini) (native capture path) and
the TARGETS pre-check in
[i3/scripts/clip-feed.sh](../../../i3/scripts/clip-feed.sh) (the `:10 ‚Üí :0`
feeder path, which needs its own gate because `copyq add` bypasses copyq's
automatic commands entirely). Both gates were verified load-bearing by
adversarial audit ‚Äî stripping either one demonstrably leaks.

The installed password manager is **gopass** (`gopass 1.16.1-2`). No
KeePassXC, no Bitwarden, no Enpass, no 1Password ‚Äî checked via `pacman` and
`flatpak`. gopass copies through vendored `atotto/clipboard`, which shells out
to `xclip`/`xsel` and takes ordinary selection ownership. **It advertises no
hint atom.** Neither gate ever fires.

**Symptom.** `gopass show -c <entry>` puts a password on CLIPBOARD; copyq
captures it into history as plain text; the quickshell picker lists it.
gopass's `core.cliptimeout = 45` spawns a `gopass unclip` process that wipes
the *clipboard* after 45s ‚Äî the *history* entry is untouched and remains
browsable until the copyq server exits.

**How this shipped.** Every test that proved the filter used the python-xlib
multi-target selection owner in
[copyq/test-copyq.sh](../../../copyq/test-copyq.sh), which advertises the atom
because the fixture is told to. That validates the *mechanism*, not its
*applicability*. No test asked whether anything on this machine emits that
atom. The gap is in the spec, not the implementations ‚Äî [[sp014]] named the
KeePassXC/Bitwarden convention and never checked it against the host.

**Severity: P1.** Confidentiality exposure, local only. Not P0 ‚Äî no data loss,
no outage, and history is memory-only (per-tab `store_items=false`), so
exposure ends at logout.

**Blast radius.** Any host running [[sp014]] with gopass as its password
manager; today that is the native desktop. Fires on every `gopass show -c`. No
bad state has accumulated ‚Äî copyq has never run persistently ‚Äî but
`clip-feed.sh` is already autostarted, so the exposure begins the moment a
copyq server starts.

**Rollback.** Available and cheap: [[sp014]] is not yet in daily use. Dropping
the two `exec_always` lines from `i3/config.d/native.conf` (or simply not
starting copyq) removes the exposure entirely. Rolling forward is preferred,
but there is no schedule pressure.

**Minimal-fix shape.** A gopass copy wrapper that owns the X selection while
advertising both `UTF8_STRING` and
`application/x-kde-passwordManagerHint`, so that the two already-shipped gates
drop the item **unmodified** ‚Äî no change to `commands.ini`, no change to
`clip-feed.sh`. The selection-owner technique already exists in-tree as the
test fixture in `copyq/test-copyq.sh`; the fix promotes it from test
scaffolding to a small production helper. Per [[adr0002]] the glue is bash,
with python-xlib for the multi-target owner (a dependency the shipped test
fixtures already carry). Strongest available guarantee: the payload is never
even requested from the owning process.

**Accepted residual risk ‚Äî the silent-failure mode.** The wrapper protects
only copies made *through* it. Plain `gopass show -c` continues to leak, and
it fails **silently** ‚Äî nothing warns that a password entered history. This
shape was chosen over unclip-process detection (which would protect the plain
command but couples the filter to a gopass implementation detail and races at
the edges). The spec must therefore treat discoverability as part of the fix,
not an afterthought: making the wrapper the path of least resistance matters
as much as the mechanism.

**Open question the spec must settle ‚Äî atom form.** Matching is exact, with no
prefix normalisation ([[poc010]]). The shipped `commands.ini` rule uses the
prefixed `application/x-kde-passwordManagerHint`; clipcat's default is the
bare `x-kde-passwordManagerHint`. Since gopass emits neither today, the wrapper
gets to *choose* which form it advertises ‚Äî and it must advertise whatever the
shipped gates actually match. Verify against the real rule rather than
assuming; a mismatch here silently reproduces the exact bug being fixed.

**Discovered, out of scope, file separately.** [[sp014]]'s `## solution`
promises a new feature `ft007` (`shared-clipboard-history`) "minted with this
spec". It was never created ‚Äî the tree holds only `ft001`‚Äì`ft006`. The shipped
capability has no feature zettel. That belongs to [[sp014]]'s outstanding
`spec-retro`, not to this hotfix.
