# Spec [[cat002]] [[board]]

## problem
The i3/sway mode-hint bar (the name-pill + keyboard-hint strip shown while
in `resize`, `screenshot`, or the system mode) is rendered inline in
`quickshell/config/Bar.qml` with three smells:

1. **Dead code.** The "Mode indicator" Rectangle (`Bar.qml` ~533-554) is
   nested in the `leftSide` Row (`visible: currentMode === "default"`) yet is
   itself `visible: currentMode !== "default"` тАФ it can never render. The
   live strip is the separate "Mode hints overlay" Row (~558-605).
2. **Color drift.** Colors are hardcoded lowercase (`#cb4b16`, `#152024`,
   `#fdf6e3`) тАФ the same Manjaro palette values the dialog family already
   names in [[ft008]]'s DialogTheme (`#CB4B16`/`#152024`/`#FDF6E3`), but
   duplicated and free to drift.
3. **Data/render tangle.** The `modeHints(mode)` map and the display-name
   ternary live inline in `Bar.qml`, mixed with the QML render tree.

This is the same refactor shape as [[sp017]], which extracted the shared
combobox control [[ft008]] (Combo + DialogTheme) out of the four dialogs.
Here the target is a new horizontal capability **ModeBar** тАФ a reusable
`quickshell/config/Common/ModeBar.qml` component plus a `ModeBarTheme`
singleton and a mode->hints registry тАФ minted as `ft###` at spec-writing.

**Boundary.** ModeBar renders any i3 mode (name-pill + hint strip) from a
static registry; ModeBarTheme owns the constants тАФ **highlight kept red**
(`#CB4B16`, key text + underline), the rest following the Manjaro palette
(`#152024` pill bg, `#FDF6E3` fg, `#707880` muted, Iosevka Nerd Font). Data
lives in the registry, render in the component (fixes smell 3).

**Category** [[cat002]] display-platform (same as [[ft008]]).

**Consumers.** Today one host, `Bar.qml`. The reuse axis is (a) per-mode
hint data and (b) future bar hosts тАФ sway bar, the phone `BarChin`, a
second-monitor bar. The mode set is defined by [[ft003]] (i3-config
layering) and the `screenshot` mode exists only to paint this strip
([[ft006]] region-capture, whose `qs-screenshot.sh` documents the
dependency). No `im###` documents the current inline mode-bar code тАФ it is
undocumented Bar.qml glue being promoted to a feature.

**Constraints.** No binding ADR under [[cat002]] touches bar rendering
(adr0010 is the clipboard publish contract, unrelated). Parity constraint:
the extracted render must match the pre-refactor pixels (same approach
[[sp017]] took for the dialogs).

**Migration.** `Bar.qml` тАФ delete the dead Mode-indicator block, drop the
`modeHints()` function, replace the overlay Row with a `ModeBar` instance
wired to the existing i3 mode-subscription Process (~90 lines removed).

## solution
Extract two new files under `quickshell/config/Common/`, mirroring the
`DialogTheme` + `Combo` split that [[sp017]] shipped for [[ft008]] тАФ a
proven extraction (commit 89ea553), so no PoC is warranted.

**`ModeBarTheme.qml`** (singleton, sibling to DialogTheme). Owns the
constants and the data: `highlight` kept red (`#CB4B16`, key text +
name-pill underline) with the rest following the Manjaro palette
(`pillBg` `#152024`, `fg` `#FDF6E3`, `muted` `#707880`, `font` Iosevka
Nerd Font); a `hints` registry mapping mode name -> `[{key, label}]`
lifted VERBATIM from `Bar.qml`'s current `modeHints()`; and
`displayName(mode)` for the pill label. Data lives here, decoupled from
render (fixes the Bar.qml data/render tangle).

**`ModeBar.qml`** (an `Item`, NOT a `Window` тАФ same pattern as
[[ft008]]'s Combo). Props: `mode` (the current i3/sway mode; `"default"`
renders nothing) and `fontSize` (the bar passes its sway/i3-aware size).
Renders the name-pill (`pillBg` background, red underline, bold `fg`
label) followed by the hint strip (red `key`, cream `label`) driven by
`ModeBarTheme.hints[mode]`, all `Text.NativeRendering` to match the bar.

Both registered in `Common/qmldir` alongside the existing dialog
components. Consumer `quickshell/config/Bar.qml` migrates: delete the
dead Mode-indicator block, drop the inline `modeHints()`, and replace the
"Mode hints overlay" Row with `ModeBar { mode: root.currentMode;
fontSize: root.fontSize }` wired to the existing mode-subscription
`Process` (~90 lines removed).

This spec mints a new capability [[ft009]] (ModeBar, under [[cat002]]) at
spec-refinement time. No runtime feature is consumed тАФ ModeBar is a
sibling to [[ft008]], sharing only the `Common/` import root and the
`Text.NativeRendering` convention. No binding ADR under [[cat002]]
constrains bar rendering (adr0010 is the unrelated clipboard-publish
contract). The one binding constraint is pixel parity with the
pre-refactor render, enforced the same way [[sp017]] enforced dialog
parity тАФ the extracted colors and geometry reproduce the current
`Bar.qml` values exactly.

**Trade-offs.** (1) One host today (`Bar.qml`) тАФ accepted; the capability
is promoted on the reuse axis of per-mode hint data plus future bar hosts
([[ft003]]'s mode set, the phone `BarChin`, a second-monitor bar).
(2) `ModeBarTheme` kept separate from [[ft008]]'s `DialogTheme` rather
than merged тАФ cost is two singletons carrying overlapping palette values,
benefit is bar/dialog theme independence.
