# Spec [[cat002]] [[board]]

## problem
The i3/sway mode-hint bar (the name-pill + keyboard-hint strip shown while
in `resize`, `screenshot`, or the system mode) is rendered inline in
`quickshell/config/Bar.qml` with three smells:

1. **Dead code.** The "Mode indicator" Rectangle (`Bar.qml` ~533-554) is
   nested in the `leftSide` Row (`visible: currentMode === "default"`) yet is
   itself `visible: currentMode !== "default"` — it can never render. The
   live strip is the separate "Mode hints overlay" Row (~558-605).
2. **Color drift.** Colors are hardcoded lowercase (`#cb4b16`, `#152024`,
   `#fdf6e3`) — the same Manjaro palette values the dialog family already
   names in [[ft008]]'s DialogTheme (`#CB4B16`/`#152024`/`#FDF6E3`), but
   duplicated and free to drift.
3. **Data/render tangle.** The `modeHints(mode)` map and the display-name
   ternary live inline in `Bar.qml`, mixed with the QML render tree.

This is the same refactor shape as [[sp017]], which extracted the shared
combobox control [[ft008]] (Combo + DialogTheme) out of the four dialogs.
Here the target is a new horizontal capability **ModeBar** — a reusable
`quickshell/config/Common/ModeBar.qml` component plus a `ModeBarTheme`
singleton and a mode->hints registry — minted as `ft###` at spec-writing.

**Boundary.** ModeBar renders any i3 mode (name-pill + hint strip) from a
static registry; ModeBarTheme owns the constants — **highlight kept red**
(`#CB4B16`, key text + underline), the rest following the Manjaro palette
(`#152024` pill bg, `#FDF6E3` fg, `#707880` muted, Iosevka Nerd Font). Data
lives in the registry, render in the component (fixes smell 3).

**Category** [[cat002]] display-platform (same as [[ft008]]).

**Consumers.** Today one host, `Bar.qml`. The reuse axis is (a) per-mode
hint data and (b) future bar hosts — sway bar, the phone `BarChin`, a
second-monitor bar. The mode set is defined by [[ft003]] (i3-config
layering) and the `screenshot` mode exists only to paint this strip
([[ft006]] region-capture, whose `qs-screenshot.sh` documents the
dependency). No `im###` documents the current inline mode-bar code — it is
undocumented Bar.qml glue being promoted to a feature.

**Constraints.** No binding ADR under [[cat002]] touches bar rendering
(adr0010 is the clipboard publish contract, unrelated). Parity constraint:
the extracted render must match the pre-refactor pixels (same approach
[[sp017]] took for the dialogs).

**Migration.** `Bar.qml` — delete the dead Mode-indicator block, drop the
`modeHints()` function, replace the overlay Row with a `ModeBar` instance
wired to the existing i3 mode-subscription Process (~90 lines removed).

## solution
Extract two new files under `quickshell/config/Common/`, mirroring the
`DialogTheme` + `Combo` split that [[sp017]] shipped for [[ft008]] — a
proven extraction (commit 89ea553), so no PoC is warranted.

**`ModeBarTheme.qml`** (singleton, sibling to DialogTheme). Owns the
constants and the data: `highlight` kept red (`#CB4B16`, key text +
name-pill underline) with the rest following the Manjaro palette
(`pillBg` `#152024`, `fg` `#FDF6E3`, `muted` `#707880`, `font` Iosevka
Nerd Font); a `hints` registry mapping mode name -> `[{key, label}]`
lifted VERBATIM from `Bar.qml`'s current `modeHints()`; and
`displayName(mode)` for the pill label. Data lives here, decoupled from
render (fixes the Bar.qml data/render tangle).

**`ModeBar.qml`** (an `Item`, NOT a `Window` — same pattern as
[[ft008]]'s Combo). Props: `mode` (the current i3/sway mode; `"default"`
renders nothing) and `fontSize` (the bar passes its sway/i3-aware size).
Renders the name-pill (`pillBg` background, red underline, bold `fg`
label) followed by the hint strip (red `key`, cream `label`) driven by
`ModeBarTheme.hints[mode]`, all `Text.NativeRendering` to match the bar.

Both registered in `Common/qmldir` alongside the existing dialog
components. Consumer `quickshell/config/Bar.qml` migrates: delete the
dead Mode-indicator block, drop the inline `modeHints()`, and replace the
"Mode hints overlay" Row with `ModeBar { mode: root.currentMode;
fontSize: root.fontSize }` wired to the existing mode-subscription
`Process` (~90 lines removed).

This spec mints a new capability [[ft009]] (ModeBar, under [[cat002]]) at
spec-refinement time. No runtime feature is consumed — ModeBar is a
sibling to [[ft008]], sharing only the `Common/` import root and the
`Text.NativeRendering` convention. No binding ADR under [[cat002]]
constrains bar rendering (adr0010 is the unrelated clipboard-publish
contract). The one binding constraint is pixel parity with the
pre-refactor render, enforced the same way [[sp017]] enforced dialog
parity — the extracted colors and geometry reproduce the current
`Bar.qml` values exactly.

**Trade-offs.** (1) One host today (`Bar.qml`) — accepted; the capability
is promoted on the reuse axis of per-mode hint data plus future bar hosts
([[ft003]]'s mode set, the phone `BarChin`, a second-monitor bar).
(2) `ModeBarTheme` kept separate from [[ft008]]'s `DialogTheme` rather
than merged — cost is two singletons carrying overlapping palette values,
benefit is bar/dialog theme independence.

## plan
Executed under the implementation card [[im007]]
(bar-mode-strip-on-modebar), which anchors the new capability [[ft009]]
(mode-bar-control) — both minted at this refinement, `proposed` until
the spec ships ([[ft008]]/[[im006]] precedent from [[sp017]]).

Binding decisions: ADR survey under [[cat002]] confirms the
`## solution` claim — no Accepted decision touches bar rendering
([[adr0004]] xrdp stack, [[adr0005]] llvmpipe gating, [[adr0009]]
preview-d embed, [[adr0010]] clip publish contract, [[adr0011]] clip
backend are all orthogonal; adr0006 is Superseded). One convention
decision binds: [[adr0002]] — the test suite is bash (UI observed
indirectly, logic asserted via sh-visible effects). Consumed surfaces:
[[ft003]] mode set (`resize`, `screenshot`, `$mode_system`) read-only,
no config/bind edits; [[ft006]] untouched — its `screenshot` mode keeps
painting through the extracted strip.

AC — derived from `## problem` boundary (feature-promotion spec, no
source `us###`); tasks reference them as:
- AC1 — pixel parity: extracted render reproduces the pre-refactor
  strip exactly — pill bg `#152024`, fg `#FDF6E3`, highlight `#CB4B16`
  (key text + 2px pill underline), muted `#707880`, Iosevka Nerd Font,
  `Text.NativeRendering`, pill width = label width + 14, bottom-anchored
  text with 1px margin, 4px pill/hints gap, two-space hint separators
- AC2 — dead code deleted: the unreachable "Mode indicator" Rectangle
  (visible-contradiction block inside `leftSide`) is gone
- AC3 — data/render decoupled: hint registry + display names live in
  `ModeBarTheme`; `Bar.qml` retains no `modeHints()`, no mode-strip
  color literals, no `(l)ock` sniff
- AC4 — mode behavior verbatim: `"default"` renders nothing; `resize` /
  `screenshot` / the long `$mode_system` string render pill + hints
  (system pill reads "system"); unknown modes fall back to pill
  "system" + `[{key: "", label: <raw mode>}]`

Refinement delta (one, from the `## solution` sketch): the sketch says
the strip is driven by `ModeBarTheme.hints[mode]`, but the system
mode's IPC name is the full `$mode_system` string (`(l)ock, (e)xit,
switch_(u)ser, ...` — `i3/config.common:255`), which `Bar.qml` matches
via `indexOf("(l)ock")`, and unknown modes get a fallback row. A plain
name-keyed map cannot reproduce either. The registry stays data
(`hints` keyed `resize`/`screenshot`/`system`) with
`resolve(mode)` / `hintsFor(mode)` / `displayName(mode)` lookups lifted
VERBATIM from the current code — data still lives in the singleton
(smell 3 stays fixed), behavior stays byte-identical (AC4).

Conventions:
- Test discipline copied from `test-combo.sh` PHASE 0/1: throwaway
  harness config in `$TMP` importing the repo `Common/` by absolute
  path, `CASE <name> <payload>` lines asserted by the script, Xvfb
  display :98, isolated `XDG_RUNTIME_DIR`/`XDG_CACHE_HOME`, named
  scenarios, `SELFTEST=1` wires one wrong expectation that MUST fail.
- Singleton style follows `Session.qml`/`DialogTheme.qml`
  (`pragma Singleton` + qmldir `singleton` entry).
- Theme constants are UPPERCASE hex (`#CB4B16`) matching `DialogTheme`;
  QML color parsing is case-insensitive so the render is identical to
  the current lowercase literals.
- All strip text renders `Text.NativeRendering` (matches the bar).

Anti-patterns (auto-reject in review):
- Duplicating any `ModeBarTheme` constant at a call site, or leaving
  `modeHints` / mode color literals / the `(l)ock` sniff in `Bar.qml`
  after T3.
- Merging `ModeBarTheme` into `DialogTheme` — trade-off (2) in
  `## solution` chose theme independence.
- ModeBar acquiring a `Window` or its own mode-subscription `Process` —
  the host owns chrome and state ([[ft009]] api_surface).
- Retuning any color or geometry value "while in there" — parity is the
  contract (AC1).
- Touching the workspace-tab Repeater in `Bar.qml` — it uses the same
  hex values for a different concern; theming it is out of scope.
- Editing i3 configs, binds, or `qs-screenshot.sh` — [[ft003]]/[[ft006]]
  surfaces are consumed as-is.

Known limitations:
- Unknown/future modes display pill "system" + the raw mode name as a
  hint row — current behavior kept verbatim; widening the registry is a
  follow-up when [[ft003]] grows a mode.
- The headless suite asserts properties and relative geometry, not
  rasterized pixels; live visual parity is a manual merge-time check
  ([[sp017]] precedent).
- One host today (`Bar.qml`); `BarChin`/sway-bar adoption is future
  scope on the reuse axis, not built here.
- [[ft009]]/[[im007]] stay `proposed` through refinement; the
  `→ stable`/`→ accepted` flips land at work-merge/spec-retro.

#### success_criteria
- `ModeBarTheme` exposes exactly the AC1 constants — `highlight`
  #CB4B16, `pillBg` #152024, `fg` #FDF6E3, `muted` #707880, `font`
  "Iosevka Nerd Font" — harness dumps and asserts each value headless
  (AC1).
- `hints` registry rows are byte-identical to `Bar.qml`'s current
  `modeHints()` returns — resize 6 rows (`j`/`k`/`l`/`;`/`←↓↑→`/`Esc`
  with their labels), screenshot 4 rows, system 8 rows — asserted as
  JSON CASE lines against expectations pinned from the pre-refactor
  function (AC3, AC4).
- `resolve` / `displayName` / `hintsFor` reproduce current behavior:
  `displayName("resize")` = "resize", `displayName("screenshot")` =
  "screenshot", `displayName(<full $mode_system string>)` = "system",
  `displayName("somefuture")` = "system"; `hintsFor("somefuture")` =
  `[{key: "", label: "somefuture"}]` (AC4).
- `Bar.qml` is NOT modified in this task — extraction is additive;
  `git diff --name-only` shows only the three files above.

#### edge_cases
- A mode string containing `(l)ock` anywhere resolves to system
  (`indexOf` semantics preserved, not equality).
- Empty-string mode → displayName "system", `hintsFor` returns the
  fallback row with empty label — matches current `modeHints("")`.
- Unicode hint labels (`←↓↑→`, `←`) survive the CASE round-trip
  byte-exact.
- Registry rows are read-only data — calling `hintsFor` twice returns
  equal content (no accumulation/mutation).

#### test_plan
`test-mode-bar.sh` PHASE 0, mirroring `test-combo.sh` PHASE 0: write a
throwaway harness config into `$TMP` that imports the repo `Common/`
by absolute path, evaluates each scenario, prints `CASE <name> <json>`
lines, exits; quickshell runs it under Xvfb (:98) and the script
asserts expected lines. Named scenarios: `theme-constants`,
`hints-resize-verbatim`, `hints-screenshot-verbatim`,
`hints-system-verbatim`, `system-long-name-resolves` (the full
`$mode_system` string from `i3/config.common:255` — an equality-match
mutant fails), `unknown-mode-fallback`, `display-names`,
`empty-mode-fallback`. The pinned JSON is lifted from the CURRENT
`modeHints()` — any drifted label or dropped row fails. `SELFTEST=1`
flips one expectation wrong and must FAIL, proving the harness can
fail.

### Task 2: Build the ModeBar component

#### bd dotfiles-80px.2

#### type
feature

#### effort
5h

#### depends
Task 1

#### files_touched
- quickshell/config/Common/ModeBar.qml (new)
- quickshell/config/Common/qmldir
- quickshell/test-mode-bar.sh (PHASE 1)

#### success_criteria
- ModeBar is an `Item` with exactly the [[ft009]] api_surface props —
  `mode` (string, default "default") and `fontSize` (int) — and is
  instantiable with only those two set; it contains no `Window` and no
  `Process` (`grep -cE 'Process|Window' Common/ModeBar.qml` is 0 —
  host owns chrome and state).
- `mode: "default"` → `visible` false; each known mode renders the
  name-pill (bg `pillBg`, 2px top underline `highlight`, bold `fg`
  label bottom-anchored with 1px margin, pill width = label
  implicitWidth + 14) then a 4px gap then the hint rows (key in
  `highlight` bold, single-space, label in `fg`, two-space separator
  between entries, bottom-anchored 1px) — structure, colors, and the
  width/underline/gap/margin deltas asserted via CASE property dumps
  (AC1, AC4).
- Every `Text` uses `font.family: ModeBarTheme.font`,
  `font.pixelSize` = the `fontSize` prop, and
  `renderType: Text.NativeRendering` (AC1).
- Runtime mode flips (default → resize → default → screenshot) update
  visibility and hint content with no stale rows left behind.

#### edge_cases
- Unknown mode renders pill "system" + one raw-name hint row (the
  `hintsFor` fallback path exercised through the component).
- `fontSize` change propagates to every Text (phone/sway sizes differ
  from desktop — the host passes its own).
- Direct flip resize → screenshot (no "default" between) rebuilds the
  hint Repeater cleanly.
- A very long unknown mode name grows `implicitWidth` — no clipping or
  eliding introduced (matches current overflow behavior).

#### test_plan
`test-mode-bar.sh` PHASE 1: persistent harness Window in `$TMP` hosts
`ModeBar` with an `IpcHandler` that sets `mode`; after each flip the
harness prints CASE lines carrying `visible`, `implicitWidth`, pill
`width - label.implicitWidth` (must be 14 — a padding-retune mutant
fails), underline height (2), gap width (4), and per-element color +
renderType values. Named scenarios: `default-invisible`,
`resize-pill-and-hints`, `screenshot-hints`, `system-long-name`,
`unknown-fallback`, `mode-flip-no-stale`, `geometry-deltas`. Coverage
is behavioral — no scenario asserts mere property existence; color
assertions compare against `ModeBarTheme.*` values so a
theme-duplication mutant (hardcoded literal in ModeBar) that drifts
also fails.
