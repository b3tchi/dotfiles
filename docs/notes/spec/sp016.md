# Spec [[cat001]] [[cat002]] [[board]]

## extends [[ft007]]

## problem

[[ft007]] (shared-clipboard-history, shipped by [[sp014]]) runs on copyq
16.0.0. copyq is a Qt6 GUI application pressed into service as a headless
daemon: it drags in `qt6-svg`, `knotifications`, `kstatusnotifieritem`,
`kguiaddons`, `qca-qt6` and `qtkeychain-qt6`, links 96 libraries, and holds
everything the user copies. Measured on the native desktop: RSS 63 MiB,
**private 16 MiB**, 7 threads, 0 idle CPU.

The runtime cost is mild — 16 MiB private and zero idle CPU are noise on a
desktop, and **this is not a performance problem**. Two other costs are real:

1. **Dependency surface on a secrets-adjacent daemon.** `qtkeychain`, `qca`, a
   scripting engine, D-Bus and notification integration are attack surface,
   and ft007 uses roughly none of it — the shipped consumer calls only
   `read`, `add` and `size`.
2. **The Qt6 chain forecloses the small platforms.** [[ft007]] is scoped to
   graphical i3 sessions today, so Termux and proot never see it. That scope
   is a consequence of the backend, not a product decision — a clipboard
   history is exactly the sort of thing a phone session wants.

**[[poc010]] validated clipcat 0.25.0 (official `extra`, not AUR) as a
replacement**, and on the security property it is *stronger* than what
shipped: with `sensitive_mime_types` configured, clipcat requested only
`TARGETS` from a hint-bearing selection owner and never `UTF8_STRING` — the
payload never left the owning process. Both negative controls flipped, so the
fixture demonstrably delivered and the filter is demonstrably live. Footprint:
**RSS 12.3 MiB, private 9.9 MiB, 4 linked libs**, no Qt, no KDE, no GUI.

**What this does NOT fix.** `dotfiles-cyg` — the secret filter is inert
against gopass, which advertises no hint atom. clipcat's filter is inert for
exactly the same reason. Swapping backends buys a lighter daemon that stores
passwords in plaintext precisely as copyq does. [[sp015]] is the fix and is
backend-agnostic (it makes gopass *emit* the atom, which both backends drop
on), so the two are independent — but shipping this one alone must not be
mistaken for improving the security posture.

**The delta.**

*Memory-only changes mechanism.* copyq guarantees it as a daemon property
(per-tab `store_items=false`, nothing on disk ever). clipcat has no
equivalent: `history_file_path` is mandatory, is a *directory*, and clips are
written continuously during the session — not just at exit. `/dev/null` makes
the daemon refuse to start; `max_history = 0` captures nothing. The property
is reproducible by pointing `history_file_path` at `$XDG_RUNTIME_DIR` (tmpfs,
0700, destroyed by logind at last logout) plus `clear_history_on_start`, but
as **config discipline rather than a daemon guarantee**. That is a genuine
weakening: a future config edit can silently start persisting secrets to disk,
where under copyq no edit could. The spec must decide how that invariant is
defended (a test asserting nothing lands on real disk is the obvious answer,
and one already exists in `copyq/test-copyq.sh` to adapt).

*The read surface changes shape.* `clipcatctl list` emits
`<hex-id>: <100-char preview>` with no JSON, ids are content hashes rather
than positional indices, and full entry text needs `get <id>` per entry. So
[[ft007]]'s `## api_surface` moves from row-keyed to id-keyed, and listing
becomes N+1 calls. This is an **incompatible surface change**, but every
consumer is in-tree and changes with it (`ClipHistory.qml` via `qs-clip.sh`,
plus `clip-set.sh`), so the call is a **body refresh of [[ft007]], not a
supersession** — the capability, the keybind and the user-visible contract are
unchanged. Content-hash ids are arguably *better* than positional rows, which
shift under the reader; the row-shift hazard was a real bug source in
[[sp014]]'s test suites.

*The feeder's own gate stays mandatory.* [[poc010]] Q3 confirmed
`clipcatctl insert` does not re-trigger the watcher — which is convenient for
seeding, and means inserted items **bypass `sensitive_mime_types` exactly as
`copyq add` bypasses copyq's automatic commands.** So `clip-feed.sh` keeps
both its TARGETS gate and the [[sp015]]-era post-read re-check; nothing about
this swap makes the feeder's own filtering redundant.

*Multi-display needs orchestration.* One `clipcatd` binds one socket; the
native host runs `:0` and `:10` ([[adr0004]]). [[poc010]] ran two daemons on
separate `--server-endpoint` sockets with zero cross-talk, but which sessions
get a daemon — and whether the feeder still bridges `:10 → :0` or each display
simply keeps its own history — is an open design question this spec must
settle, not a mechanical port.

**Binding decisions.** [[adr0002]] (bash for tight glue) governs the rewritten
`qs-clip.sh` backend layer. [[adr0010]] (the picker publishes, never pastes) is
untouched — the swap is below that decision. No accepted ADR is overturned:
[[sp014]] deliberately declined to promote the backend choice to an ADR,
keeping it in `## solution`. **That decision now looks wrong** — a vendor
choice that gets revisited is exactly what an ADR is for, and the swap should
mint one at spec-retro so the next person sees why clipcat, and what copyq
cost.

**Open questions for `spec-writing`.**

- **Atom form.** Matching is exact with no prefix normalisation ([[poc010]]).
  clipcat's default is bare `x-kde-passwordManagerHint`; the shipped copyq rule
  uses prefixed `application/x-kde-passwordManagerHint`. Neither is verified
  against a real emitter — no password manager on this host emits either
  ([[sp015]]). Configure clipcat to match **both** forms unless there is a
  reason not to.
- **Rollback.** Keep `copyq/` in-tree behind an unused rotz dot for one cycle,
  or delete it in the same change? The swap touches every layer of a system
  that shipped two days earlier.
- **Termux/proot.** Does this spec actually extend [[ft007]] to those
  platforms, or only remove the blocker and leave the extension to a later
  story? Scoping both into one change risks a large, hard-to-review spec.

## solution

Replace copyq with **clipcat 0.25.0** as [[ft007]]'s backend on the platforms
that run it today — native `:0`, xrdp `:10`, WSL. [[ft007]]'s user-visible
contract is unchanged: same `$mod+Shift+v` keybind via [[ft003]], same picker,
same publish-don't-paste behaviour per [[adr0010]]. What changes is everything
below `qs-clip.sh`. Validated end-to-end by [[poc010]].

**Four composed parts:**

1. **Backend** — a `clipcatd` per session, configured with
   `sensitive_mime_types` carrying **both** atom forms (bare
   `x-kde-passwordManagerHint` and prefixed
   `application/x-kde-passwordManagerHint`), since matching is exact with no
   prefix normalisation and no emitter on this host is available to verify
   against ([[sp015]]). Memory-only is reproduced by pointing
   `history_file_path` at a `$XDG_RUNTIME_DIR` subdirectory (tmpfs, 0700,
   destroyed by logind at last logout) plus `clear_history_on_start`.
   Config version-controlled and rotz-linked, as `copyq/dot.yaml` is today.

2. **The `qs-clip.sh` backend layer, rewritten** ([[adr0002]] — bash for tight
   glue). `list` moves from row-keyed to id-keyed: `clipcatctl list` emits
   `<hex-id>: <100-char preview>`, so full entry text needs `get <id>` per
   entry (N+1 calls). Content-hash ids are *stabler* than positional rows,
   which shift under the reader — the row-shift hazard that broke two of
   [[sp014]]'s suites disappears. `ClipHistory.qml` keeps talking only to
   `qs-clip.sh`, so the QML changes at most in what it passes back, not in
   what it knows.

3. **Feeder + publisher ported.** `clip-feed.sh` swaps `copyq add` for
   `clipcatctl insert` and **keeps both of its secret gates** — the TARGETS
   pre-check and the post-read re-check. This is not optional: [[poc010]] Q3
   confirmed `clipcatctl insert` does not re-trigger the watcher, so inserted
   items bypass `sensitive_mime_types` exactly as `copyq add` bypassed copyq's
   automatic commands. `clip-set.sh` is backend-agnostic below its row lookup
   and keeps its 0/1/2 exit contract verbatim.

4. **Autostarts** in the [[ft003]] overlays, replacing `copyq --start-server`.
   One `clipcatd` binds one socket, and the native host runs two displays
   ([[adr0004]]) — `native.conf` therefore runs a daemon per session on
   distinct `--server-endpoint` sockets ([[poc010]] Q5: zero cross-talk),
   with `clip-feed.sh` continuing to bridge `:10 → :0` so one history remains
   the shared one. `wsl.conf` stays single-daemon, no feeder.

**Scope: swap only.** Termux/proot extension is explicitly out. Removing the
Qt6 chain unblocks those platforms, but the picker's whole delivery mechanism
([[ft003]] keybinds, quickshell) does not exist on Termux — that is a design
problem, not a backend port, and belongs to a follow-up story.

**Rollback: `copyq/` stays in-tree for one cycle**, its rotz links removed but
files kept, deleted by a follow-up once the swap proves out in daily use. The
system shipped two days before this spec and its end-to-end path is still
unverified by hand; a flippable rotz dot is cheaper insurance than a revert.
`copyq/test-copyq.sh` also remains the source to adapt for the disk-leak
assertion.

**Trade-offs.** Memory-only degrades from a daemon property to **config
discipline** — under copyq no config edit could persist history to disk; under
clipcat a wrong `history_file_path` silently can. The spec must carry a test
asserting nothing reaches real disk (adapting the existing one), because that
invariant no longer defends itself. Listing costs N+1 calls instead of one.
And this buys **no security improvement** on its own: clipcat's filter is as
inert against gopass as copyq's, so `dotfiles-cyg` / [[sp015]] remain the
actual fix.

Against those: no Qt6/KDE chain on a daemon holding everything the user
copies — `qtkeychain` and `qca` leave the process entirely — RSS 63→12.3 MiB,
private 16→9.9 MiB, 96→4 linked libs, and a secret gate that never even
requests the payload from a hint-bearing owner.

**No ADR is overturned.** [[sp014]] deliberately kept the backend choice out of
ADRs, in `## solution` only. That now looks like the wrong call — a vendor
decision revisited within days is exactly what an ADR records — so this spec's
retro should mint one covering both the clipcat choice and what copyq cost.

## plan

**File tree**

```
clipcat/dot.yaml                   NEW  rotz install clipcat + link config
clipcat/clipcat.toml               NEW  daemon config: sensitive atoms, tmpfs history
clipcat/test-clipcat.sh            NEW  backend suite (adapted from copyq/test-copyq.sh)
quickshell/qs-clip.sh              EDIT list/set move row-keyed -> id-keyed
quickshell/test-clip-history.sh    EDIT id-keyed fixtures
i3/scripts/clip-set.sh             EDIT `copyq read <row>` -> `clipcatctl get <id>`
i3/scripts/test-clip-set.sh        EDIT id-keyed seeding
i3/scripts/clip-feed.sh            EDIT `copyq add -` -> `clipcatctl insert`
i3/scripts/test-clip-feed.sh       EDIT backend swap; BOTH secret gates stay
i3/config.d/native.conf            EDIT two clipcatd (one per display) + feeder
i3/config.d/wsl.conf               EDIT one clipcatd, no feeder
i3/scripts/test-clip-integration.sh EDIT autostart assertions
copyq/dot.yaml                     EDIT links removed; files kept for rollback
```

**The id contract** (fixed here so tasks 2-4 can proceed in parallel)

`qs-clip.sh` keeps its two-verb shape; only the key changes:

```
qs-clip.sh list        ->  "<hex-id>\t<preview>" lines, newest first
qs-clip.sh set <id>    ->  publish entry <id>; exit 0/1/2 unchanged
clip-set.sh <id>       ->  same, one layer down
```

Ids are clipcat content hashes — opaque to every consumer, and **stable**
under concurrent capture, unlike the positional rows they replace. No consumer
may parse, order, or arithmetic on an id.

**Conventions**

- Glue stays `#!/bin/sh` + `set -u`, every `xclip` call `timeout`-wrapped
  ([[adr0002]]; the `clip-sync.sh` mould). `clipcatctl` calls are
  `timeout`-wrapped on the same grounds — a wedged daemon must not hang a
  picker keypress.
- clipcat config version-controlled and rotz-linked, mirroring what
  `copyq/dot.yaml` does today; its header carries the client contract and the
  memory-only rationale, since `.toml` comments survive (unlike copyq's
  QSettings inis, which mangled them).
- One `clipcatd` per display, each with its own `local_socket`; clients reach
  the session daemon by the socket path, never by a guessed `DISPLAY`.
- Autostarts live in the [[ft003]] per-platform overlays; the keybind stays in
  the base config, untouched by this spec.

**Anti-patterns (hard rejects in review)**

- **Do NOT drop `clip-feed.sh`'s secret gates.** `clipcatctl insert` does not
  re-trigger the watcher ([[poc010]] Q3), so inserted items bypass
  `sensitive_mime_types` exactly as `copyq add` bypassed copyq's automatic
  commands. Both the TARGETS pre-check and the post-read re-check stay.
- **No `history_file_path` outside `$XDG_RUNTIME_DIR`.** Clips are written
  continuously while the session runs, not at exit — a path on persistent
  disk puts every copied secret on the filesystem for the session's lifetime.
- **No positional assumptions about ids.** Ids are content hashes; treating
  one as a row index reintroduces the shift-under-the-reader bug class that
  broke two [[sp014]] suites.
- Never trust inherited `DISPLAY` — derive it, as the shipped scripts do.
- No `clipcatctl` call without `timeout`.

**Known limitations**

- Memory-only is **config discipline**, not a daemon property. A wrong
  `history_file_path` silently persists secrets; Task 1's disk-leak assertion
  is the only thing defending the invariant.
- A plaintext clips file exists on tmpfs for the whole session and survives
  SIGKILL; only logind teardown at last logout clears it. Both designs remain
  swap-exposed.
- Listing costs N+1 `clipcatctl` calls (one `list`, one `get` per entry needing
  full text). Acceptable at the 200-entry cap; revisit if the cap rises.
- This spec ships **no security improvement**. The filter stays inert against
  gopass — `dotfiles-cyg` / [[sp015]] are the fix.

## tasks

### Task 1: clipcat backend — install, config, memory-only, secret atoms

#### type
task

#### effort
6h

#### depends
none

#### files_touched
`clipcat/dot.yaml`, `clipcat/clipcat.toml`, `clipcat/test-clipcat.sh`

#### success_criteria
- `rotz install clipcat` installs clipcat and links the config; `clipcatd`
  starts against the current display and `clipcatctl length` answers.
- A plain external copy is captured: `clipcatctl length` increments and
  `get <id>` returns the exact bytes.
- A copy advertising **either** `x-kde-passwordManagerHint` **or**
  `application/x-kde-passwordManagerHint` is dropped — length unchanged — and
  the daemon requests only `TARGETS`, never the payload.
- With `history_file_path` under `$XDG_RUNTIME_DIR`, no clipboard content
  appears anywhere on persistent disk: a canary copy is not found by grepping
  `~/.cache/clipcat`, `~/.local/share/clipcat`, `~/.config/clipcat` or the
  isolated tree; `clear_history_on_start` yields `length == 0` after restart.

#### edge_cases
Empty selection; multi-MB entry; image/non-text owner (skipped, not crashed);
two rapid copies both recorded in order; daemon restarted while a client is
mid-call; `$XDG_RUNTIME_DIR` unset (must fail loudly, never silently fall back
to a persistent path).

#### test_plan
`clipcat/test-clipcat.sh`, headless via Xvfb, adapting the selection-owner
fixture from `copyq/test-copyq.sh` (xclip cannot advertise multi-target
selections). Named scenarios: plain-capture-byte-exact; secret-dropped-bare-atom;
secret-dropped-prefixed-atom; payload-never-requested (instrument
SelectionRequests); no-content-on-persistent-disk; empty-history-after-restart.
Negative controls mandatory — with `sensitive_mime_types = []` the same
hint-bearing copy MUST be captured, or the drop assertions prove nothing.
Mutation-test: pointing `history_file_path` at a persistent path must fail the
disk assertion.

### Task 2: qs-clip.sh backend layer — id-keyed list and set

#### type
task

#### effort
6h

#### depends
Task 1

#### files_touched
`quickshell/qs-clip.sh`, `quickshell/test-clip-history.sh`

#### success_criteria
- `qs-clip.sh list` emits `"<hex-id>\t<preview>"` newest-first, one line per
  entry, count equal to `clipcatctl length` up to the configured cap.
- Previews are the first non-blank line, truncated to the configured width;
  the full entry is still published intact on `set`.
- `qs-clip.sh set <id>` publishes that entry and propagates `clip-set.sh`'s
  0/1/2 exit verbatim.
- Session derivation is unchanged in behavior: with two live sessions and no
  match it still refuses and names them rather than guessing.

#### edge_cases
Empty history (empty output, exit 0); entry whose text contains tabs or
newlines (preview must not break the line protocol); unicode previews; an id
that vanished between `list` and `set` (must exit 1, nothing published);
daemon not running; cap exceeded.

#### test_plan
Extend `quickshell/test-clip-history.sh`. Named scenarios: list-count-matches-
length; preview-is-first-nonblank-line; tab-in-entry-does-not-break-protocol;
set-publishes-the-selected-id-not-a-position; stale-id-exits-1-publishes-
nothing. The `set-publishes-the-selected-id` scenario is the anti-bug of this
task — filter to a single match that is NOT the newest entry and assert the id
passed through is that entry's. Mutation: making `set` publish the first id
instead of the selected one must fail it.

### Task 3: clip-set.sh — id-keyed lookup

#### type
task

#### effort
4h

#### depends
Task 1

#### files_touched
`i3/scripts/clip-set.sh`, `i3/scripts/test-clip-set.sh`

#### success_criteria
- `clip-set.sh <id>` fetches via `clipcatctl get <id>` and publishes to
  CLIPBOARD+PRIMARY on every live display, byte-exact.
- The 0/1/2 exit contract is unchanged and still verified: 1 only when no
  selection was written anywhere, 2 only on a partial write.
- All precondition checks still run before the first selection write.

#### edge_cases
Unknown/stale id (exit 1, zero selection writes); empty entry; multiline;
unicode; 1 MB entry through INCR; one display absent; no display at all;
`clipcatctl` missing from PATH.

#### test_plan
Adapt `i3/scripts/test-clip-set.sh`, keeping its two-display Xvfb harness and
its four untouched-sentinel assertions on the guard paths. Named scenarios
carried over: both-displays-byte-exact; survivor-display-still-succeeds;
guards-write-nothing. New: unknown-id-exits-1-writes-nothing. Mutation: break
the second-display write and the suite must fail (the assertion that was
missing in [[sp014]]'s first attempt).

### Task 4: clip-feed.sh — insert via clipcatctl, both gates intact

#### type
task

#### effort
5h

#### depends
Task 1

#### files_touched
`i3/scripts/clip-feed.sh`, `i3/scripts/test-clip-feed.sh`

#### success_criteria
- A `:10` CLIPBOARD change appears in the `:0` clipcat history exactly once
  within 1s.
- The TARGETS pre-check and the post-read re-check both still drop a
  hint-bearing copy — neither is removed nor made conditional on the backend.
- Feeder survives `:10` absent or torn down: no crash, no busy-loop, resumes;
  repeat copies still deduped; `flock` single-instance still holds across the
  i3-reload restart path.

#### edge_cases
`:10` down at start; hung selection owner (timeout, not block); repeat copy;
image/binary selection; double-start; daemon unreachable when a feed fires
(must not lose the single-instance lock).

#### test_plan
Adapt `i3/scripts/test-clip-feed.sh` — it already carries the deterministic
TOCTOU race fixture (`race-owner.py`) and the negative control that strips a
gate to prove it load-bearing. **Both must survive the port and still pass**:
`secret-not-laundered` (payload never requested) and `toctou-race` (handoff
fires, secret absent from history). Mutation: strip the re-check and the raced
secret must appear in history; strip the pre-check and the payload must be
requested. Scoping rule from `dotfiles-92w.5`: process helpers match the
test's absolute path, never a bare basename.

### Task 5: autostarts, integration, copyq decommission

#### type
task

#### effort
5h

#### depends
Task 1, Task 2, Task 3, Task 4

#### files_touched
`i3/config.d/native.conf`, `i3/config.d/wsl.conf`,
`i3/scripts/test-clip-integration.sh`, `copyq/dot.yaml`

#### success_criteria
- `native.conf` autostarts one `clipcatd` per display on distinct
  `local_socket`s plus `clip-feed.sh`; `wsl.conf` autostarts one daemon and no
  feeder.
- Autostarts are idempotent across `exec_always` reloads: exactly one daemon
  per display after a simulated reload, and the feeder's `flock` is not
  defeated.
- `copyq/dot.yaml` no longer links anything; `copyq/` files remain in-tree and
  no copyq process is started by any config.
- The base `i3/config` keybind is unchanged — no diff to the `$mod+Shift+v`
  line.

#### edge_cases
Daemon already running on reload; a display absent at autostart time; both
displays present; stale socket file from a killed daemon; rollback path
(re-enabling `copyq/dot.yaml` links restores the old backend).

#### test_plan
Adapt `i3/scripts/test-clip-integration.sh`, preserving its non-grep
discipline: strip comments before matching, assert every extraction's match
count, answer "is it bound" through `i3 -C` rather than text search, and prove
idempotency by *executing* the extracted `exec_always` strings and counting
processes from `/proc`. Named scenarios: one-daemon-per-display-after-reload;
wsl-has-no-feeder; copyq-not-started-by-any-config; keybind-line-unchanged.
Mutation anchors must be proven to land by byte comparison, not line delta
alone (the false-survivor trap from `dotfiles-92w.4`).
