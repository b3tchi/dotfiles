# Spec [[cat001]] [[cat002]] [[board]]

## extends [[ft007]]

## problem

[[ft007]] (shared-clipboard-history, shipped by [[sp014]]) runs on copyq
16.0.0. copyq is a Qt6 GUI application pressed into service as a headless
daemon: it drags in `qt6-svg`, `knotifications`, `kstatusnotifieritem`,
`kguiaddons`, `qca-qt6` and `qtkeychain-qt6`, links 96 libraries, and holds
everything the user copies. Measured on the native desktop: RSS 63 MiB,
**private 16 MiB**, 7 threads, 0 idle CPU.

The runtime cost is mild — 16 MiB private and zero idle CPU are noise on a
desktop, and **this is not a performance problem**. Two other costs are real:

1. **Dependency surface on a secrets-adjacent daemon.** `qtkeychain`, `qca`, a
   scripting engine, D-Bus and notification integration are attack surface,
   and ft007 uses roughly none of it — the shipped consumer calls only
   `read`, `add` and `size`.
2. **The Qt6 chain forecloses the small platforms.** [[ft007]] is scoped to
   graphical i3 sessions today, so Termux and proot never see it. That scope
   is a consequence of the backend, not a product decision — a clipboard
   history is exactly the sort of thing a phone session wants.

**[[poc010]] validated clipcat 0.25.0 (official `extra`, not AUR) as a
replacement**, and on the security property it is *stronger* than what
shipped: with `sensitive_mime_types` configured, clipcat requested only
`TARGETS` from a hint-bearing selection owner and never `UTF8_STRING` — the
payload never left the owning process. Both negative controls flipped, so the
fixture demonstrably delivered and the filter is demonstrably live. Footprint:
**RSS 12.3 MiB, private 9.9 MiB, 4 linked libs**, no Qt, no KDE, no GUI.

**What this does NOT fix.** `dotfiles-cyg` — the secret filter is inert
against gopass, which advertises no hint atom. clipcat's filter is inert for
exactly the same reason. Swapping backends buys a lighter daemon that stores
passwords in plaintext precisely as copyq does. [[sp015]] is the fix and is
backend-agnostic (it makes gopass *emit* the atom, which both backends drop
on), so the two are independent — but shipping this one alone must not be
mistaken for improving the security posture.

**The delta.**

*Memory-only changes mechanism.* copyq guarantees it as a daemon property
(per-tab `store_items=false`, nothing on disk ever). clipcat has no
equivalent: `history_file_path` is mandatory, is a *directory*, and clips are
written continuously during the session — not just at exit. `/dev/null` makes
the daemon refuse to start; `max_history = 0` captures nothing. The property
is reproducible by pointing `history_file_path` at `$XDG_RUNTIME_DIR` (tmpfs,
0700, destroyed by logind at last logout) plus `clear_history_on_start`, but
as **config discipline rather than a daemon guarantee**. That is a genuine
weakening: a future config edit can silently start persisting secrets to disk,
where under copyq no edit could. The spec must decide how that invariant is
defended (a test asserting nothing lands on real disk is the obvious answer,
and one already exists in `copyq/test-copyq.sh` to adapt).

*The read surface changes shape.* `clipcatctl list` emits
`<hex-id>: <100-char preview>` with no JSON, ids are content hashes rather
than positional indices, and full entry text needs `get <id>` per entry. So
[[ft007]]'s `## api_surface` moves from row-keyed to id-keyed, and listing
becomes N+1 calls. This is an **incompatible surface change**, but every
consumer is in-tree and changes with it (`ClipHistory.qml` via `qs-clip.sh`,
plus `clip-set.sh`), so the call is a **body refresh of [[ft007]], not a
supersession** — the capability, the keybind and the user-visible contract are
unchanged. Content-hash ids are arguably *better* than positional rows, which
shift under the reader; the row-shift hazard was a real bug source in
[[sp014]]'s test suites.

*The feeder's own gate stays mandatory.* [[poc010]] Q3 confirmed
`clipcatctl insert` does not re-trigger the watcher — which is convenient for
seeding, and means inserted items **bypass `sensitive_mime_types` exactly as
`copyq add` bypasses copyq's automatic commands.** So `clip-feed.sh` keeps
both its TARGETS gate and the [[sp015]]-era post-read re-check; nothing about
this swap makes the feeder's own filtering redundant.

*Multi-display needs orchestration.* One `clipcatd` binds one socket; the
native host runs `:0` and `:10` ([[adr0004]]). [[poc010]] ran two daemons on
separate `--server-endpoint` sockets with zero cross-talk, but which sessions
get a daemon — and whether the feeder still bridges `:10 → :0` or each display
simply keeps its own history — is an open design question this spec must
settle, not a mechanical port.

**Binding decisions.** [[adr0002]] (bash for tight glue) governs the rewritten
`qs-clip.sh` backend layer. [[adr0010]] (the picker publishes, never pastes) is
untouched — the swap is below that decision. No accepted ADR is overturned:
[[sp014]] deliberately declined to promote the backend choice to an ADR,
keeping it in `## solution`. **That decision now looks wrong** — a vendor
choice that gets revisited is exactly what an ADR is for, and the swap should
mint one at spec-retro so the next person sees why clipcat, and what copyq
cost.

**Open questions for `spec-writing`.**

- **Atom form.** Matching is exact with no prefix normalisation ([[poc010]]).
  clipcat's default is bare `x-kde-passwordManagerHint`; the shipped copyq rule
  uses prefixed `application/x-kde-passwordManagerHint`. Neither is verified
  against a real emitter — no password manager on this host emits either
  ([[sp015]]). Configure clipcat to match **both** forms unless there is a
  reason not to.
- **Rollback.** Keep `copyq/` in-tree behind an unused rotz dot for one cycle,
  or delete it in the same change? The swap touches every layer of a system
  that shipped two days earlier.
- **Termux/proot.** Does this spec actually extend [[ft007]] to those
  platforms, or only remove the blocker and leave the extension to a later
  story? Scoping both into one change risks a large, hard-to-review spec.

## solution

> **REVISED mid-execution** (2026-07-22). The original solution (clipcat
> 0.25.0 as backend, [[poc010]]) was falsified during execution by three
> upstream defects unresolvable at the `clipcatctl` surface:
> `dotfiles-8il` (list order non-deterministic — not newest-first, not even
> stable across calls), `dotfiles-i9i` (`get` irreversibly escapes embedded
> control chars — a literal `\n` substring and a real newline render
> byte-identical), and `dotfiles-apl` (~25% of daemon starts go permanently
> deaf on an unguarded `GetProperty` → `BadAtom`). An alternative backend,
> clipse, was evaluated and rejected ([[poc011]]: kills every process named
> `st` on listener start via a backwards containment check; persistent-disk
> image leak). The revised backend below is validated by [[poc012]].
> Tasks 1 and 4 landed against the original solution; their gates, fixtures
> and config carry forward — clipcat itself is demoted to the rollback layer
> beside copyq.

Replace copyq with a **bespoke clipnotify file-store** as [[ft007]]'s backend
on the platforms that run it today — native `:0`, xrdp `:10`, WSL. [[ft007]]'s
user-visible contract is unchanged: same `$mod+Shift+v` keybind via [[ft003]],
same picker, same publish-don't-paste behaviour per [[adr0010]]. What changes
is everything below `qs-clip.sh`. Validated by [[poc012]]: a
`clipnotify`-blocked loop captured 5 copies at 50 ms intervals with zero
drops and stored a real-newline / real-tab / literal-backslash-n entry
byte-exact.

**Why bespoke wins here.** The requirements are narrower than any
general-purpose manager: text-only capture, memory-only, single user, and the
secret gates are already ours (`clip-feed.sh`, adversarially tested in Task
4). The hard clipboard problems are delegated — `clipnotify` (official
`extra` package) handles polling-free event watching via XFixes, `xclip`
handles INCR and target negotiation. What remains is ~150 lines of `#!/bin/sh`
glue, which fits [[adr0002]] *better* than the clipcat CLI did and strictly
better than the gRPC client that fixing clipcat's read surface would have
required. Three upstream defects across two evaluated backends is the
argument for owning those lines: general-purpose is where their bugs live.

**The store, by construction, dissolves every open blocker:**

| blocker | resolution |
|---|---|
| `dotfiles-8il` ordering | monotonic seq-numbered filenames — `ls` is newest-last, deterministically |
| `dotfiles-i9i` byte-exactness | `get` is `cat` of a file — raw bytes by definition |
| `dotfiles-apl` deaf daemon | no daemon; `clipnotify` blocks on X-delivered events, and a dead loop is restartable/observable, not silently deaf |
| `dotfiles-t74` live-socket test hazard | no socket; the store is a directory the harness points anywhere |

**Four composed parts:**

1. **Backend: `clip-store.sh`** — one store loop per display, blocking on
   `clipnotify -s clipboard` (no polling, no daemon). On each event it runs
   the **TARGETS secret gate first** — both atom forms (bare
   `x-kde-passwordManagerHint` and prefixed
   `application/x-kde-passwordManagerHint`), reusing the exact gate pattern
   Task 4 shipped and adversarially proved in `clip-feed.sh` — and only then
   reads the payload via `timeout`-wrapped `xclip -o` into
   `$XDG_RUNTIME_DIR/clip-store/<display>/NNNNNN.clip`. Memory-only is now a
   **filesystem-level property** (tmpfs, 0700, destroyed by logind at last
   logout) — stronger than clipcat's config discipline and equal to copyq's
   daemon guarantee. Consecutive identical entries dedup; cap enforced by
   pruning oldest files. The gate moving into the store loop means the drop
   decision is *our tested code* on every path, not daemon config.

2. **The `qs-clip.sh` backend layer, rewritten** ([[adr0002]]). `list` reads
   the store directory: seq-descending filename order **is** newest-first —
   deterministic, no N+1 calls, no daemon round-trip. Ids are filenames
   (`NNNNNN.clip`) — opaque to consumers, stable under concurrent capture.
   `ClipHistory.qml` keeps talking only to `qs-clip.sh`.

3. **Feeder + publisher.** `clip-feed.sh` keeps both secret gates (TARGETS
   pre-check + post-read re-check, Task 4) and writes the fed entry **as a
   store file** on the destination — recording history without touching the
   destination's X selections, preserving the no-clipboard-stealing property
   Task 4 discovered and enforced (`-k secondary` finding). `clip-set.sh`
   publishes `cat <store>/<id>` to CLIPBOARD+PRIMARY, byte-exact, 0/1/2 exit
   contract verbatim.

4. **Autostarts** in the [[ft003]] overlays: one `clip-store.sh` loop per
   display (`exec_always`, `flock` single-instance like the feeder),
   `clip-feed.sh` still bridging `:10 → :0`. `wsl.conf` single loop, no
   feeder. A dead loop is visible (`pgrep`) and restartable on i3 reload —
   unlike clipcat's silently-deaf daemon (`dotfiles-apl`).

**Scope: swap only.** Termux/proot extension is explicitly out, as before.

**Rollback: two layers stay in-tree for one cycle** — `copyq/` (links removed)
and now `clipcat/` (links removed, its `test-clipcat.sh` retained as the
reference for the disk-leak and secret-atom assertions). Both deleted by a
follow-up once the store proves out in daily use.

**Trade-offs.** We own the store loop's bugs forever — no upstream. Accepted:
three upstream defects across two evaluated backends (eight counting clipse's)
showed general-purpose managers are where the bugs live, and our surface is
~150 lines. The clipnotify→read window can in principle coalesce a
faster-than-read burst to its newest entry ([[poc012]] could not provoke a
drop at 50 ms spacing; X buffers events during the read). A plaintext store
exists on tmpfs for the session lifetime and survives SIGKILL — same exposure
as both prior backends. And this ships **no security improvement**: the gate
still only fires on hint atoms no local password manager emits —
`dotfiles-cyg` / [[sp015]] remain the actual fix.

Against those: no daemon at all on the secrets-adjacent path — RSS ~0 beyond
a blocked clipnotify (~1 MiB), libs = clipnotify's X11 pair, byte-exact and
ordering correct *by construction*, and the secret gate is the same tested
code on every capture path.

**ADR at retro** should now cover the full chain: copyq → clipcat (validated
then falsified in execution) → bespoke store, and why vendor choice on this
surface kept failing ([[poc010]], [[poc011]], [[poc012]]).

```
                                        (absorbs dotfiles-t74)

```

**The store contract** (fixed here so tasks proceed in parallel)

```
store dir     $XDG_RUNTIME_DIR/clip-store/<display>/     (0700, tmpfs)
entry         NNNNNN.clip — monotonic seq, raw bytes, no trailing-newline games
id            the filename; opaque to consumers, never parsed or ordered on
qs-clip.sh list      ->  "<id>\t<preview>" lines, newest first (seq desc)
qs-clip.sh set <id>  ->  publish entry <id>; exit 0/1/2 unchanged
clip-set.sh <id>     ->  same, one layer down
```

**Conventions**

- Glue stays `#!/bin/sh` + `set -u`, every `xclip` call `timeout`-wrapped
  ([[adr0002]]; the `clip-sync.sh` mould). `$XDG_RUNTIME_DIR` unset = fail
  loudly, never fall back to a persistent path.
- The store loop's secret gate is the **same code shape** as
  `clip-feed.sh`'s TARGETS pre-check — both atom forms, checked before any
  payload read. One gate pattern, every path.
- Writes are atomic: write to `.tmp` in the same dir, `mv` into place — a
  reader never sees a half-written entry.
- One loop per display, `flock` single-instance each; autostarts in the
  [[ft003]] overlays; keybind untouched.

**Anti-patterns (hard rejects in review)**

- **Do NOT read the payload before the TARGETS gate passes.** The gate's
  whole point is the payload never leaves a hint-bearing owner.
- **No store outside `$XDG_RUNTIME_DIR`.** A persistent path puts every
  copied secret on disk for the session's lifetime.
- **No consumer parses or arithmetics ids** beyond opaque equality.
- Never trust inherited `DISPLAY` — derive it, as the shipped scripts do.
- No unbounded `xclip` call — `timeout` everything.

**Known limitations**

- A plaintext store exists on tmpfs for the whole session and survives
  SIGKILL; only logind teardown clears it. Same exposure as both prior
  backends.
- A faster-than-read copy burst may coalesce to newest ([[poc012]]: not
  provokable at 50 ms; X buffers events during the read).
- We own the loop's bugs; no upstream. The suites are the warranty.
- This spec ships **no security improvement**. The gate stays inert against
  gopass — `dotfiles-cyg` / [[sp015]] are the fix.
