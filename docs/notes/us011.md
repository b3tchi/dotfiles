# Story [[cat002]] [[product]]

## role
[[pn001|developer]]

## want
image copies (screenshots, browser right-click copies) captured into the
shared clipboard history alongside text, with the picker showing a
thumbnail for image entries and republishing the image byte-exact on
select

## because
[[ft007]] is text-only by scope; image owners are deliberately skipped.
The [[sp016]] file-store backend makes images a clean extension for the
first time — an entry is just a file, so a `.png` entry beside the
`.clip` entries needs no daemon, no protocol change, and no new id
scheme, where the copyq/clipcat backends would have needed vendor
format negotiation. Deferred out of [[sp016]] scope by explicit user
direction (epic dotfiles-egm notes, 2026-07-22).

## acceptance_criteria
- an image-only CLIPBOARD owner (e.g. `image/png` target) is captured as
  a store entry instead of skipped; text capture behavior is unchanged
- the secret gate still runs first on every image path — TARGETS
  pre-check + TOCTOU re-check, both hint-atom spellings, payload never
  read on a gated copy
- image entries respect the store contract: opaque ids, atomic writes,
  dedup, cap prune, nothing outside `$XDG_RUNTIME_DIR`
- `qs-clip.sh list` distinguishes image entries; `ClipHistory.qml`
  renders a thumbnail for them; `set` republishes the original bytes
  (`cmp`-exact) with the correct target type
- multi-MB images neither wedge the loop (timeout holds) nor evict the
  whole text history (cap policy accounts for size or count sanely)
