# ADR [[cat001]] [[product]]

## title
rust for rust only bindings

## title
Rust is allowed for a compiled helper when its required library is Rust-only

## context
[[adr0003]] binds compiled helpers to Go (stdlib-first, single static
binary, the akm-graph / d2-router / preview-d lineage). sp013 needed a
webview client host for [[ft005]]: [[poc007]] benchmarked the available
engines and chose Wry (WebKitGTK) for lightest RAM and a preinstalled
engine — and the wry/tao binding exists only in Rust. QML/WebKit was
ruled out (QtWebKit removed in Qt 5.6, no Qt6 port; QtWebView on Linux
falls back to QtWebEngine/Chromium, the heaviest option), so no Go or
Qt path reached the chosen engine.

## decision
A compiled helper MAY be Rust instead of Go when **both** hold:

1. The capability is delivered by a specific library/engine binding that
   exists only in Rust (not merely "Rust is nicer") — e.g. wry/tao for
   WebKitGTK webview hosting.
2. The [[adr0003]] engine-not-interface stance still applies unchanged:
   the binary is an engine behind the mandatory nushell wrapper
   ([[adr0001]]); no consumer invokes it directly.

Go remains the default for compiled helpers. This ADR complements
[[adr0003]] — it does not supersede it; the adr0003 three-condition test
for *whether* a compiled helper is warranted still gates first, then
language follows the ecosystem.

## consequences
- First Rust crate in the repo (`preview-webview/`): adds a cargo
  toolchain guard to its rotz recipe (mirrors the go-guard pattern) and
  a `Cargo.lock` committed for reproducible builds.
- Precedent scoped narrowly: future Rust proposals must name the
  Rust-only binding they need, or default back to Go.
- Mixed-language repo cost accepted: one extra toolchain on dev
  machines that build the preview client (build is guarded — hosts
  without cargo skip it with an actionable message).
