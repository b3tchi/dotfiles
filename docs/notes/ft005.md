# Feature [[cat001]] [[cat002]] [[product]]

## providing
A localhost Go daemon (`preview-d`) plus its mandatory nushell wrapper
that renders any file on demand and drives a live floating webview which
follows the editor cursor. Two route modes compose: a stateless
`/file/<path>` that renders one file as-is (renderer chosen by the path's
type) and a stateful `/preview<N>` live window that hot-swaps which file
it shows as the cursor moves. Bidirectional — the editor pushes the
cursor's file in, and selecting a node/element in the webview pushes a
path back out to open in nvim. Renders code, markdown, images, video, and
STL, and embeds [[ft004]] (akm graph) as a cross-origin iframe and
[[ft002]] (d2 diagrams) through a same-origin frame-header-stripping
proxy ([[adr0006]]). Consumed by nvim and nnn.

Note: the node/element → nvim reverse channel ships on the shell side
(a `postMessage` listener → `POST /open`) against an *assumed* message
contract — [[ft002]] / [[ft004]] do not yet emit the selection event.
That page-side emit and the final contract are pending sp009 / sp010.

## api_surface
HTTP on `localhost:$PREVIEW_PORT` (default 4200):

- `GET /file/<path>` — render one file as-is; renderer selected by the
  path's extension/type: code (syntax HTML), markdown (HTML), image
  (thumbnail, full-res on `?full`), video (poster frame, `<video>` on
  `?live`), STL (orbit viewer, raw bytes on `?full`). An akm-zettel path
  resolves to a cross-origin `<iframe>` of [[ft004]]; a `.d2` path
  resolves to a `<iframe>` of the same-origin `/d2embed/` proxy of
  [[ft002]] ([[adr0006]]). Embedded backends are lazy-spawned if not
  already listening. A `?slot=N` on the request is threaded onto the akm
  iframe src so a selection there routes back to slot `N`'s nvim
  ([[sp009]]).
- `GET /d2embed/<path>` — same-origin reverse-proxy of the [[ft002]]
  d2 page for `<path>`, with frame-blocking headers stripped ([[adr0006]]);
  path-jailed like `/file/<path>`.
- `GET /preview<N>` — live webview window `N`: a shell that loads the
  current `/file/<path>` and hot-swaps it. The same `/preview<N>` path
  upgrades to a websocket for the per-window redraw push. Multiple `N` =
  multiple concurrent windows, each an isolated slot (path + client set).
- `POST /preview<N>` body `{path}` — set window `N`'s current path (the
  editor cursor signal). An absolute path under the root is canonicalized
  to root-relative at ingestion; an absolute path outside the root is a
  `400` (never broadcast as a guaranteed 404). Broadcast is conditional
  ([[sp011]]): an akm-zettel path is first forwarded server-to-server as
  `POST /api/highlight {path, slot}` to [[ft004]] (scoped-timeout client,
  the [[adr0007]] pattern); on an akm→akm transition with a successful
  forward the window is NOT redrawn — the running [[ft004]] embed
  highlights in place instead of reloading. Every other transition (and
  any forward failure — degrade to reload, never a dead preview)
  broadcasts the redraw as before.
- `POST /open` body `{path, slot?}` — open the path in the running editor
  via `nvim --server <addr> --remote` (the reverse channel); path-jailed +
  arg-vector exec. With `slot`, routes to that slot's registered parent
  nvim ([[sp009]]); without `slot`, uses the process-global `$NVIM`
  (back-compat). `424` when the target addr is unbound / `$NVIM` unset.
- `POST /register` body `{nvim, slot?}` — bind an nvim's `v:servername` to
  a window slot; omitting `slot` allocates a free one and returns
  `{slot: N}` (concurrent registers get distinct slots). This is how each
  nvim claims its own preview window + reverse-open target ([[sp009]] /
  [[adr0007]]).
- `GET /status`, `POST /stop` — health snapshot / graceful stop
  (ft002/ft004 parity).

Mandatory nushell wrapper ([[adr0003]]) at
[nushell/actions/preview](../../nushell/actions/preview), the user-facing
API: `preview start | stop | status | register [--slot N] | open <path> | send <path> [--window N] | window [N] [--close]`.
No consumer invokes the binary directly. `window [N]` ([[sp013]]) spawns
the slot's native webview window via `preview-wv` — idempotent (per-window
pidfile, alive + `/proc/<pid>/cmdline`-verified pid → no-op; stale or
recycled pid → cleaned and respawned, never signaled); `--close` SIGTERMs
the cmdline-verified pid only; `stop` closes all windows before the
daemon. `PreviewStart` (nvim) calls `register` with `v:servername` to
claim a slot, then spawns that slot's window and `send` targets it
([[sp009]] / [[sp013]]).

## data_model
Ephemeral, no persistence. In-memory session map: window slot `N` →
`{ current path, set of websocket clients, parent nvim address }`, swapped
atomically under a mutex (the [[ft004]] hub pattern). Slots are allocated
on `/register` (lowest free N, mutex-guarded so concurrent registers stay
distinct); the parent nvim address drives per-slot reverse-open ([[sp009]]). Video poster frames are cached
per-path under a mutex (ffmpeg-extracted, first-frame); the cache is
currently unbounded — eviction is deferred (bd `dotfiles-e6o`). Embedded
peer daemons are lazy-spawned once under a double-checked lock and
health-polled with a timeout. Config via env `$PREVIEW_PORT` (default
4200). Window-client lifecycle state is wrapper-owned:
`~/.cache/preview/wv-<N>.pid` + `wv-<N>.log` per window ([[sp013]]). All viewer libs (three.js / `<model-viewer>`, syntax assets) are
compiled into the binary via `go:embed` — offline, no CDN, no
target-side build ([[ft004]] precedent).

## sample
```nu
preview start                       # spawn daemon (daemon only — no window)
preview window 1                    # open native webview window 1 (preview-wv, idempotent)
preview send ~/notes/us006.md       # push a file to window 1 -> it redraws
preview open ~/notes/us006.md       # (reverse) open that path back in nvim
preview window 1 --close            # close window 1; `preview stop` closes all + daemon
```

nvim side (`CursorHold` autocmd calls the wrapper):

```lua
vim.api.nvim_create_autocmd("CursorHold", {
  callback = function() vim.system({ "preview", "send", vim.fn.expand("%:p") }) end,
})
```

## components
- [preview/](../../preview) — Go daemon (`preview-d`): routes, renderers, session hub
- [nushell/actions/preview](../../nushell/actions/preview) — wrapper (start/stop/status/open/send)
- [nvim/plugins/preview.lua](../../nvim/plugins/preview.lua) — nvim cursor-signal + reverse-open integration
- [preview-webview/](../../preview-webview) — webview client host
  (`preview-wv <N> [--port P]`, shipped [[sp013]]): thin Rust wry/tao app
  (system webkit2gtk-4.1), chosen for lightest RAM + tiny binary +
  preinstalled engine ([[poc007]] benchmark). First Rust helper in the
  repo — [[adr0008]] scopes the exception to Rust-only bindings; the
  [[adr0003]] engine-not-interface stance holds (wrapper-only entry).
  Chrome-less, WM_CLASS `preview-wv`, title `preview <N>`, forced
  `GDK_BACKEND=x11` (uniform class matching — runs via XWayland under
  sway). Tiles/docks into the WM layout by design (no floating rule).
  Inherits the caller's `DISPLAY`/`XAUTHORITY`, so windows mix across
  local + xrdp sessions (xrdp needs no software-GL env — libEGL DRI3
  warnings there are cosmetic); multi-window scales sub-linearly
  (~110 MiB/added window, shared engine `.so`). Known gap: WSL sway
  profile sets `xwayland disable`, which blocks the forced-X11 window
  there (bd `dotfiles-qdl`). Reserve the web engine
  for HTML/graph/STL routes; the cheap image/video/code renderers stay
  native where practical.
