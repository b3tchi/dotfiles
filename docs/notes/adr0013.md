# ADR [[cat002]] [[cat001]] [[product]]

## title
persistent age-capped notification store

## context
The notification history store ([[ft010]], [[sp019]]) needs a home.
The house stance for the nearest sibling — clipboard history — is
**memory-only as a filesystem property**: [[ft007]]'s store lives
under `$XDG_RUNTIME_DIR` (tmpfs, destroyed at last logout) and
[[adr0011]] made that a hard property, with every store script
refusing to start rather than fall back to persistent disk. The
sibling posture [[adr0010]] runs the same direction: minimize what a
convenience surface can leak.

Notifications are not clipboard content. The recall value is highest
*after* a session ends ("what was that code / that message from
yesterday?"), which tmpfs destroys by design; and the exposure is
different in kind — clipboard history routinely holds deliberately
copied secrets (passwords, tokens), while notification text holds what
applications chose to broadcast. It still can carry sensitive material
(2FA codes, private-message previews), so unlimited persistence is not
acceptable either.

## decision
The notification store is **persistent, on real disk, age-capped**:
file-per-entry under `${XDG_STATE_HOME:-$HOME/.local/state}/qs-notif/`
(mode 0700), with entries older than the retention cap —
`QS_NOTIF_MAX_AGE`, default 172800 s ≈ 2 days — pruned on every
write. This is a deliberate, recorded departure from the
[[ft007]]/[[adr0011]] memory-only stance, scoped to notifications
only: the clipboard store's memory-only property is untouched and
remains binding.

The bounded window is the trade: recall survives logout and reboot,
while the age cap keeps the plaintext exposure to a rolling ~2 days
rather than an unbounded archive. Retention is enforced in the store
script's write path (prune-on-write), not by an external timer that
can silently not run.

## consequences
Notification text — including any codes or message previews an app
puts in a notification — lands on persistent disk in plaintext for up
to the cap, survives SIGKILL and power loss, and is subject to disk
forensics in a way the tmpfs clipboard store is not. Accepted
knowingly, for the bounded window only.

Prune-on-write means a store that stops receiving notifications stops
pruning: the last ≤2 days of entries before the daemon went quiet
persist until the next write. Accepted — the residue is small and
bounded by the cap at the moment writing stopped.

[[adr0011]]'s memory-only property is NOT weakened for the clipboard:
any future attempt to persist clipboard history must supersede
[[adr0011]] explicitly and cannot cite this ADR as precedent — the
justification here (broadcast content, post-session recall value,
short cap) is specific to notifications. Raising the cap
substantially, or making it unbounded, reopens this decision.
