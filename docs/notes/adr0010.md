# ADR [[cat001]] [[cat002]] [[product]]

## title
the clipboard picker publishes to the clipboard, it never synthesizes a paste

## context
[[sp014]] originally specified paste-under-cursor: on select, set the
selection, detect the focused window's class with `xdotool`, then synthesize
`ctrl+shift+v` for terminals and `ctrl+v` for GUI apps. That shipped as
`clip-paste.sh` and passed 39 assertions before review rejected it.

Two things surfaced during execution. First, the input path is where the
bugs live: the script had to identify *which* X display the user was
actually looking at, and its socket-glob probe returned displays in
lexicographic order, so `:0` always beat `:10`. On this host both sessions
are permanently live ([[adr0004]]), so a user working over xrdp would have
had `ctrl+v` fired into whatever Firefox on the unattended native session
happened to have focused — with clipboard-history entries that may be
passwords. The suite could not have caught it: it pinned the display on
every invocation, leaving the branch uncovered.

Second, the correctness surface is open-ended in a way the copy path is not.
Terminal-vs-GUI paste bindings are a maintained list that is wrong for every
app not on it; a modifier still physically held during the synthetic
keystroke corrupts it; and the "did it land at the caret?" property cannot be
verified headlessly at all — the audit accepted it on design argument alone.
Dvorak safety was likewise unobservable under Xvfb.

The user, seeing the rejection, chose to drop the paste half entirely rather
than fix the display derivation.

## decision
The picker's job ends at the clipboard. On select it publishes the entry to
CLIPBOARD and PRIMARY on **every** live display and closes; the user pastes
with their own paste key. No window-class detection, no keystroke synthesis,
no `xdotool` in the path. Writing to all displays rather than choosing one
eliminates the wrong-session bug class by construction rather than patching
it — enumeration order stops being a decision anyone can get wrong.

The published entry is left on the clipboard afterwards; a pick is a copy.

## consequences
The entire input path disappears, and with it the class of failure that gets
a clipboard tool blamed for typing a password into a chat window. What
remains is verifiable: an X selection either holds the exact bytes or it does
not, and `clip-set.sh`'s suite asserts that byte-exactly, including multiline,
unicode and a 1 MB entry through INCR. Layout (dvorak), keymap and app-specific
paste bindings become irrelevant — nothing is ever retyped.

The cost is one extra keypress per pick, every time. That is the whole
trade: a keystroke of friction bought the removal of a silent-wrong-window
failure mode.

Publishing to *every* live display is deliberate but not free — the picked
entry lands on the clipboard of sessions the user is not attending. The user
accepted this over publishing only to the picker's own display, on the
grounds that the history is already secret-filtered upstream. That reasoning
holds only as far as the filter does: see `dotfiles-cyg` and [[sp015]], where
a password manager that publishes no hint atom defeats it entirely.

This also forecloses a future "paste directly" convenience without
re-litigating the decision — reintroducing it means reintroducing display
derivation, and that is exactly what was removed.
