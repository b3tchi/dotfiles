# Story [[cat002]] [[product]]

## role
[[pn001|developer]] on a graphical i3 session (native :0, and/or an xrdp
:10 session reached from an Android RDP client)

## want
a shared clipboard *history* — every text copy is retained and any past
entry can be recalled from a picker and re-published to the clipboard

## because
each X display owns an independent CLIPBOARD holding only the latest copy;
a copy is lost the moment the next one replaces it, and nothing is shared
between a display and its siblings. Backed by [[ft007]] (shipped by
[[sp014]], backend re-based to the clipnotify file store by [[sp016]]).

## acceptance_criteria

SHIPPED (met):
- any recent *text* copy is recallable from a picker opened by a keybind
  (mod+v), on every graphical session; a chosen entry is re-published to
  CLIPBOARD+PRIMARY of every live display
- the picker PUBLISHES the selection, never auto-pastes ([[adr0010]])
- password-manager copies are never captured into history — the hint-atom
  secret gate fires on every capture and feed path (the [[sp016]] + wtr
  gates), independent of backend
- history is MEMORY-ONLY: nothing reaches persistent disk, and a session
  teardown (logind) clears that session's store
- text-only today: images / rich clipboard are not captured (see the image
  target below)
- the history merges one-way :10 -> :0: copies on the xrdp :10 session
  appear in the native :0 picker (:0 is the merged history), verified live
  from an Android RDP client 2026-07-22
- the picker is visually consistent with the launcher/projects overlays —
  480px floating dialog, shared chrome, no divergent focus border
  (was [[us015]], landed)

TARGET (not yet shipped — user-observable behavior each drives a future spec):
- BIDIRECTIONAL merge — a :0 copy also appears in the :10 picker; each entry
  appears exactly once (no bridge round-trip duplicate); both secret gates
  still fire on every crossing; NO feedback loop (a file-write bridge must
  not re-trigger the destination capture loop — clip-feed already writes a
  store file, not an X selection); a defined cross-source merge order that
  does NOT reintroduce the [[dotfiles-8il]] non-determinism; the
  two-feeders vs shared-store choice recorded as an ADR (revisits
  [[adr0004]]); degrades cleanly when only one session is live (was us016)
- IMAGE support — image copies are captured as store entries and the picker
  shows thumbnails; store stays memory-only and byte-capped (tmpfs pressure);
  text entries unaffected (was us011)

Purely-technical follow-ups are NOT acceptance criteria of this story —
they carry no user-observable behavior and live as bd chores: the nushell
glue rewrite ([[adr0002]] boundary), meta-package backend install on fresh
machines, and deleting the copyq/clipcat rollback trees after one cycle.
