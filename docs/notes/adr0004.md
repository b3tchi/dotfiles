# ADR [[cat001]] [[product]]

## title
Spec zettels carry one or more category links in their H1

## context
Spec zettels (`sp###`) are board citizens with an H1 that links back to
the board. Two H1 shapes had drifted into the corpus: sp001 and sp002
carried categories (`# Spec [[cat001]] [[board]]`), while sp003 and sp004
were tagless (`# Sp [[board]]`) — which is also what the old `akm write
sp` stub emitted. Meanwhile the AKM spec schema in `docs/notes/akm.md`
and the `idea-brainstorming` mint step both described the categorized
form. The `sp` migration (sp004 type 3/6) had to pick one shape for
`akm sp write` to emit, so the drift had to be resolved.

Options surveyed:
- Tagless `# Sp [[board]]` — matched sp003/sp004 + the old stub, simplest,
  but loses category grouping and contradicts the documented schema.
- Categorized `# Spec [[cat###]]... [[board]]` — matched sp001/sp002 + the
  schema; lets the board group workstreams by the categories they touch.

A related question: how many categories? ADRs take exactly one (the
[[adr0003]]-style rule) because a decision that spans categories is
usually two decisions. A spec/workstream is different — it legitimately
cuts across several categories (a single deliverable can touch security,
data, and tooling at once), so forcing exactly one would be wrong.

## decision
Spec H1 is the categorized form `# Spec [[cat###]]... [[board]]`, carrying
**one or more** `[[cat###]]` links followed by `[[board]]`. `akm sp write
--category` takes a comma-separated list (`--category cat003,cat006`) and
validates each against an on-disk `cat###`; at least one is required. This
differs deliberately from the ADR rule of exactly one category — a
workstream may span categories where a decision should not. The tagless
`# Sp [[board]]` form is retired; sp003 and sp004 were re-H1'd to match.

## consequences
Positive: the board can group specs by the categories they touch; the H1
is consistent with the `docs/notes/akm.md` spec schema and the
`idea-brainstorming` mint step; `akm sp write` enforces it at mint time so
the drift cannot recur. Multiple-category support means cross-cutting
workstreams are represented honestly rather than squeezed into one bucket.

Negative / lock-in: every spec must now pick at least one category at mint
(small friction for a genuinely cross-cutting or exploratory spec, eased
by allowing several); the two pre-existing tagless specs (sp003, sp004)
required a one-time H1 rewrite; and the categories must exist before the
spec is minted (the `--category` validation refuses dangling `cat###`).
