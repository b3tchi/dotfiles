# ADR [[cat002]] [[cat001]] [[product]]

## title
notification server is a user-bus singleton daemon, bars are read-only
consumers

## context
`org.freedesktop.Notifications` is a **user session-bus singleton**:
one name, one owner, per user bus. On this host two graphical sessions
are permanently live — native `:0` and xrdp `:10` ([[adr0004]]) — and
both share `/run/user/1000/bus`. The quickshell bar hosted a
`NotificationServer` inside every per-display instance
(`quickshell/config/shell.qml`), so both bars requested the name, the
first owner won, and every notification rendered on whichever session
happened to start first while the other session's server sat
permanently dead. The failure is structural, not a race to be patched:
N per-session servers cannot share a singleton bus name.

The repo already proves the correct shape twice: [[ft007]]'s clipboard
store (one writer, per-display consumers) and the `qs-stats-daemon`
singleton in `qs-start.sh` (one machine-wide daemon writing an atomic
state file that every session's bar follows with `tail -F`, guarded by
flock + pgrep so concurrent session starts collapse to one owner).

A per-display axis was considered and rejected: `qs-session.sh`'s
`qs_kill_session` scoping correctly keeps two *bars* apart, but a bus
singleton is per-user, not per-display — scoping the server per
session is exactly the bug being fixed.

## decision
The notification server is a **single user-scoped daemon**
(`quickshell -p notif`) that owns `org.freedesktop.Notifications` for
all sessions; **no bar instance ever hosts a `NotificationServer` or
requests the bus name**. The daemon is launched from `qs-start.sh`
behind a user-level flock + pgrep guard (the `qs-stats-daemon` mould):
the first session up starts it, every later session's attempt no-ops
on the lock. The per-session kill sweep spares the daemon.

Bars are read-only consumers: the daemon broadcasts live state
(count / last text / seq / critical) through an atomic state file
under `$XDG_RUNTIME_DIR`, and every bar drives its ticker/bell from a
`tail -F` on that file — the same source model `Bar.qml` already uses
for qs-stats. Mutation flows one way: dismiss commands travel over a
FIFO to the daemon, which is the sole writer of both the state file
and the history store ([[ft010]]).

## consequences
The first-owner-wins collision is impossible by construction — there
is exactly one owner and it is nobody's bar. Every session renders
identical notification state because all sessions read one file, and
a bar restart ($mod+Shift+d reload, i3 restart) no longer churns bus
ownership.

The daemon is a new single point of failure: it inherits the spawning
session's X connection, so that session's death kills it and
notifications go unowned until any session's next `qs-start.sh` run —
accepted; the guard restores it on every bar start and the failure
window replaces a *permanent* wrong-owner state with a transient one.
Notifications sent while no daemon runs are lost (the bus does not
queue for absent names). Live tracked state (bell count) resets on
daemon restart; history persists in the store.

This forecloses per-bar notification servers and any per-display
scoping of the server. Reopening either means re-litigating the
singleton bus-name constraint recorded here. Popup rendering, if ever
wanted, must be built as another read-only consumer of the broadcast
— never as a second server.
