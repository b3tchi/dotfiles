# ADR [[cat001]] [[product]]

## title
When to use a compiled helper (Go) instead of nushell

## context
[[adr0001]] establishes nushell as the default scripting language for
all workflow-tooling above the bash boot chain. [[adr0002]] pins the
exact five conditions under which bash wins over nushell. Together they
cover the entire space of *short-lived* processes — interactive scripts,
data-pipeline modules, one-liner shims, POSIX bootstrap scripts.

Neither ADR addresses **long-running daemon processes** that must hold
open TCP listeners, proxy websocket connections, or serve concurrent
HTTP clients. A daemon is not a "script" in any sense either ADR
contemplates:

- It does not compose structured data pipelines (the [[adr0001]]
  nushell use case).
- It does not meet any of the five bash conditions in [[adr0002]]:
  it is not a bootstrap script, not a shell startup file, not a tmux
  wrapper, not a one-liner where startup cost dominates, and it carries
  no hard POSIX contract from an external consumer.
- Nushell's process model (blocking pipeline evaluation, no async
  server loop, no goroutine-equivalent concurrency primitive) makes it
  structurally unsuitable for long-lived network servers.

The immediate driver is [[sp005]] (d2-watch-router): a daemon
`d2-router-d` that must hold an HTTP listener on a fixed port, reverse-
proxy websocket connections from `d2 --watch` children, rewrite streamed
bodies inline, and reap idle children — all concurrently. A nushell
script cannot do this. Bash cannot do this cleanly either. The question
is: what compiled language, and under what rule?

Options surveyed:

- **Nushell** — structurally impossible for a long-running async server.
  Ruled out.
- **Bash** — possible with background jobs and `nc`/`socat`, but
  brittle for websocket proxying, body rewriting, and concurrent
  client tracking. None of the five adr0002 conditions apply.
  Ruled out.
- **Go** — statically compiled, trivial cross-arch builds, Go stdlib
  covers HTTP + websocket reverse-proxy (`net/http`,
  `httputil.ReverseProxy`), goroutines map cleanly to concurrent client
  handling, and zero runtime dependencies on the target machine (binary
  drops into `~/.local/bin/`). Already in the toolchain on Arch/WSL
  targets via rotz.
- **Python / Rust** — heavier or more complex build chains for this
  narrow use case. Rejected in favour of Go's simplicity for small
  daemons.

## decision
A compiled Go helper is allowed — and *required over bash/nushell* —
when **all three** of the following conditions hold:

1. **Long-running daemon** — the process is expected to run
   indefinitely (until explicit stop), holding one or more open network
   listeners.
2. **Websocket/stream proxying or hot-path latency** — the process must
   proxy websocket frames, proxy streaming HTTP bodies, or execute a
   request-handling path where interpreter startup or GIL-style
   serialisation would dominate measured latency.
3. **No nushell structural fit** — the work cannot be modelled as a
   structured pipeline that completes and exits.

When these conditions hold, use Go (stdlib + minimal well-known
libraries). When they do not all hold, the default ([[adr0001]],
[[adr0002]]) applies — nushell or bash as appropriate.

**Mandatory: nushell wrapper front-end.** Every compiled helper MUST
expose a nushell action wrapper (under `nushell/actions/`) as its
user-facing API, per [[adr0001]]. The Go binary is an engine, not an
interface. The wrapper owns start / stop / status / open / reload (or
equivalent verbs for the domain). No user or script invokes the binary
directly.

**Scope boundary — explicit.** This decision does NOT authorise
rewriting existing nushell scripts, modules, or actions in Go. It is
strictly limited to new daemons/proxies that cannot be modelled as
pipelines. Migrating `nushell/scripts/` or `nushell/actions/` to Go
is explicitly out of scope and would require a separate ADR superseding
[[adr0001]].

**Does not contradict [[adr0002]].** The five bash conditions in
[[adr0002]] are unaffected. A daemon does not satisfy any of those
conditions — it is a fourth category (daemon/proxy), not an exception
carved out of the five.

## consequences

Positive:
- Daemon/proxy use cases have a clear, auditable rule: all three
  conditions must hold, or the default applies.
- The nushell-wrapper mandate keeps orchestration unified — callers
  never need to know whether the back-end is nu or Go.
- Go's stdlib (`httputil.ReverseProxy`, `net/http`, `sync.Mutex`) is
  sufficient for the [[sp005]] daemon with zero additional runtime
  dependencies.
- Statically compiled binaries distribute cleanly via rotz install
  (`go build -o ~/.local/bin/`); no interpreter version pinning needed
  on the target.

Negative:
- Go toolchain becomes a build-time dependency on every target that
  runs a compiled helper. Rotz install must guard `command -v go`
  before the build step.
- Compiled binaries are architecture-specific. Cross-arch builds (e.g.
  aarch64 for Termux) require `GOARCH`/`GOOS` overrides or a pre-built
  release download — rotz install must handle this.
- A new language in the repo raises the review and debugging surface.
  Kept narrow by the three-condition gate and the mandatory nu wrapper.
- The scope boundary (no rewriting existing nu scripts) must be
  actively enforced in review — this ADR is the reference.
