# ADR [[cat001]] [[product]]

## title
Typed AKM zettels expose per-type CLI namespaces; file-I/O in the CLI, lifecycle in skills

## context
The AKM workspace is driven by an `akm` nushell CLI plus a set of
Claude skills (one per zettel type: us/pn/ft/im/adr/cat/sp). Originally
each typed skill hand-composed the full markdown body and saved it via
raw `$AKM_ROOT/docs/notes/<id>.md` writes, duplicating id allocation,
frontmatter shape, H1 categorization, footer, and staging policy across
seven skills. That boilerplate diverged subtly — a skill-creator eval on
adr-write caught an H1 mismatch nobody noticed in normal use — proving the
untouched skills carried their own quiet drift.

Options surveyed:
- Keep the flat `akm write <type> <name>` form + skills hand-compose raw
  markdown. Status quo; every skill re-implements the schema and they
  drift apart.
- Move file-I/O AND lifecycle (status flips, board moves, supersession)
  into the CLI. One surface, but couples the CLI to the bd/board state
  machine and bloats it; lifecycle is inherently skill/workflow-shaped.
- Per-type CLI namespaces for file-I/O only, lifecycle stays in skills.

## decision
Every typed AKM zettel exposes a per-type CLI namespace —
`akm <type> read|list|write` — and the file-I/O concerns (id allocation,
frontmatter, H1 categorization, footer, structural backlinks, staging on
the default branch) are owned by the CLI. The lifecycle verbs (status
flips, `board.md`/`archive.md` section moves, supersession chains) stay in
the owning skills. The adr namespace was the guinea pig (sp003); the
pattern then propagated to the remaining five types plus sp (sp004).

## consequences
Positive: each owning skill shrinks (it composes only the body and pipes
it via `--stdin`); there is one audit point for every zettel mutation;
the schema lives in exactly one place, so the cross-skill drift that
motivated this is structurally eliminated; shared helpers
(`compose_tagless_zettel`, `parse_and_validate_cats`,
`compose_h1_with_links`) keep the per-type writers consistent.

Negative / lock-in: two mint paths coexist during the transition — the
flat `akm write <type>` form is not yet retired, so both must be kept
working until a cleanup spec removes the flat form. Supersession is still
done raw at the skill layer (no `akm <type> supersede` verb yet). The CLI
is now a load-bearing dependency of all seven typed skills, so its
interface (the structured `Id:` stdout line, the `--stdin` contract, the
flag shapes) is itself a contract that is expensive to change.
