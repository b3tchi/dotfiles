# Story [[file-preview]] [[product]]

## role
[[pn001|developer]]

## want
when the preview window already shows the embedded akm graph and my
cursor moves to another zettel in nvim, the running graph should mark
that zettel's node with a visual highlight instead of reloading the
whole viewer

## because
[[ft005]] treats every cursor move as an iframe src swap, so
zettel→zettel navigation tears down and reboots the [[ft004]] viewer on
each step — losing camera position, layout, and any isolate/legend state,
and burning a full graph rebuild render for what is conceptually just a
"current node changed" signal. Keeping the viewer alive and pushing a
highlight makes sweeping across notes fluid and turns the graph into a
live minimap of where the cursor is.

## acceptance_criteria
- zettel→zettel cursor moves do NOT reload the embedded [[ft004]] viewer;
  the already-running instance receives the new current zettel and
  applies a visual highlight (distinct ring/color + label emphasis) to
  that node PLUS the click-isolate visual around it (self + direct
  neighbors kept, rest dimmed — revised post-ship 2026-07-15); no
  camera motion
- zettel→non-zettel moves keep today's behavior: the preview swaps to the
  file's normal renderer; the next zettel preview may reload the viewer
- the first zettel preview (viewer not yet loaded) loads the viewer as
  today and highlights the current node once ready
- a previewed zettel with no node in the graph (unlinked/new file) is a
  silent no-op — no error, previous highlight cleared
- the highlight signal is origin-guarded on the receiving side, matching
  the [[ft005]]/[[us009]] postMessage discipline
