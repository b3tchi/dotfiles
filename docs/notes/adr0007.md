# ADR [[cat001]] [[product]]

## title
preview reverse channel routes server-side daemon-to-daemon, not via browser postMessage relay

## context
[[sp008]] Task 10 shipped a browser `postMessage` relay as [[ft005]]'s
reverse channel (backend page → preview-shell listener → `POST /open`),
but the emit side was never built, the relay only works while the backend
is embedded in the preview shell, and it cannot identify *which* nvim owns
the window. [[sp009]] needed an akm node-select to open the zettel source
in the owning window's parent nvim, including the multi-window case.

## decision
The reverse channel (selecting a node/element → open its source in the
editor) is delivered **server-side, daemon-to-daemon**. The backend daemon
([[ft004]] akm-graph) resolves the selection to a file path from its own
graph and POSTs [[ft005]] preview-d `/open {path, slot}`; preview-d routes
to the parent nvim of that slot via `nvim --server <addr> --remote`. The
per-slot nvim address is registered by each nvim (`POST /register`). The
browser client only issues a same-origin `fetch` to its *own* daemon
carrying the node `id` (never a raw path). The [[sp008]] Task 10
`postMessage` listener is retained but superseded for the akm case.

## consequences
The backend page needs no cross-origin `postMessage` emit — only a
same-origin fetch; the daemon holds the trusted `id → path` map, so a
client cannot inject an arbitrary path (it sends `id`). Works standalone
and embedded; per-slot routing enables multiple windows each bound to
their own nvim. Cost: every backend daemon wanting reverse-open needs a
small `/api/open` + a peer HTTP client, and nvim must register its
`v:servername`. [[sp010]] (d2 reverse channel) will follow this same
pattern. The dormant `postMessage` listener stays until a same-shell case
justifies it. Implementation surface in [[ft004]] + [[ft005]]; shipped by
[[sp009]].
