# ADR [[cat002]] [[product]]

## title
preview-d embeds peer daemons via same-origin proxy or cross-origin iframe by frame-header

## context
[[ft005]]'s webview must embed peer daemons ([[ft004]] akm-graph on 4810,
[[ft002]] d2-router on 4800) as live iframes. A genuinely cross-origin
iframe gives preview-d zero visibility into the embedded response, so it
cannot strip a frame-blocking header the backend emits. [[poc006]] found
the two backends differ: akm-graph sends no frame-blocking headers, but
the d2 page can carry `X-Frame-Options` / CSP `frame-ancestors` that
would refuse embedding.

## decision
preview-d embeds a frame-clean backend (akm-graph) as a direct
cross-origin `<iframe>`, and a frame-blocking backend (d2-router) through
a same-origin reverse-proxy route (`/d2embed/`) that strips
`X-Frame-Options` and frame-blocking CSP before relaying the response.
The proxy path validates the requested file through the same
`resolveInRoot` path-jail as `/file/<path>` before any upstream fetch.

## consequences
Header-blocking backends can be embedded without modifying the backend,
but each such backend needs a dedicated proxy route in preview-d and
pays a proxy hop. Frame-clean backends stay zero-cost cross-origin. The
proxy is a new same-origin surface, so its input must stay path-jailed
(SSRF/traversal). Alternatives foreclosed: "always cross-origin" cannot
strip headers; "always proxy" taxes frame-clean backends and widens the
attack surface needlessly. Implementation surface in [[ft005]] (`preview/proxy.go`);
evidence in [[poc006]].
