# Spec [[cat001]] [[cat003]] [[board]]

## solves
[[us003|rotate service credentials without downtime]]

## implements
[[im002|vault-policy credential rotation for live services]]

## problem
The platform-engineer [[pn002]] needs to rotate service credentials without
downtime. Story [[us003]] asks for live rotation with a 5-minute overlap and
zero 5xx during the rotation window.

## solution
Adopt the vault-rotate-policy pattern via [[ft002|vault-secrets]]: writers
stage the new credential under a versioned alias using `vault.secret(name)`
read path and an internal `vault.rotate_secret(name)` helper from
[[im002]]. Readers continue calling `secret(name)` — they transparently get
either the new or the prior value during the 5-minute overlap. After the
window, the prior version expires.

Binds [[adr0001]] (services stay on ft001 basic-auth — no new auth path)
and [[adr0002]] (Postgres retention is unaffected). Categories [[cat001]]
security and [[cat003]] infrastructure both apply.


Implementation detail (added during spec-writing): we'll also call
`vault.set_default_ttl(seconds=300)` once at boot to standardize the
overlap window globally, and the rotation flow will emit structured
audit records via [[ft002]]'s audit-log channel so security team has a
trail of every rotation.

### Conventions

- All vault access goes through `acme.lib.vault` — no direct vault SDK calls
  inside services.
- Public read path remains `vault.secret(name)`. Internal rotate path is
  `vault.rotate_secret(name)`; not exported from `__init__`.
- Tests live under `tests/` and import via `src.lib.vault`.
- Logging: structured JSON to stdout. The audit-log channel for rotation
  events is **gated on a Feature extension** (see Task 1) — do **not**
  invent a side-channel inside `vault.py` to ship the audit hook ahead of
  the extension.

### Anti-patterns (auto-reject during review)

- Adding `vault.set_default_ttl(...)` call without [[ft002]] exposing it in
  `## api_surface`. Calling a private/undocumented helper = Feature
  contract violation.
- Emitting "audit" records directly to stdout in `vault.py` under the
  guise of "logging" — that's a hidden Feature extension. Either ft002
  provides the audit-log channel after `idea-extend`, or the spec drops
  the audit requirement.
- Hard-coding the 5-minute overlap inside `rotate_secret` instead of
  reading the ttl from ft002 once it's exposed.
- Letting a service fall back to direct vault SDK when `secret()` errors.
  Errors propagate; the synthetic check is what proves zero-5xx.

### Known limitations

- ft002 today (read at `aliases: vault-secrets`, `status: stable`) exposes
  only `secret(name)`. The spec's `## solution` references two surfaces
  (`set_default_ttl`, audit-log channel) that ft002 does **not** provide.
  **Tasks 2–4 cannot ship until ft002 is extended via `idea-extend`.**
- The 5-minute overlap is wall-clock, not per-request. Long-running
  requests started before the flip will continue using the old value
  until they finish — acceptable for AC `Old secret stays valid for 5
  minutes after rotation` but worth surfacing.

### Task 2: Implement `rotate_secret(name)` helper in `src/lib/vault.py`

#### type
feature

#### effort
4h

#### depends
- Task 1 (extension scoped; otherwise this task ships and silently
  violates ft002's surface)

#### files_touched
- `src/lib/vault.py`

#### success_criteria
- New function `rotate_secret(name: str) -> None` exists in
  `src/lib/vault.py`. Not exported from `__init__`.
- Calling `rotate_secret("foo")` stages a new versioned alias for `foo`
  in vault; the prior version stays readable for the overlap window.
- After the overlap window expires, `vault.secret("foo")` returns the new
  value and the prior version is no longer reachable.
- Maps to us003.AC `Rotation script can swap secrets while services
  run`.

#### edge_cases
- `rotate_secret` called for a name that doesn't exist yet → raise
  `KeyError`, do **not** stage a phantom alias.
- `rotate_secret` called twice in rapid succession (e.g. retry storm) →
  the second call extends the overlap window; concurrent stages must not
  race (use vault's CAS or a lib-side lock keyed on `name`).
- Vault unreachable mid-rotation → the prior version must remain readable
  via `secret(name)`; partial rotation must roll back.

#### test_plan
- `tests/test_vault_rotation.py::test_rotate_creates_versioned_alias` —
  call `rotate_secret("svc/db_url")`, assert vault contains both
  versions, assert `secret("svc/db_url")` returns either value during
  the overlap.
- `tests/test_vault_rotation.py::test_rotate_unknown_name_raises` —
  `rotate_secret("does-not-exist")` raises `KeyError`. Catches a real
  bug (silent phantom alias) — not a tautology.
- `tests/test_vault_rotation.py::test_concurrent_rotate_serializes` —
  two threads call `rotate_secret("foo")` concurrently; vault ends up
  with exactly one new version, not two. Catches the race.
- `tests/test_vault_rotation.py::test_vault_outage_rolls_back` —
  simulate vault `ConnectionError` mid-stage; assert `secret("foo")`
  still returns the prior value.

### Task 4: Emit rotation audit records via ft002 audit-log channel

#### type
feature

#### effort
3h

#### depends
- Task 1 (audit-log channel must exist in ft002's `## api_surface`)
- Task 2 (rotate_secret is the emit site)

#### files_touched
- `src/lib/vault.py`

#### success_criteria
- Every successful call to `rotate_secret(name)` produces exactly one
  audit record with at least: `event="vault.rotate"`, `name`,
  `timestamp`, `actor` (if available from context), `outcome="ok"`.
- Failed rotations produce one audit record with `outcome="failed"` and
  the error class. Failures do **not** raise from the audit emit path
  itself.
- Maps to us003.AC `No 5xx during rotation window in synthetic check`
  (audit visibility is what proves the rotation didn't silently fail).

#### edge_cases
- Audit channel is itself unreachable → log a fallback warning to
  stdout, do **not** fail the rotation. The rotation outcome is the
  user-visible signal; audit is observation.
- Audit record at high rotation rate → ensure emit is non-blocking
  (async or fire-and-forget). Synchronous emit on a hot path
  re-introduces the 5xx risk.

#### test_plan
- `tests/test_vault_rotation.py::test_successful_rotation_emits_audit`
  — patch the audit channel, rotate, assert one record with
  `outcome="ok"` and the right fields.
- `tests/test_vault_rotation.py::test_failed_rotation_emits_audit_and_raises`
  — force vault failure, assert one record with `outcome="failed"` AND
  the rotation raised to the caller.
- `tests/test_vault_rotation.py::test_audit_channel_outage_does_not_break_rotation`
  — patch the audit channel to raise on emit, assert `rotate_secret`
  still completes and the synthetic-check semantics hold.
