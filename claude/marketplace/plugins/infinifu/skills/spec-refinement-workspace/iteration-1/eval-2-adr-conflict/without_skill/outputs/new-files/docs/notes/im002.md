# Implementation [[cat001]] [[cat003]]

## solves
[[us003|rotate service credentials without downtime]]

## approach
Rotate service credentials in place using [[ft002|vault-secrets]] with
a dual-secret overlap window. Vault issues `current` and `previous`
versions of each secret name (e.g. `reports/db_url`); writers always
publish the new version as `current` and demote the prior to
`previous`. Consumers accept either version for a 5-minute overlap,
then drop `previous`. No protocol change, no new CA, no impact on
[[ft001|basic-auth]] for user-facing endpoints — [[adr0001]] is
untouched. [[adr0003]] (smtplib direct) is unaffected.

## features
- [[ft002|vault-secrets]]

## data_model
No schema change. Vault holds the dual-version state; service-side
code keeps an in-process cache keyed by secret name with a 5-minute
TTL so overlap is enforced even if a service starts mid-rotation.

## api_surface
- `acme.lib.vault.secret(name)` — extended to return the active
  value; gains an internal `secret_versions(name)` that returns
  `(current, previous)` for handshake-style consumers (e.g. DB
  reconnect on auth failure → retry with `previous`).
- New CLI: `acme-rotate <name>` — issues a new version in Vault,
  records rotation timestamp, exits 0 on success.
- New synthetic check: `/healthz/rotation` — emits 200 only when
  no 5xx observed for the trailing 5 minutes spanning the rotation
  window.

## components
- `src/lib/vault.py` — add version-aware lookup + cache.
- `src/lib/rotation.py` — rotation CLI entrypoint.
- `src/services/reports/` — adopt the new lookup for DB URL.
- `src/services/auth/` — adopt the new lookup for its DB URL.
- `src/services/metrics/` — adopt the new lookup for its scrape token.
