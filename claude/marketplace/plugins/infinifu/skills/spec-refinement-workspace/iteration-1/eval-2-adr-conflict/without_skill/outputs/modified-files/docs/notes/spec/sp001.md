# Spec [[cat001]] [[cat003]] [[board]]

## solves
[[us003|rotate service credentials without downtime]]

## implements
[[im002|vault-policy credential rotation for live services]]

## problem
The platform-engineer [[pn002]] needs to rotate service credentials
without downtime. Story [[us003]] asks for live rotation with a
5-minute overlap and zero 5xx during the rotation window. Today
secret rotation is a maintenance-window task because every service
reads its secret once at startup and never re-reads.

## solution
Use [[ft002|vault-secrets]] with a dual-secret overlap window. Vault
keeps `current` and `previous` versions of each rotated secret;
writers publish the new value as `current` and demote the prior to
`previous`. Consumers accept either version for a 5-minute overlap,
then drop `previous`. Scope is deliberately the **secrets that
actually need rotation in production today** ‚Äî DB URLs and the
metrics scrape token. User-facing authentication ([[ft001]],
[[adr0001]]) is **out of scope** and untouched: that decision is
about service‚Üîuser auth, which is orthogonal to the
service‚Üîdependency credentials this spec rotates. No new CA, no
mTLS, no protocol change.

Binds [[cat001]] security and [[cat003]] infrastructure.

**Conventions.**

- All three services consume secrets through `acme.lib.vault.secret`
  ‚Äî no service reads Vault directly. The dual-version logic lives
  in one place.
- In-process secret cache TTL = 5 minutes; matches the overlap
  window so a service that boots mid-rotation cannot get stuck on
  `previous` longer than the contract allows.
- Rotation CLI is idempotent on the Vault side: re-running
  `acme-rotate reports/db_url` within the overlap window is a no-op
  (returns the already-staged `current`).
- Synthetic check polls `/healthz` across all three services every
  10s during a rotation and fails the test if any non-2xx appears
  within the 5-minute window.

**Anti-patterns to avoid.**

- Do **not** add a hot-reload signal handler; the 5-minute cache TTL
  is the rotation contract ‚Äî a signal path would let one service
  diverge from the contract.
- Do **not** thread the previous secret through every call site
  manually; the retry-with-previous logic stays inside `vault.py`
  via the `secret_versions(name)` helper, used only by clients
  that authenticate on connect (DB drivers).
- Do **not** widen scope to user-facing auth credentials
  ([[ft001]]) ‚Äî that would touch [[adr0001]] and is not what
  [[us003]] asks for. If user-auth rotation is needed, file a
  fresh story.
- Do **not** introduce a new CA or mTLS in this spec ‚Äî out of
  scope; if the platform later wants service-to-service identity
  beyond shared secrets, that is a separate Feature and ADR.

**Known limitations.**

- Only DB URLs and the metrics scrape token participate. Other
  secrets (e.g. SMTP creds ‚Äî see [[adr0003]]) still rotate on the
  old maintenance-window path; that is acceptable because they are
  not on the hot path of any user-visible request today.
- Vault must support versioned KV (kv-v2) ‚Äî already deployed.
- A rotation that spans a service restart still requires that the
  service can fetch from Vault during boot; this spec does not add
  offline-bootstrap resilience.

#### success_criteria
- `secret(name)` returns the current version transparently.

#### edge_cases
- First read after rotation but before TTL expiry on another
  process ‚Äî returns the old `current` (which is now `previous`).

  `secret_versions`.
- Vault transient 5xx during read ‚Äî surface as
  `VaultUnavailable`, do not poison the cache with `None`.

### Task 2: Implement `acme-rotate` CLI

#### type
feature

#### effort
2h

#### depends
- Task 1

#### files_touched
- `src/lib/rotation.py`
- `tests/lib/test_rotation_cli.py`

#### success_criteria
- `acme-rotate <name> --value <v>` writes a new version to Vault,
  demotes the prior to `previous`, records a rotation timestamp,
  exits 0.
- Re-running within the 5-minute window is a no-op (exits 0,
  prints "already rotated at <ts>").
- Exits non-zero with a clear message on Vault failure.

#### edge_cases
- Secret does not exist yet ‚Äî rotation creates v1, no `previous`.
- Vault rejects write (policy) ‚Äî propagate error, do not partially
  rotate.

#### test_plan
- `test_rotation_cli.py::test_first_rotation_creates_v1`.
- `test_rotation_cli.py::test_second_rotation_demotes_to_previous`.
- `test_rotation_cli.py::test_idempotent_within_window`.
- `test_rotation_cli.py::test_vault_failure_nonzero_exit`.

#### success_criteria
- All three services route DB / token reads through
  `acme.lib.vault.secret`.
- DB drivers retry once with `secret_versions(...)[1]` on auth
  failure during the overlap window; no manual config reload.
- No service caches the secret independently of the lib.

#### edge_cases
- DB connection pool already has open connections holding the old
  secret ‚Äî they remain valid because Postgres holds the session;

  scrape, not per process start.

#### test_plan

  CLI, assert next request succeeds with new secret and old

  Vault + service stack.

#### edge_cases
- Service restarts mid-window ‚Äî the synthetic should still pass
  (acceptance criterion explicitly forbids 5xx, restarts are not
  in scope of this story).

  outage*; the latter fails the test loudly with a different
  message so we don't silently green a broken rotation.
