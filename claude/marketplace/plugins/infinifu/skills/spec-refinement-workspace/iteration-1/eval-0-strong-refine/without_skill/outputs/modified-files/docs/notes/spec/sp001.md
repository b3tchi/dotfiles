# Spec [[cat001]] [[cat003]] [[board]]

## solves
[[us003|rotate service credentials without downtime]]

## implements
[[im002|vault-policy credential rotation for live services]]

## problem
The platform-engineer [[pn002]] needs to rotate service credentials without
downtime. Story [[us003]] asks for live rotation with a 5-minute overlap and
zero 5xx during the rotation window.

## solution
Adopt the vault-rotate-policy pattern via [[ft002|vault-secrets]]: writers
stage the new credential under a versioned alias using `vault.secret(name)`
read path and an internal `vault.rotate_secret(name)` helper from
[[im002]]. Readers continue calling `secret(name)` — they transparently get
either the new or the prior value during the 5-minute overlap. After the
window, the prior version expires.

Binds [[adr0001]] (services stay on ft001 basic-auth — no new auth path)
and [[adr0002]] (Postgres retention is unaffected). Categories [[cat001]]
security and [[cat003]] infrastructure both apply.

## plan

### File tree

```
src/lib/
  vault.py          # extended: rotate_secret(), versioned read path
  vault_test.py     # new: unit tests
scripts/
  rotate_secret.py  # new: operator CLI wrapping vault.rotate_secret
tests/
  rotation_e2e.py   # new: end-to-end synthetic check (no 5xx under load)
```

### Conventions

- All vault access continues to go through `acme.lib.vault.secret(name)`.
  Services never read Vault directly; they never see version numbers.
- `rotate_secret(name, new_value)` is the only write/rotate entry point.
  CLI wraps it.
- Versioned alias pattern in Vault: `kv/<service>/<name>` metadata carries
  `current_version` and `previous_version`. `secret(name)` reads
  `current_version`; on a `current_version` lease miss it transparently
  falls back to `previous_version` for the overlap window.
- Overlap window is a single module-level constant
  `ROTATION_OVERLAP_SECONDS = 300` in `vault.py`. Not per-secret. Not
  exposed to callers.
- All new code lives in `src/lib/` (per repo convention: shared libs).
  No service-level changes; services keep calling `secret(name)` and
  must remain unaware rotation happened.

### Anti-patterns to avoid

- Do **not** add a `version` parameter to `secret(name)`. Callers stay
  version-blind so rotation is invisible.
- Do **not** cache secret values inside services beyond the request
  scope — a cached prior value past the 5-minute window will trigger
  5xx after expiry.
- Do **not** introduce a second auth path or new ADR conflicting with
  [[adr0001]] (basic-auth stays).
- Do **not** touch `report_runs` retention or schema — [[adr0002]] is
  out of scope.
- Do **not** rotate during a maintenance window in tests. The whole
  point is live rotation; the synthetic check must run against live
  traffic.
- Do **not** log secret values. Log the secret *name* and the new
  `rotated_at`, never the bytes.

### Known limitations

- 5-minute overlap is a hard floor. Services with request latency
  >5min (none today) would still 5xx; accepted, no such services in
  scope.
- Rotation script is single-tenant per invocation (one secret at a
  time). Bulk rotation can wrap it in a shell loop.
- Vault dev-mode tokens used in tests; production uses the existing
  AppRole flow already wired into `vault.py`.
- No metrics emitted yet for rotation events (counter / gauge).
  Out of scope; can land in a follow-up spec if [[pn002]] asks.

### Task 1: extend vault.py with versioned read path

#### type

#### effort
3h

#### depends
- — none

#### files_touched
- `src/lib/vault.py`
- `src/lib/vault_test.py`

#### success_criteria
- `secret(name)` returns the `current_version` value when present.
- `secret(name)` falls back to `previous_version` if the
  `current_version` fetch fails within the overlap window.
- After the overlap window expires, `previous_version` is no longer
  returned even if `current_version` fetch fails (raises instead).
- Existing callers compile unchanged — signature `secret(name) -> str`
  preserved.

#### edge_cases
- Vault unreachable entirely: raise, do not silently serve stale.
- `previous_version` absent (first-ever write): raise on
  `current_version` miss — nothing to fall back to.
- Clock skew between rotator and reader: window measured against
  Vault metadata `rotated_at`, not local wall-clock.

#### test_plan
- `test_secret_returns_current` — happy path, returns current value.
- `test_secret_falls_back_within_overlap` — current miss within 300s
  of rotation returns previous.
- `test_secret_raises_after_overlap` — current miss after 300s raises
  `VaultMissError`.
- `test_secret_raises_when_no_previous` — first-write scenario raises.
- `test_signature_unchanged` — callers using `secret(name)` keyword
  or positional still work.

### Task 2: implement rotate_secret() helper

#### type
feature

#### effort
3h

#### depends
- Task 1

#### files_touched
- `src/lib/vault.py`
- `src/lib/vault_test.py`

#### success_criteria
- `rotate_secret(name, new_value)` writes `new_value` as
  `current_version`, demotes prior `current_version` to
  `previous_version`, stamps `rotated_at = now()`.
- Idempotent on retry within the same overlap window (writing the
  same new value twice does not chain-demote a valid current to gone).
- Concurrent rotate-call protection via Vault CAS on metadata version.

#### edge_cases
- Two operators rotate at once: second call fails with
  `RotationConflict` (CAS miss); no partial state.
- New value identical to current: no-op, do not demote.
- Vault write fails after metadata update: rollback via CAS, raise.

#### test_plan
- `test_rotate_basic` — current/previous swap as expected.
- `test_rotate_idempotent` — same value twice, second is no-op.
- `test_rotate_concurrent_cas_fails` — simulated CAS conflict raises
  `RotationConflict`.
- `test_rotate_records_timestamp` — `rotated_at` stamped on metadata.
- `test_rotate_never_logs_value` — assert log output never contains
  the new value bytes.

#### success_criteria
- `python scripts/rotate_secret.py <name> --value <v>` rotates and
  prints confirmation + `rotated_at`.
- `--value -` reads from stdin so secrets never appear in shell
  history.
- Non-zero exit on `RotationConflict`, `VaultMissError`, or auth
  failure, with a clear stderr message.

#### edge_cases
- Missing `--value`: fail loudly; never rotate with empty string.
- Operator lacks Vault policy: surface 403 as a readable error,
  exit code 2.
- `--value` passed as plain arg: warn that history will capture it
  (but still proceed).

- `test_cli_propagates_conflict` — CAS conflict → exit 1 + stderr.

### Task 4: synthetic end-to-end no-5xx check

#### type
task

#### effort
4h

#### depends
- Task 3

#### files_touched
- `tests/rotation_e2e.py`

#### success_criteria
- Test boots a live service (reports) against dev-mode Vault.
- Runs sustained traffic (≥10 req/s) for the duration of a rotation
  triggered mid-flight.
- Asserts **zero 5xx** responses across the rotation window
  (us003 acceptance criterion).
- Asserts traffic served by `previous_version` for a non-empty
  portion of the window (proof the overlap actually engaged, not
  just that nothing changed yet).

#### edge_cases
- Rotation timed at request boundary: service must not 5xx if a
  request lands in the microseconds between demote and promote.
- Vault dev-mode token expiry mid-test: test fails loudly (real bug,
  not flake).

#### test_plan
- `test_rotation_under_load_no_5xx` — the core acceptance test.
- `test_overlap_actually_used` — assert the served-version histogram
  contains both `current` and `previous` during the overlap window.
