# Spec [[cat002]] [[cat001]] [[board]]

## solves
[[us004|quarterly retention purge tool with deletion proof]]

## problem

Legal asks the platform-engineer once a quarter for proof that the
90-day retention rule set in [[adr0002]] is actually being enforced
against the `report_runs` table. Today the nightly delete-job runs
in the background and the platform-engineer reconstructs evidence by
hand — there is no operator-facing tool, no signed artifact, and no
way to re-run the proof on demand.

The single consumer of this tool is [[pn002|platform-engineer]];
expected cadence is once a quarter. It is *not* a horizontal
capability that other services will reuse — it is a one-shot
operator workflow that wraps the existing retention rule.

Scope at idea stage:

- Initial table in scope: `report_runs` (the only retention-bound
  table today per [[adr0002]]).
- Out of scope: schedule changes, retention-window changes, adding
  other tables, multi-tenant proof signing.
- Open questions for the spec stage:
  - Where does the deletion-proof artifact live (file on disk,
    Postgres table, S3-equivalent)?
  - Does it need a cryptographic signature, or is "human-readable
    + checked into a git-tracked operator log" enough?
  - Is the existing nightly cron still authoritative, or does the
    tool *become* the authoritative purge with the cron removed?

Routing note: the seeded request named the *idea-feature* skill, but
the shape ("single consumer, once a quarter, no other implementation
will consume it") is a vertical user-story flow, not a horizontal
reusable capability. Captured here as a fresh `us004` + `sp001` at
`idea` stage so spec-writing can confirm or pivot before any `ft###`
is minted.
