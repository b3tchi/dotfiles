# Spec [[cat003]] [[cat004]] [[board]]

## problem

Three services (`auth`, `metrics`, `reports`) each carry an ad-hoc
email helper today — `auth` shells out to `mail`, `metrics` and
`reports` paste their own `smtplib` snippets. `metrics` now also
wants Slack alerts, and the metrics team is asking for pager-style
SMS for after-hours alerts. The current state is what `adr0003`
explicitly accepted at the time ("No external SMTP relay — services
use smtplib directly") and its consequences have caught up with us:
copy/paste drift across services, MTA-host changes require editing
three places, and there is no shared metric for send rate / bounces
/ latency. We have no shared channel layer at all — email is hard-
coded per service, Slack/SMS don't exist.

This spec proposes a **horizontal notifications capability** the
system will provide once and each service consumes. At
spec-writing time it will mint a new `ft###` under `[[cat003]]
infrastructure` + `[[cat004]] observability`. No `ft###` is minted
here — the capability boundary is still under discussion.

**Capability boundary (proposed).**

- A single `notify(channel, recipient, template, payload)` API
  surface consumed by every service.
- Channels at launch: `email`, `slack`. Channel `sms` is plausible
  later for after-hours metrics paging — the API and the channel-
  adapter shape must leave room for it without rework, but no SMS
  adapter ships now (YAGNI).
- Channel adapters live behind a single interface so adding `sms`
  (or `webhook`, `pagerduty`) later is a new adapter, not a new
  API.
- Owns its own send-log / retry / dedup state. Inherits
  `ft002` (vault-secrets) for SMTP creds, Slack webhook URLs, and
  any future SMS provider key.

**Plausible consumers (concretely surveyed).**

- `auth` service — welcome emails (today: shells out to `mail`,
  see `src/services/auth/__init__.py`).
- `metrics` service — alert paths (today: stdout + ad-hoc smtplib,
  see `src/services/metrics/__init__.py`). Wants Slack now,
  after-hours SMS later.
- `reports` service — report-ready CSV emails (today: smtplib,
  see `src/services/reports/__init__.py`). Backs `[[pn001]]`'s
  `primary_goal` "Get notified when long-running reports finish"
  and the `[[us001]]` reports-flow.

Three real consumers across two personas (`[[pn001]] analyst`,
`[[pn002]] platform-engineer`) — clears the "a feature with one
consumer is not a feature" bar.

**Ad-hoc implementations to migrate.**

No existing `im###` zettels capture the ad-hoc helpers (the only
implementation card today is `[[im001]]` and it consumes `[[ft001]]`
basic-auth, not any email path). Migration targets are the three
service files directly:

- `src/services/auth/__init__.py` — replace `mail` shell-out with
  `notify("email", ...)`. Low cost: single send site.
- `src/services/metrics/__init__.py` — replace stdout+smtplib with
  `notify("email", ...)` + add `notify("slack", ...)`. Medium
  cost: must keep stdout fallback for local dev.
- `src/services/reports/__init__.py` — replace smtplib with
  `notify("email", ...)`. Low cost: single send site backed by
  `report_runs` rows.

The retro will file these as a fresh `im###` per service (or one
combined card) once the feature ships.

**Binding ADRs.**

- `[[adr0003]]` ("No external SMTP relay — services use smtplib
  directly") is filed under `[[cat003]]` and **directly contradicts
  this proposal**. Spec-writing must produce a new ADR that
  supersedes `adr0003` before the `ft###` is minted. The
  consequences `adr0003` listed (copy/paste drift, MTA-host edits
  in three places, no send metrics) are exactly the costs this
  feature pays back.
- `[[adr0001]]` (under `[[cat001]]` security) — every service
  authenticates via `ft001`. The notifications feature is a
  shared library, not a network-callable service, so `ft001`
  doesn't apply to the feature itself — but spec-writing should
  confirm the trust model when the channel adapters fetch secrets.
- `[[adr0002]]` (under `[[cat002]]` data, 90-day retention on
  `report_runs`) — if the send log persists in Postgres, the same
  retention policy is the natural default. Decide at spec-writing.

**Constraints inherited from categories.**

- `[[cat003]] infrastructure` — cross-service plumbing; must be a
  library every service can import, not a separate service
  (consistent with how `ft001` and `ft002` ship today).
- `[[cat004]] observability` — alerting paths; emit Prometheus
  metrics for send count / failures / latency so `metrics` can
  alert on the feature itself.

**Intent.** At spec-writing time, mint `ft###` ("notifications")
with channel-adapter `api_surface`, file the superseding ADR for
`adr0003`, and list the three service migration touch-points in
the implementation card(s) that follow.
