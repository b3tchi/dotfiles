# Spec [[cat004]] [[board]]

## problem

We need a shared **metrics scraping** capability — a horizontal building
block that every service uses to expose counters, gauges, and histograms
in a Prometheus-compatible format and to scrape them on a shared
schedule. Today this is ad-hoc: only `src/services/metrics/` runs any
Prometheus calls, and there is no shared client library — every other
service emits no metrics at all.

**No existing `ft###` covers this.** Surveyed features: [[ft001]]
(basic-auth) and [[ft002]] (vault-secrets) — both [[cat001]] security,
unrelated to observability. We'd mint a new `ft###` at spec-writing time
once the api_surface is locked.

**Granularity.** This sp### scopes **one** ft### — the metrics-scraping
capability. The wider observability ask was decomposed into 5 atomic
features (metrics, logging, tracing, alerting, dashboards) so each can
ship independently with its own `## providing` / `## api_surface` /
lifecycle, per the AKM feature schema's singletons. The other four
features ride in [[sp002]] (logging), [[sp003]] (tracing), [[sp004]]
(alerting), [[sp005]] (dashboards).

**Consumers (via existing `im###`).** Surveyed [[im001]] (reports
dashboard) — no metrics today, candidate consumer. The three running
services (`auth`, `metrics`, `reports`) are migration targets:

- `src/services/metrics/` — already scrapes Prometheus ad-hoc; would
  migrate to providing instead of bespoke calls.
- `src/services/reports/` — currently emits no metrics; would consume
  the shared client to publish job durations and success counters.
- `src/services/auth/` — currently emits no metrics; would consume the
  shared client to publish login attempt counters.

**Category.** [[cat004]] (observability) is the primary; [[cat003]]
(infrastructure) covers cross-service scrape topology if the new feature
introduces a central scraper.

**Binding ADRs.** No accepted ADR under [[cat004]] yet; the new feature
will likely produce one (retention budget, label cardinality cap). No
conflict with [[adr0001]] (auth), [[adr0002]] (reports retention), or
[[adr0003]] (smtplib alerts — alerting is a different sp###).

**Migration intent.** At spec-writing time, mint `ft###` for the shared
metrics client; replace the ad-hoc Prometheus calls in
`src/services/metrics/` with the client; add metric publication to
`src/services/auth/` and `src/services/reports/` as a follow-up
implementation per service.
