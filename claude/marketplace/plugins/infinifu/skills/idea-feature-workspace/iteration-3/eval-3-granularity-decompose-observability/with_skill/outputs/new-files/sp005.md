# Spec [[cat004]] [[board]]

## problem

We need a shared **dashboards** capability — a horizontal building
block that renders the metrics, logs, and traces signals from
[[sp001]], [[sp002]], [[sp003]] into operator-facing visualizations
(panels, drill-downs, time-range controls, saved views). Today nobody
has a dashboard for platform health: the `auth`, `metrics`, and
`reports` services all run blind apart from `metrics`'s ad-hoc stdout
alert path.

**No existing `ft###` covers this.** Surveyed features: [[ft001]]
(basic-auth) and [[ft002]] (vault-secrets) — both [[cat001]],
unrelated. [[im001]] (reports dashboard) is an *analyst-facing* report
dashboard, **not** an observability dashboard — different
consumer/purpose, different data source — so it is **not** a dedup
match. We'd mint a new `ft###` at spec-writing time.

**Granularity.** This sp### scopes **one** ft### — the dashboard
provisioning + rendering capability for observability signals.
Companion observability features ride in [[sp001]] (metrics),
[[sp002]] (logging), [[sp003]] (tracing), [[sp004]] (alerting).
Dashboards are split out because the api_surface (declare panel, query
signal, render) and data_model (saved-view definitions, query
templates) are fundamentally different from a signal-producing feature.
[[im001]]'s analyst dashboard could later be re-skinned on top of this
feature, but that's a separate migration question.

**Consumers (via existing `im###`).** Surveyed [[im001]] (reports
dashboard) — partial precedent but *not* a current consumer of this
feature (it is its own bespoke HTML page). The real consumers will be
operations roles wanting a "platform health" view across all three
services. No `im###` exists yet that this feature serves directly; that
gap is real and expected for a brand-new horizontal capability — the
first consuming `im###` will be authored at spec-writing time alongside
the `ft###`.

**Category.** [[cat004]] (observability) is the primary; [[cat001]]
(security) is secondary because dashboard access must go through
`ft001` (only authenticated operators see internal panels).

**Binding ADRs.** [[adr0001]] (basic-auth) binds the auth gate on
dashboard access. No accepted ADR under [[cat004]] yet. The new feature
may produce an ADR choosing Grafana over a bespoke front-end.

**Migration intent.** At spec-writing time, mint `ft###` for the
dashboards feature; ship a starter "platform health" panel set covering
[[sp001]] metrics, [[sp002]] logs, [[sp003]] traces, [[sp004]] fired
alerts; consider whether [[im001]]'s analyst dashboard should later
migrate onto this feature (separate sp###).
