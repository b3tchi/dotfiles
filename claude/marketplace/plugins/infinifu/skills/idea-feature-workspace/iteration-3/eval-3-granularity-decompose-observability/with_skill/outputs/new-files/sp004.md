# Spec [[cat004]] [[board]]

## problem

We need a shared **alerting** capability — a horizontal building block
that evaluates rules against the metrics signal (from [[sp001]]) and
routes notifications to operators (email today, additional channels
later). Today the alert path is ad-hoc and broken: `metrics`
("alerts via stdout for now", `src/services/metrics/__init__.py`)
prints alerts to stdout where nobody sees them.

**No existing `ft###` covers this.** Surveyed features: [[ft001]]
(basic-auth) and [[ft002]] (vault-secrets) — both [[cat001]], no overlap
with alerting. We'd mint a new `ft###` at spec-writing time.

**Granularity.** This sp### scopes **one** ft### — the alerting engine
(rule evaluation + notification routing). Companion observability
features ride in [[sp001]] (metrics — the signal source), [[sp002]]
(logging — the audit trail for fired alerts), [[sp003]] (tracing),
[[sp005]] (dashboards — the silencing UI). Alerting is its own ft###
because the api_surface (declare rule, route notification) and
data_model (rules, silences, fired-alert history) are fundamentally
different from a metrics scraper or a log shipper.

**Consumers (via existing `im###`).** Surveyed [[im001]] (reports
dashboard) — not a direct consumer; alerting consumers are operations
roles, not analyst features. The migration target is
`src/services/metrics/` — its stdout alert prints become declared rules
in the new feature, with notification routing to the right channel.

**Category.** [[cat004]] (observability) is the primary; [[cat001]]
(security) is secondary because alert routing carries sensitive content
(who, when, what failed) and the notification channel must respect
auth/audit norms.

**Binding ADRs.** [[adr0003]] **constrains this feature directly**: "No
external SMTP relay — services use smtplib directly." The alerting
feature's notification path inherits that ADR — its email channel must
call `smtplib` against the internal MTA, not introduce a relay. Whether
[[adr0003]] should be revisited (its own `## consequences` list "no
metrics on send rate, bounces, or latency" as a known pain) is a
question for spec-writing, not this idea-stage zettel. [[adr0001]]
(basic-auth) is binding for the silencing/acknowledge UI — alert
management must go through `ft001`.

**Migration intent.** At spec-writing time, mint `ft###` for the
alerting engine; replace `src/services/metrics/`'s stdout alerts with
declared rules; route notifications via the smtplib path (per
`adr0003`) or a follow-up notification feature if one is later
proposed.
