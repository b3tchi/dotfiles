# Spec [[cat004]] [[board]]

## problem

We need a shared **distributed tracing** capability — a horizontal
building block that every service uses to emit spans (with a propagated
trace_id) and ship them to a central collector so cross-service request
flows are observable end-to-end. Today there is **zero tracing** across
the platform per the user's framing; no service starts a trace, no
service propagates a trace context across the wire.

**No existing `ft###` covers this.** Surveyed features: [[ft001]]
(basic-auth) and [[ft002]] (vault-secrets) — both [[cat001]] security,
unrelated. We'd mint a new `ft###` at spec-writing time.

**Granularity.** This sp### scopes **one** ft### — the tracing client
plus its trace-context propagation contract. Companion observability
features ride in [[sp001]] (metrics), [[sp002]] (logging), [[sp004]]
(alerting), [[sp005]] (dashboards). Tracing is split out because the
api_surface is fundamentally different (spans vs log records vs metric
samples) and the wire-level propagation header is a contract every
service must respect — pulling that into a unified "stack" would mask
the cross-service coupling the trace context actually introduces.

**Consumers (via existing `im###`).** Surveyed [[im001]] (reports
dashboard) — candidate consumer; the report-run path crosses services
(analyst hits reports → reports calls metrics for scrape data → reports
emails CSV) and is the obvious first trace target. All three services
(`auth`, `metrics`, `reports`) are migration targets — each needs the
tracing client wired into its request entry point and its outbound
calls.

**Category.** [[cat004]] (observability) is the primary; [[cat003]]
(infrastructure) covers the propagation-header contract and the central
collector deployment.

**Binding ADRs.** No accepted ADR under [[cat004]] or [[cat003]]
constrains tracing today. The new feature will probably produce an ADR
choosing OpenTelemetry over a bespoke wire format.

**Migration intent.** At spec-writing time, mint `ft###` for the tracing
client; wire it into all three services as a first migration; pick the
W3C `traceparent` header (or equivalent) as the propagation contract.
