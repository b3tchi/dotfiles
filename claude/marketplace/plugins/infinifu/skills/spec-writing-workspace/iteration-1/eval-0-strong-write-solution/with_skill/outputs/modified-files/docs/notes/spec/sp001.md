# Spec [[cat001]] [[cat003]] [[board]]

## solves
[[us003|rotate service credentials without downtime]]

## problem
The platform-engineer [[pn002|platform-engineer]] needs to rotate service
credentials without downtime. Today this is a maintenance-window task; the
story [[us003]] asks for live rotation with a 5-minute overlap and zero 5xx.

The solution will inherit from [[adr0001]] (basic-auth via [[ft001]]) and
[[adr0002]] (Postgres retention). [[ft002|vault-secrets]] is the obvious
consumed feature — every service already calls `secret(name)` and the
rotation flow needs that surface to swap values atomically. Categories
[[cat001]] security and [[cat003]] infrastructure both apply.

Out of scope at idea stage: the actual rotation script, the synthetic-check
hook, the secret-overlap window mechanism. spec-writing chooses the shape
across vault-rotate-policy vs zero-downtime-swap vs blue-green-cred patterns.

## solution

Adopt a **dual-secret overlap-window** rotation pattern layered on
[[ft002|vault-secrets]]. A new credential is written to vault alongside the
current one (versioned key), services continue resolving via the existing
`secret(name)` surface so no service-side code changes are required, and the
old value is revoked after a 5-minute overlap. This is a [[cat001|security]] +
[[cat003|infrastructure]] change — consumers stay untouched, only the rotation
choreography lives outside the request path.

Bound features:
- [[ft002|vault-secrets]] — the rotation flow writes-then-revokes through this
  surface; AC "rotation script can swap secrets while services run" maps
  directly onto vault versioning + lazy `secret(name)` resolution.

Binding ADRs:
- [[adr0001]] (cat001) — unaffected. Services still authenticate via
  [[ft001|basic-auth]]; the rotation only changes the value of the secret
  basic-auth reads, not the auth mechanism itself.
- [[adr0003]] (cat003) — unaffected. No new cross-service plumbing added; the
  rotation is an out-of-band operator script plus a vault interaction.

Trade-offs taken:
- Picked vault-versioning + overlap over **blue-green credential cycling**
  (new service instances brought up with new secrets, old drained). Blue-green
  satisfies AC but costs deploy churn per rotation, would need a new ADR for
  deployment topology, and bypasses the existing `secret(name)` lazy-cache
  indirection that already exists in [[ft002]]. Overlap is cheaper and reuses
  the bound feature.
- Picked vault-versioning over **app-coordinated reload** (each service exposes
  a `/reload-secrets` endpoint). That would require touching every consumer
  and introduce a new auth-cred surface — both out of scope for a story whose
  intent is operational, not architectural.
- The 5-minute overlap is honored by configuring vault to keep version N
  readable for 5 minutes after version N+1 is written; revocation is the
  explicit end-of-overlap step.

No existing [[im###]] solves [[us003]] — clean mint downstream at
spec-refinement. No supersession candidates against [[adr0001]] or [[adr0003]].
