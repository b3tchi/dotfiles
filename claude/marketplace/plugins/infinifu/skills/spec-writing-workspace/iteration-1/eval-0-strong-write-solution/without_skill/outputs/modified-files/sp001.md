# Spec [[cat001]] [[cat003]] [[board]]

## solves
[[us003|rotate service credentials without downtime]]

## implements
[[im002|zero-downtime credential rotation via vault dual-secret overlap]]

## problem
The platform-engineer [[pn002|platform-engineer]] needs to rotate service
credentials without downtime. Today this is a maintenance-window task; the
story [[us003]] asks for live rotation with a 5-minute overlap and zero 5xx.

The solution will inherit from [[adr0001]] (basic-auth via [[ft001]]) and
[[adr0002]] (Postgres retention). [[ft002|vault-secrets]] is the obvious
consumed feature — every service already calls `secret(name)` and the
rotation flow needs that surface to swap values atomically. Categories
[[cat001]] security and [[cat003]] infrastructure both apply.

Out of scope at idea stage: the actual rotation script, the synthetic-check
hook, the secret-overlap window mechanism. spec-writing chooses the shape
across vault-rotate-policy vs zero-downtime-swap vs blue-green-cred patterns.

## solution

**Chosen shape: dual-secret overlap window inside `ft002` vault-secrets.**

Of the three candidate patterns surfaced at idea stage:

- *vault-rotate-policy* — push rotation into Vault's native rotation
  policy. Rejected: ties us to Vault-specific features, no per-service
  control over the overlap window, and the synthetic-check hook in the
  acceptance criteria has nowhere natural to live.
- *blue-green-cred* — stand up a parallel deployment with the new
  secret, drain traffic. Rejected: heavy infra change, conflicts with
  [[adr0003]] preference for minimal infra (everything in-process), and
  overkill for a 5-minute overlap requirement.
- *zero-downtime-swap via dual-secret overlap* — **chosen.** Vault
  holds `current` and `previous` versions of each secret simultaneously
  for an explicit overlap window. Services accept either during the
  window. After the window, `previous` is destroyed.

The contract change lives in [[ft002]] vault-secrets. Today `secret(name)`
returns one value; we widen it so that consumers transparently accept
either the current or previous version during the overlap. A new
`rotate(name, new_value, overlap=300)` admin entry point writes the new
value as `current`, demotes the old to `previous`, and schedules the
`previous` destruction after `overlap` seconds.

A `bin/rotate-secret` CLI wraps `rotate()` for the platform-engineer.
A synthetic check (curl loop against an authenticated endpoint of each
of the three services — auth, reports, metrics) runs continuously during
the overlap window; rotation aborts and rolls back if any 5xx is observed.

Inherits from:

- [[adr0001]] — all services authenticate via [[ft001]]; rotation must
  not break that flow. The overlap window covers the propagation gap
  between Vault write and in-process secret cache refresh.
- [[adr0002]] — Postgres-backed `report_runs` is untouched; rotation is
  credential-only, no schema impact.
- [[ft002]] — consumed and *extended* by this spec. The `secret(name)`
  surface keeps backward compatibility (returns `current`), while the
  new dual-read behavior and `rotate()` admin entry point widen the
  contract. The ft002 zettel will be updated to reflect the widened
  api_surface as part of this spec landing.

Categories: [[cat001]] security (credential handling) and [[cat003]]
infrastructure (cross-service rotation plumbing).

Affected components:

- `src/lib/vault.py` — widen `secret()`, add `rotate()`.
- `bin/rotate-secret` — new CLI (out of `src/lib/`, top-level script).
- `src/services/{auth,reports,metrics}/__init__.py` — no code change;
  each already calls `secret(name)` via ft002, so dual-read is
  transparent.
- Synthetic check script — new file under `src/lib/` or `bin/`,
  spec-refinement will pin location.

Open at spec-refinement:

- Where the secret-version cache lives in-process (TTL vs explicit
  invalidation on rotate).
- Whether the synthetic check is one binary or a shell loop.
- Rollback path when the synthetic check trips mid-window.
