# Spec [[cat001]] [[cat003]] [[board]]

## solves
[[us003|rotate service credentials without downtime]]

## implements
[[im002|vault-policy credential rotation for live services]]

## problem
The platform-engineer [[pn002|platform-engineer]] needs to rotate service
credentials without downtime. Today this is a maintenance-window task; the
story [[us003]] asks for live rotation with a 5-minute overlap and zero 5xx.

The solution will inherit from [[adr0001]] (basic-auth via [[ft001]]) and
[[adr0002]] (Postgres retention). [[ft002|vault-secrets]] is the obvious
consumed feature — every service already calls `secret(name)` and the
rotation flow needs that surface to swap values atomically. Categories
[[cat001]] security and [[cat003]] infrastructure both apply.

Out of scope at idea stage: the actual rotation script, the synthetic-check
hook, the secret-overlap window mechanism. spec-writing chooses the shape
across vault-rotate-policy vs zero-downtime-swap vs blue-green-cred patterns.

## solution

This spec executes [[im002]], which already chose the **vault-rotate-policy**
shape and is `accepted`. We do not mint a new implementation card — the
solution shape is fixed by im002. This spec is the execution plan against
that shape.

**Approach (per [[im002]]).** Writers stage the new credential under a
versioned alias in [[ft002|vault-secrets]]; readers fall back to the prior
version for up to 5 minutes; the vault alias flips at the end of the window.
Service code keeps reading via `acme.lib.vault.secret(name)` unchanged; the
overlap is invisible to call sites.

**Surface added by this spec (delta on top of [[ft002]]).**

- New internal helper `acme.lib.vault.rotate_secret(name)` in `src/lib/vault.py`
  that stages a new version, holds the prior version live for the overlap
  window, then flips the alias.
- A CLI / runbook hook so the platform-engineer can drive rotation per
  service or across a service set.
- A synthetic-check hook that probes the affected services across the
  5-minute overlap and fails fast if any 5xx surfaces.

**Inherited constraints.**

- From [[ft002]]: vault is the source of truth; no local state owned by
  this rotation flow.
- From [[adr0001]]: rotation must not disturb the basic-auth session layer
  (sessions in Redis, 24h TTL).
- From [[adr0002]]: report data retention is unaffected — no Postgres
  schema or retention change in scope.

**Consumed features.**

- [[ft002|vault-secrets]] — the only read surface; `rotate_secret` lives
  next to `secret` in the same lib.

**Categories.** [[cat001]] security (credential handling, overlap window
correctness) and [[cat003]] infrastructure (cross-service plumbing,
rotation operability).

**Out of scope for this spec.**

- Replacing `smtplib` per [[adr0003]] — orthogonal.
- SSO / external identity per [[adr0001]] consequences — orthogonal.
- Vault HA / DR topology changes — `ft002` carries that contract.
- Rotating long-lived TOTP seeds (covered by [[ft001]] separately).

**Acceptance maps to [[us003]] criteria.**

- *Rotation script can swap secrets while services run* → `rotate_secret`
  exposed via a small CLI wrapper; a service stays up across the call.
- *Old secret stays valid for 5 minutes after rotation* → vault holds the
  prior version through the overlap window; reads succeed during it.
- *No 5xx during rotation window in synthetic check* → synthetic probe
  driven through the auth surface ([[ft001]]) sees zero 5xx across the
  window.

The execution plan, file tree, and per-task breakdown (with bd ids) will
be filled in by spec-refinement and spec-ready under `## plan` and
`## tasks`.
