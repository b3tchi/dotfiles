# Spec [[cat001]] [[cat003]] [[board]]

## solves
[[us003|rotate service credentials without downtime]]

## implements
[[im002|vault-policy credential rotation for live services]]

## problem
The platform-engineer [[pn002|platform-engineer]] needs to rotate service
credentials without downtime. Today this is a maintenance-window task; the
story [[us003]] asks for live rotation with a 5-minute overlap and zero 5xx.

The solution will inherit from [[adr0001]] (basic-auth via [[ft001]]) and
[[adr0002]] (Postgres retention). [[ft002|vault-secrets]] is the obvious
consumed feature — every service already calls `secret(name)` and the
rotation flow needs that surface to swap values atomically. Categories
[[cat001]] security and [[cat003]] infrastructure both apply.

Out of scope at idea stage: the actual rotation script, the synthetic-check
hook, the secret-overlap window mechanism. spec-writing chooses the shape
across vault-rotate-policy vs zero-downtime-swap vs blue-green-cred patterns.

## solution
**Dedup decision: adopt the existing [[im002|vault-policy credential
rotation]] — do not mint a new implementation card.** A survey of
`docs/notes/im*.md` shows [[im002]] (status: `accepted`) already `solves
[[us003]]` with the vault-rotate-policy pattern: writers stage a new
credential under a versioned alias, readers fall back to the prior version
for up to 5 minutes, the alias flips at the end of the window. That shape
satisfies every acceptance criterion on [[us003]] (live swap, 5-minute
old-secret overlap, zero 5xx during rotation in synthetic check) and aligns
with the two binding ADRs under categories [[cat001]] and [[cat003]] —
[[adr0001]] (auth via [[ft001]] is untouched by rotation) and [[adr0002]]
(Postgres retention is orthogonal). No `Accepted` ADR conflicts with the
chosen shape, so no supersession candidate is filed. The single consumed
feature is [[ft002|vault-secrets]] (the existing `secret(name)` read path
plus the new internal `rotate_secret(name)` helper that [[im002]] already
specifies in `src/lib/vault.py`). The trade-off taken: bind to the
vault-policy approach over the zero-downtime-swap or blue-green-cred
patterns — vault-policy keeps all rotation logic inside the existing vault
client and avoids per-service swap orchestration, at the cost of requiring
vault to hold both old and new versions for the 5-minute overlap (already
supported by [[ft002]]). This spec therefore executes [[im002]] as-written;
spec-refinement will add `## plan` + `## tasks` (rotation script,
synthetic-check hook, secret-overlap-window verification) against the
components [[im002]] already lists.
