# Spec [[cat001]] [[cat003]] [[board]]

[[us003|rotate service credentials without downtime]]

## implements
[[im002|zero-downtime credential rotation via dual-secret overlap]]

## problem
The platform-engineer [[pn002|platform-engineer]] needs to rotate service
credentials without downtime. Today this is a maintenance-window task; the
story [[us003]] asks for live rotation with a 5-minute overlap and zero 5xx.

The solution will inherit from [[adr0001]] (basic-auth via [[ft001]]) and
[[adr0002]] (Postgres retention). [[ft002|vault-secrets]] is the obvious
consumed feature — every service already calls `secret(name)` and the
rotation flow needs that surface to swap values atomically. Categories
[[cat001]] security and [[cat003]] infrastructure both apply.

Out of scope at idea stage: the actual rotation script, the synthetic-check
hook, the secret-overlap window mechanism. spec-writing chooses the shape
across vault-rotate-policy vs zero-downtime-swap vs blue-green-cred patterns.

## solution

**Chosen shape: dual-secret overlap inside [[ft002|vault-secrets]].**

Of the three candidate patterns surveyed at idea stage, we pick
**zero-downtime-swap via dual-secret read** because it requires no
service restarts, no LB cutover, and reuses the existing `secret(name)`
contract every service already calls.

### Approach

1. **Versioned secrets in Vault.** For every rotatable secret we maintain
   two versions in Vault: `current` and `previous`. The rotation flow
   writes the new value to `current` (moving the old `current` to
   `previous`) and starts a 5-minute overlap window during which both
   versions are valid against the downstream system (DB user, API key,
   etc.).

2. **`secret(name)` returns `current` only.** No change to the
   read-side contract for callers. New writers always pick up `current`
   on their next call (Vault client TTL is 30s, which fits inside the
   overlap window with margin).

3. **Downstream acceptance of both versions during overlap.** This is
   what makes "zero 5xx" possible. Concretely:
   - For Postgres credentials: pre-create the new role, grant the same
     permissions, and let the old role keep working until the overlap
     window expires; then revoke.
   - For internal HTTP API keys: gateway validates against the set
     `{previous, current}` for the overlap window, then drops
     `previous`.

4. **Rotation CLI.** A new `acme-rotate` command (lives in
   `src/lib/vault.py` package, exposed via console script) drives the
   flow:
   ```
   acme-rotate <secret-name> [--overlap=5m] [--dry-run]
   ```
   Steps: mint new value → write to Vault `current` → wait for
   downstream to accept both → poll synthetic check → expire
   `previous` after overlap window.

5. **Synthetic check hook.** During the overlap window the CLI polls a
   service-specific health endpoint (`GET /healthz/auth`) that the
   service implements to confirm it can mint a session using the
   currently-cached `secret(name)`. Two consecutive green polls within
   the window are required before `previous` is expired. A red poll
   triggers immediate rollback (re-promote `previous` → `current`).

### Inherited constraints

- [[ft002|vault-secrets]] keeps its existing `secret(name)` API
  surface — no changes for consumers. The dual-version semantics are
  internal to Vault and the rotation CLI.
- [[adr0001]] basic-auth via [[ft001]]: rotation does not touch user
  passwords or TOTP seeds; those live in `users` / `totp_secrets`
  tables, not in Vault.
- [[adr0002]] Postgres retention: not affected — rotation is operational
  state, not report data.

### Components touched

- `src/lib/vault.py` — extend with version-aware read path (still
  returns scalar to callers) and helpers used by the CLI.
- `src/lib/vault_rotate.py` *(new)* — rotation orchestrator + CLI
  entry point.
- `src/services/*/healthz.py` *(new per service)* — synthetic
  `/healthz/auth` endpoint that exercises `secret(name)`.
- Vault policy update — write path for the rotation principal.

### Anti-patterns explicitly rejected

- **Coordinated restart with new secret.** Caused the maintenance
  window in the first place. Rejected on the zero-downtime requirement.
- **Blue-green deploy of services per rotation.** Heavy, expensive
  for a rotation flow that should be boring.
- **Vault lease-based dynamic credentials.** Promising long-term but
  requires reshaping each service to handle credential expiry
  mid-request. Out of scope; revisit as a separate story.

### Open questions surfaced (defer to refinement)

- **AC vagueness on the story.** [[us003]] currently lists "it should
  work" and "be fast enough" as acceptance_criteria. The concrete
  targets (5-minute overlap window, zero 5xx during rotation, rollback
  on red synthetic) live in this spec's problem statement, not in the
  story. spec-refinement should either (a) tighten [[us003]] to carry
  those targets verbatim or (b) accept the spec as the source of truth
  for AC and note the story as informally-scoped.
- **Synthetic-check coverage.** Each service needs the
  `/healthz/auth` endpoint before its first rotation; gating order is
  a refinement-stage decision.
- **Audit trail.** Should rotations emit an audit-log event? No shared
  audit feature exists yet (would be a new ft###); spec-refinement
  decides whether to defer that or pull it in.
