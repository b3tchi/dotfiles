# Implementation [[cat001]] [[cat003]]

## solves
[[us003|rotate service credentials without downtime]]

## approach
Adopt the vault-rotate-policy pattern: writers stage the new credential
under a deterministic versioned alias backed by Vault's Transit secrets
engine; readers fall back to the prior version for up to 5 minutes; alias
flips at the end of the window. Original [[sp001]] plan was KV-backed; in
flight we shifted to Transit for native deterministic versioning — see
[[adr0004|Vault Transit secrets engine over plain KV for credential rotation]]
for the trade-off record. The Transit primitives are wrapped once in
[[ft003|vault-transit-wrapper]] so future encrypted-rotation flows
(alerting, signed-URL) consume the same surface.

## features
- [[ft002|vault-secrets]] — Vault connectivity + plain `secret(name)` read path
- [[ft003|vault-transit-wrapper]] — encrypt / decrypt / rotate_key primitives

## data_model
No schema change. Vault holds versioned aliases; key versions are owned
by the Transit engine.

## api_surface
- `acme.lib.vault.secret(name)` continues to be the only read path
  (unchanged from [[ft002]])
- `acme.lib.vault.rotate_secret(name, new_value)` is the internal helper
  that stages a new versioned alias and serializes concurrent rotations
  per name; raises `VaultError` on backend failure, `ValueError` on
  empty input
- `acme.lib.vault_transit.encrypt / decrypt / rotate_key` provide the
  Transit-backed primitives consumed by `rotate_secret` for the
  deterministic version flip (surface owned by [[ft003]])

## components
- `src/lib/vault.py` — `rotate_secret` helper + per-name lock map + alias bookkeeping
- `src/lib/vault_transit.py` — Transit wrapper (consumed surface; owned by [[ft003]])
- `src/lib/vault_rotate.py` — orchestration: 5-minute overlap timer + expiry + synthetic-check hook

## binds

- [[adr0002]] — Postgres retention (unaffected; rotation is Vault-side)
- [[adr0004]] — Transit over KV for credential rotation (new, recorded post-merge)
