# ADR [[cat003]] [[product]]

## title
Vault Transit secrets engine over plain KV for credential rotation

## context
[[sp001|rotate service credentials without downtime]] originally planned to
back versioned aliases on Vault's KV v2 engine — same client surface as
[[ft002|vault-secrets]], minimal new infra. During implementation we
re-evaluated against the [[us003]] acceptance criteria (5-minute overlap,
deterministic version flip, zero 5xx in the synthetic window) and weighed
KV against Vault's Transit secrets engine.

Trade-offs:

- **KV v2 (original plan):** Backend-agnostic. We could swap Vault for any
  other secret store later (AWS Secrets Manager, GCP Secret Manager, an
  internal store) with a thin adapter. Versioning is on the *secret*, not
  on a key, so we'd have to roll our own deterministic alias-rotation
  protocol and overlap window in application code.
- **Transit (chosen):** Deterministic versioned key rotation is a first-class
  primitive — `rotate_key` returns the new version, encrypt/decrypt honor
  prior versions for as long as we keep them. The 5-minute overlap reduces
  to "keep version N-1 readable for 5 minutes", which Transit supports
  natively. Cost: we are now bound to HashiCorp Vault specifically — any
  future provider migration requires a real port, not a config swap.

Cross-cutting: every future encrypted-rotation capability (alerting
signing keys, signed-URL flows in reports) will land on Transit, not KV.

## decision
Adopt Vault Transit as the backing engine for all credential-rotation
flows. KV v2 remains in use for plain secret retrieval via [[ft002]];
Transit is the new substrate for anything that needs deterministic
versioned rotation. New code that needs encrypted rotation primitives
consumes the wrapper described in [[ft003|vault-transit-wrapper]].

## consequences
- Lock-in to HashiCorp Vault for the rotation paradigm. Provider migration
  is now an explicit rewrite, not a refactor.
- Availability dependency on the Transit engine; if Transit is down,
  rotation is blocked even if KV is healthy. Operations needs a Transit
  health probe in the synthetic-check fleet.
- Forecloses the "back rotation on a non-Vault store" option that was
  implicit in the original [[sp001]] plan. Future ADRs that want to lift
  this constraint will need to supersede this one.
- Existing KV consumers (read paths in [[ft002]]) are unaffected — the two
  engines coexist in the same Vault.
