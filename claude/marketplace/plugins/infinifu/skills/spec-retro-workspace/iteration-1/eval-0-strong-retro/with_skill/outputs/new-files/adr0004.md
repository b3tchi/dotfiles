# ADR [[cat001]] [[product]]

## title
Credential rotation lives in vault.py — no separate orchestration module

## context
[[sp001]] / [[im002]] originally planned a split: `vault.py` would hold
the `rotate_secret` write helper, and a new `vault_rotate.py` would own
the 5-minute overlap timer, expiry scheduling, restart-resume logic,
and a synthetic-check hook. During execution two facts emerged: (1)
versioned-alias retention is already managed by the backend vault, so
an in-process expiry timer was redundant; (2) concurrent serialization
naturally belongs next to the alias state, not in a sibling module
calling back into it. Splitting would have created a thin shim that
forwarded everything to `vault.py`.

## decision
Keep credential rotation in `src/lib/vault.py` — one module owns
`secret`, `rotate_secret`, per-name locks, alias bookkeeping, and
`VaultError`. Do not ship `vault_rotate.py`. Do not ship an in-process
expiry scheduler or a synthetic-check hook as part of [[sp001]];
backend-managed retention plus the per-name lock cover the
zero-downtime acceptance criterion for single-region rotation. See
[[im002]] for the resulting solution shape.

## consequences
- Smaller surface: one module, one import, no scheduler lifecycle to
  reason about.
- The 5-minute overlap is a vault-backend contract, not an in-process
  guarantee — if the backend ever stops honouring versioned aliases,
  the overlap silently disappears and rotations become
  read-during-overlap-unsafe.
- No synthetic-check hook ships with this spec — observability for
  rotation windows is deferred and surfaces as follow-up scope (see
  [[us006]] for the cross-region case where in-process verification
  becomes necessary).
- Future multi-step orchestration (e.g. cross-region staging) cannot
  be added by editing `vault.py` without bloating it — that work
  should ship a new module and likely supersede this ADR.
