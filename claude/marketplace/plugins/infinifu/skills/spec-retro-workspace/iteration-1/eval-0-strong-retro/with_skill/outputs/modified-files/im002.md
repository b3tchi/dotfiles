# Implementation [[cat001]] [[cat003]]

## solves
[[us003|rotate service credentials without downtime]]

## approach
Rotation is implemented entirely inside the vault client module — writers
call `rotate_secret(name, new_value)` which stages a new versioned alias
under a per-name `threading.Lock`; readers continue to call `secret(name)`
and transparently get the latest staged version. The originally-planned
split into a separate `vault_rotate.py` orchestration module (with an
explicit T+5min scheduler and a synthetic-check hook) was collapsed:
versioned-alias retention is now backend-managed at the vault layer, so
no in-process scheduler / expiry timer / synthetic-check ships in this
spec. See [[adr0004]] for the rationale. Binds [[adr0001]] and
[[adr0002]].

## features
- [[ft002|vault-secrets]]

## data_model
No schema change. The module keeps an in-memory `_ALIASES: dict[str,
list[str]]` of staged versions per name and a per-name lock map
(`_LOCKS`) guarded by a single module-level mutex (`_LOCKS_GUARD`).
`secret(name)` returns the most recently staged version. Vault remains
the source of truth for actual storage and version retention.

## api_surface
- `acme.lib.vault.secret(name) -> str` — unchanged read path.
- `acme.lib.vault.rotate_secret(name: str, new_value: str) -> None` —
  stages a new versioned alias; per-name serialization via lock.
  Raises `ValueError` on empty `name` or empty / `None` `new_value`;
  raises `VaultError` if the backend rejects.
- `acme.lib.vault.VaultError(RuntimeError)` — error class for backend
  failures during rotation.

## components
- `src/lib/vault.py` — single module: read path, rotate path, locks,
  alias bookkeeping, and `VaultError`.
- `tests/lib/test_vault.py` — 5 unit tests covering stage, flip,
  concurrent serialization, empty value rejection, empty name
  rejection.
