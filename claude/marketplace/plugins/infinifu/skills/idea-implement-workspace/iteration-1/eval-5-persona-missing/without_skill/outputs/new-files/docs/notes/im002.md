# Implementation [[cat001]] [[cat002]]

## solves
[[us005|quarterly audit-log export]]

## approach
Two-part solution. (1) Introduce a new reusable audit-log Feature
[[ft003]] — an append-only `audit_deletions` table plus a thin
`record_deletion()` helper. Wire the existing 90-day retention job
(adr0002) so every hard-delete it performs also writes one row to
`audit_deletions`. (2) Add a CLI command `reports audit-export
<quarter>` in the reports service that selects the trailing quarter
of `audit_deletions`, emits a CSV, and detached-signs it using a
vault-managed key from [[ft002]]. The signed pair (`.csv` +
`.csv.sig`) is the artefact handed to legal.

Bounded by adr0001 (compliance-officer authenticates via ft001
basic-auth — no SSO) and adr0002 (the 90-day retention job is the
*source* of deletion events this card captures). adr0003 (smtplib
direct, no relay) is touched only insofar as we currently have no
shared delivery mechanism — for the first quarter we hand the file
over out-of-band rather than email it, and revisit if the manual
hand-off proves painful.

## features
- [[ft003|audit-log]]
- [[ft002|vault-secrets]]
- [[ft001|basic-auth]]

## data_model
No additional tables beyond [[ft003]]'s `audit_deletions`. The
reports service does not own audit state; it owns the export
operation.

## api_surface
- CLI: `reports audit-export <YYYYqQ> --out <path>` — writes
  `<path>.csv` + `<path>.csv.sig`.
- Optional follow-up (not in scope of first ship): authenticated
  `GET /audit/export/<quarter>` returning the signed pair as a tar.

## components
- `src/services/reports/audit_export.py` — CLI + quarter→date-range
  resolution, CSV writer, signing call.
- `src/services/reports/tests/test_audit_export.py` — round-trip
  sign + verify, plus boundary tests for quarter edges.
- Hook into existing retention job: one new line calling
  `record_deletion(...)` per hard-deleted row.
