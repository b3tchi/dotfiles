# Spec [[cat001]] [[cat003]] [[board]]

## solves
[[us003|rotate service credentials without downtime]]

## implements
[[im002|zero-downtime secret rotation via ft002 dual-version cache]]

## problem
Quarterly secret rotation on the Acme internal platform is currently a
maintenance-window task: the platform-engineer takes the three services
(auth, metrics, reports) offline, swaps the Vault secret, and brings
them back up. The rotation itself takes seconds; the outage window is
the actual cost — it forces off-hours work and burns the team's SLA
budget for something that should be routine.

The blocker is that every service reads its secrets at process start
via `acme.lib.vault.secret(name)` and caches the value for the
lifetime of the worker. When Vault flips to a new version, in-flight
requests using the old cached value continue to authenticate against
the *new* downstream credential and fail with 5xx until the workers
are restarted. So the safe play today is: drain, rotate, restart, un-drain.

Goal: make a single-secret rotation a runtime operation — the operator
runs one command, both old and new secret values stay valid for a
short overlap window (default 5 minutes), the workers pick up the new
value asynchronously, and the synthetic check observes zero 5xx for
the duration. Scope is bounded to the three current services and the
existing ft002 vault-secrets client; SSO/SAML, external KMS, and
multi-region replication are explicit non-goals.

This spec executes [[im002]] which proposes solving the problem by
extending ft002 itself (dual-version read with a TTL overlap cache)
plus a new `rotate-secret` CLI, rather than duplicating rotation
glue per service.
