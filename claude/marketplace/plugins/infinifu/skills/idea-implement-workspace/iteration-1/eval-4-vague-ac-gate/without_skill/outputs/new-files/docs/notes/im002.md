# Implementation [[cat002]] [[cat003]]

## solves
[[us004|search reports by title and date]]

## approach
Extend the existing reports-service `/dashboard` route (see [[im001]])
with optional `q`, `from`, `to` query params. Translate to a single
SQL query against `report_runs` using `ILIKE` on `title` and a
`run_at` range. No new tables, no new service — same auth path
([[ft001]]) and same retention envelope as [[adr0002]] (90 days).
Server-side filtering replaces the current client-side filter for
this code path, so reports older than the dashboard's last-20 window
become reachable. Add a B-tree index on `report_runs(run_at)` and a
trigram index on `report_runs(title)` to keep p95 under 500 ms across
the full 90-day window.

## features
- [[ft001|basic-auth]]

## data_model
No schema change. Two new indexes on the existing `report_runs`
table:
- `idx_report_runs_run_at` (B-tree on `run_at`)
- `idx_report_runs_title_trgm` (GIN, `pg_trgm` extension on `title`)

## api_surface
- `GET /dashboard?q=<text>&from=<YYYY-MM-DD>&to=<YYYY-MM-DD>` → HTML
  page; empty params = current behavior
- `GET /dashboard/data.json?q=&from=&to=` → JSON same shape as today
  (search params optional)

## components
- `src/services/reports/dashboard.py` (extend query builder)
- `src/services/reports/templates/dashboard.html` (search form)
- `src/services/reports/migrations/00X_search_indexes.sql` (new)
